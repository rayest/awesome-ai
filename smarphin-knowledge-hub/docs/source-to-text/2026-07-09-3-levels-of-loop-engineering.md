# 3 Levels of Loop Engineering：从 Prompt 到可验证工作循环

## 一句话总结

Loop Engineering 的重点不是把 prompt 写得更漂亮，而是把任务设计成可触发、可拆分、可验证、可复用的工作循环；越复杂的任务，越需要清晰的数据结构、独立验证和人类对关键决策的审查。

## 核心观点

1. **从 prompt engineering 转向 task description engineering**  
   普通的一次性 prompt 很容易让 agent 误解目标。更有效的做法是把需求写成清晰的任务描述，明确输入、输出、边界、验收标准和人工检查点。

2. **Loop 的本质是工作流，而不是单次对话**  
   一个 loop 通常包括：创建任务、细化规格、派发 agent、实现、验证、复审、合并或迭代。它把 AI 从“临时助手”变成“流程节点”。

3. **复杂任务要靠拆分和独立验证，而不是继续堆 prompt**  
   文中最有价值的例子是把“检查历史报告是否已在生产修复”拆成两个专业角色：`git diff checker` 找候选修复提交，`patch validator` 独立验证候选提交是否真的修复问题并映射到 release。这个小改动让确定性结果从 35% 提升到 90%。

4. **AI 可以设计 workflow，但你必须能审查它**  
   Level 3 允许多个前沿模型互相设计、批判 workflow，再交给实现 agent 执行。这适合你不知道最佳路径的大任务，但风险是：如果你不懂这个领域，就很难判断 workflow 设计得好不好。

5. **真正值得人类设计的是“决策变量”和数据结构**  
   讲解者补充的关键洞察是：不要只问 AI 要做什么，而要定义每一步要产出什么结构化决策。比如目标、假设、证据、风险、候选方案、验证状态、边界条件。这些结构化产物才是人类可以审查、复用、组合和自动化的核心。

## 内容结构

### Level 1：推进产品路线图

适合日常、边界清晰的软件任务。典型流程是：

1. 在 Linear / Kanban / Trello 类系统里创建 ticket。
2. 当任务从 backlog 移到 todo 时，自动触发 implement agent。
3. agent 实现后把任务移到 review。
4. 独立 review agent 做代码审查。
5. 可以要求 agent 上传截图、录屏或进度说明，让人类作为审批 gate。
6. 审查通过后再合并。

这个层级的变化是：团队不再每次手写 prompt，而是在任务管理系统里写清楚任务描述，再让固定 workflow 执行。

### Level 2：设计自定义 workflow

适合你理解领域，但任务太大、太复杂，单个 agent 容易上下文腐烂的情况。

关键方法是：

- 把大任务拆成职责明确的小角色。
- 每个 agent 只处理一类判断。
- 下游 agent 独立验证上游 agent 的结果。
- 多个 agent 结论不一致时，由裁决逻辑或人类判断。

文中的修复追踪案例说明：角色拆分比“更聪明的 prompt”更有效。因为拆分降低了上下文负担，也让验证更明确。

### Level 3：让 AI 生成 workflow

适合你不知道最佳流程，但可以事后验证结果的大任务。例子是让多个前沿模型设计区块链客户端实现 workflow，互相 critique，最后交给 Codex 实现，并由独立 verifier 生成场景测试。

这个方法可能非常强，但也有明显风险：

- workflow 看起来漂亮，不代表结果可靠。
- 如果原始材料太大，研究型任务可能出现 context rot。
- 如果你不懂领域，就很难判断 AI 的 workflow 是否走偏。
- AI 生成的流程仍然需要验证器、测试、审查和人工判断。

## 关键方法或步骤

### 1. 先定义输入

输入不是“帮我做一个功能”这种模糊请求，而应该至少包括：

- 任务背景
- 目标结果
- 不做什么
- 现有约束
- 验收标准
- 可用工具
- 期望输出格式

### 2. 把任务描述细化到可交接

文中提到 `grill me` 类 skill 的作用：让 agent 反问你，直到任务描述足够清楚，可以交给实现 agent。

可以让 AI 追问：

- 你希望最终产物是什么？
- 哪些边界不能碰？
- 哪些行为算成功？
- 哪些行为算失败？
- 是否需要截图、测试、日志或报告？

### 3. 设计 loop 节点

一个实用 loop 至少包含：

1. `Plan`：创建任务和目标。
2. `Specify`：细化需求，补齐上下文。
3. `Dispatch`：派发给 agent。
4. `Implement`：执行任务。
5. `Verify`：独立验证结果。
6. `Review`：人类或独立 agent 审查。
7. `Merge / Publish / Archive`：交付或沉淀。

### 4. 给每个节点定义结构化输出

这是材料里最值得沉淀的方法论。每一步最好都输出一个结构化文件或字段，而不是只输出自然语言长文。

示例字段：

```yaml
goal: 要解决的问题
assumptions: 当前假设
inputs: 使用了哪些输入
candidate_decisions: 候选判断
evidence: 支撑证据
risks: 风险和不确定性
verification_status: 验证状态
next_action: 下一步动作
human_review_needed: 是否需要人工审查
```

### 5. 让验证独立于实现

不要让同一个 agent 既实现又给自己打分。更稳的设计是：

- implement agent：负责完成任务。
- verifier agent：只检查结果是否满足要求。
- adjudicator / human：处理冲突、判断品味、决定是否推进。

## 案例 / 演示

### 案例 1：产品路线图自动推进

把任务管理工具作为入口，用状态变化触发 agent：

```text
Backlog -> Todo -> In Progress -> Review -> Merge
```

每个状态变化对应一个自动化节点。非工程人员可以写 ticket，工程团队则把最佳实践编码进 workflow。

### 案例 2：历史 bug 修复追踪

失败做法：

- 对每个 report 同时派出 4 个 investigator。
- 3/4 一致就裁决。
- 结果未知项很多，确定性较差。

改进做法：

- `git diff checker`：从 report 日期开始向后查提交，找候选修复 commit。
- `patch validator`：独立确认候选 commit 是否真的修复 report，并映射到 release。

结果：明确状态从 35% 提升到 90%。

### 案例 3：区块链客户端实现

做法：

- 让两个前沿模型分别设计 workflow。
- 让它们互相 critique 多轮。
- 再交给 Codex 实现。
- 用独立 verifier 生成 scenario tests。

启发：当任务有成熟参考实现和强测试体系时，AI 的翻译和实现能力会被大幅放大；测试本身是高价值资产。

## 可照做操作步骤

1. **选一个具体工作，不要一开始就设计万能系统**  
   例如：小红书图文生成、短视频脚本分析、需求 ticket 实现、文章摘要入库、竞品资料整理。

2. **写清楚输入模板**  
   至少包含：任务背景、原始材料、目标产物、风格要求、输出格式、验收标准。

3. **定义每一步的结构化输出**  
   不要只让 AI “总结一下”。要求它输出固定字段，例如 `summary`、`key_points`、`risks`、`next_actions`、`review_needed`。

4. **把流程拆成 2-4 个角色**  
   例如内容工作流可以拆成：
   - `extractor`：提取事实和观点。
   - `designer`：设计图文结构。
   - `writer`：生成文案。
   - `reviewer`：检查事实、风格和平台适配。

5. **给 reviewer 明确否决权**  
   reviewer 不要只说“不错”，而要检查：
   - 是否偏离原文？
   - 是否有虚构事实？
   - 是否符合平台比例和风格？
   - 是否有过度营销感？
   - 是否需要人工确认？

6. **保留人工 gate**  
   对涉及品牌调性、代码架构、商业判断、发布前内容，最好保留人工确认节点。

7. **记录每次 loop 的输入、输出和失败点**  
   每跑一次都沉淀：
   - 哪一步最容易错？
   - 哪个字段不够清楚？
   - 哪个检查项应该加入 reviewer？
   - 哪个产物可以模板化？

8. **逐步把经验写回 skill / AGENTS.md / workflow 配置**  
   真正的复利来自把“这次学到的判断”写回规则里。

## 常见误区

1. **误以为 Level 3 一定比 Level 1 高级**  
   这三个层级不是线性升级关系，而是适配不同任务。日常明确任务用 Level 1 反而更稳。

2. **让 AI 自己实现、自己验证、自己批准**  
   这会制造假安全感。复杂任务至少要独立验证，关键任务要有人类 gate。

3. **把 workflow 设计得很漂亮，但没有可验证输出**  
   没有验收标准和结构化结果，再复杂的 loop 也只是自动化幻觉。

4. **忽略测试和参考实现的价值**  
   区块链客户端案例能成功，很大程度依赖已有规格、测试和参考实现。AI 不是凭空创造可靠性。

5. **只优化 prompt，不优化数据结构**  
   prompt 是入口，数据结构才是可复用的认知资产。

## 值得质疑的地方

- 讲者认为“很快不会再写普通代码”，这个判断可能有方向性价值，但时间尺度和适用范围需要保守看待。
- AI-only review 在低风险任务里有价值，但对于架构、数据安全、计费、权限、品牌发布等场景仍不应完全替代人类。
- “让 AI 生成 workflow”对未知领域很诱人，但也最容易让人无法判断结果质量。

## 对我的启发

1. 设计 AI 自动化时，先问“我要审查哪些决策”，再问“AI 要做哪些步骤”。
2. 每个 loop 都应该有产物，不只是对话记录。
3. 复杂任务要把 AI 当成多个职责不同的节点，而不是一个万能员工。
4. 真正可持续的自动化，是把失败经验写回 skill、workflow 和检查清单。
5. 人类的价值会更多集中在：定义结果、设计变量、判断品味、处理风险、决定边界。

## 一句话复盘

Loop Engineering 的核心是把 AI 工作从“单次生成”升级为“有输入、有结构、有验证、有审查、有沉淀”的系统；prompt 只是起点，决策变量和验证机制才是长期复利。
