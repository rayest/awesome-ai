# Agent 不缺能力，缺的是可验证交付

![行动建议](./2026-07-06-summary-03.png)

AI Agent 的讨论正在从“它能不能自动完成任务”转向一个更现实的问题：它完成之后，谁来验证？今天最值得关注的信号不是某个模型又多会聊天，而是 Agent 正在被要求留下演示、记录推理边界、接受代码审查、控制成本、被纳入权限与身份治理。换句话说，Agent 的下一阶段不是更会表演，而是更能交付。

Simon Willison 在 newsletter 中展示了一个非常具体的方向：让 coding agent 为自己的工作录制产品演示。TAAFT、The Neuron、Linas 等邮件也从不同角度指向同一件事：强模型正在被用于长任务、项目审查、代码修复、科学工作台和金融支付流程，但它们一旦进入真实业务，就必须面对证据、成本、权限、身份和责任链。

## 一、Agent 从“生成答案”走向“交付证据”

过去我们评价 AI 工具，常看它能否把需求写成代码、把文档总结成要点、把灵感变成方案。但在真实工作里，结果本身还不够。一个 Agent 如果修改了代码、跑了浏览器、操作了数据库或完成了客户流程，用户最需要的不只是“它说完成了”，而是能复核的证据。

Simon Willison 的 `shot-scraper video` 正是这个方向的典型例子。这个新命令通过 Playwright 执行 storyboard.yml 中定义的浏览器流程，并录制视频演示。示例里，Agent 使用 YAML 描述操作步骤，启动本地 Datasette 服务，演示从 CSV/TSV/JSON 批量插入数据、创建表、预览结果、确认保存等完整路径。

这背后的行业意义很清楚：Agent 工作要进入团队协作，就必须从“文本汇报”升级为“可视化交付物”。视频、日志、测试结果、差异对比、回滚说明，会成为 Agent 交付包的一部分。

| 交付方式 | 过去常见做法 | 更适合 Agent 时代的做法 |
| --- | --- | --- |
| 功能完成 | Agent 口头说明“已完成” | 自动录制演示视频，展示关键路径 |
| 代码变更 | 直接提交 diff | 附带测试、边界条件、风险说明和复核清单 |
| 数据操作 | 输出结果摘要 | 保留输入、处理步骤、预览与确认记录 |
| 产品验收 | 人工逐页点查 | 让 Agent 生成可重复执行的验收脚本 |
| 团队交接 | 聊天记录散落 | 形成需求、执行、证据、审查的一体化包 |

这也解释了为什么“会操作浏览器和终端”的 Agent 不应只被看作更强的聊天机器人。它更像一个初级执行者：可以做事，但必须留下做事过程。没有证据的自动化，只是更快地产生不确定性。

## 二、强模型应该被当作“判断资源”，不是无限劳动力

The Neuron 的邮件把 Claude Fable 5 描述成适合长任务的“周末承包商”，并引用 CAIS 与 Scale 的 Remote Labor Index 数据：Fable 5 得到 16.1%，约为下一个最佳模型的两倍。邮件的建议很实用：不要拿强模型做小跑腿，而要把它用在长上下文、复杂判断、项目审查、代码重构、PRD 到软件等任务上。

Simon Willison 的 sqlite-utils 4.0rc2 案例进一步说明了强模型的价值和边界。邮件提到，这次候选版本大部分由 Claude Fable 辅助完成，估算成本约 149.25 美元；整个过程经历 37 个提示、34 次提交、30 个文件，代码变化约 +1,321 / -190 行。模型帮助发现了 `delete_where()` 未提交导致后续事务被污染、`db.query("update ...")` 在抛错前已提交副作用、`INSERT ... RETURNING` 迭代不完整导致事务未关闭等问题。

这些不是“写一个函数”的任务，而是需要理解代码库、文档、事务语义、发布兼容性和用户预期的综合判断。强模型在这里像一个高级审查员：它能发现人容易忽略的边缘条件，但仍需要人类通过实验、文档审查和 PR 复核来确认。

| 模型/工具用法 | 适合交给谁 | 关键依据 |
| --- | --- | --- |
| 资料整理、草稿、清单 | 低成本模型或普通工具 | 容错率较高，主要是准备上下文 |
| 长任务规划、架构审查、复杂重构 | 强模型 | 判断链长、上下文多，错误代价较高 |
| 具体执行、批量修改、格式转换 | 工具链或低成本模型 | 任务边界明确，可用测试约束 |
| 发布前复核、风险找茬 | 另一个强模型 + 人类 | 跨模型审查能发现盲点，但不能替代验证 |
| 客户/财务/权限动作 | 人类审批 + 可审计 Agent | 涉及责任、合规和资金流 |

这引出一个非常重要的产品机会：模型路由会成为企业 AI 能力的一部分。未来成熟团队不会把所有任务都丢给最贵模型，而会区分“准备、判断、执行、验证”四个阶段。强模型用于最需要判断的节点，便宜模型与确定性工具用于上下文整理和重复执行，最终由测试、日志、视频和人类复核闭环。

## 三、AI 工具正在围绕“工作流痛点”重新分层

今天的邮件里出现了不少工具和产品信号，它们看似分散，实际上都在解决同一个问题：AI 需要进入真实工作链，而不是停留在一个聊天窗口。

TAAFT 汇总提到 ChatPlayground 可以让 ChatGPT、Claude、Gemini 等模型在同一个提示上回答，再互相审查、挑战并合并结果；Vellum 主打带长期记忆的个人助手，能记住偏好并在 Slack 中像队友一样协调任务；ZCode 提供基于 GLM 的免费 coding agent，支持代码库编辑和终端命令；Context.dev 把网站转换为 markdown、HTML 或结构化数据，让 Agent 更容易读取真实网页；Safari MCP server 则让 Agent 连接真实 Safari Technology Preview 窗口，检查页面、截图和调试 Web 应用。

Product Hunt 邮件里的 WorkBuddy 更简洁：它的定位是“用一组 AI 专家更快产出更锐利的结果”。这类产品说明，AI 工具正在从单模型能力走向“角色分工 + 工作流封装”。

| 工具/产品 | 目标用户 | 解决的痛点 | 值得关注的差异 |
| --- | --- | --- | --- |
| shot-scraper video | 开发者、产品团队、coding agent 使用者 | Agent 做完事后缺少可复核演示 | 把浏览器操作变成可录制、可复现的交付证据 |
| ChatPlayground | 研究、写作、产品和决策团队 | 单模型回答容易偏执或遗漏 | 多模型互审，适合做观点合成和盲点发现 |
| Vellum | 知识工作者、团队协作者 | 个人助手缺少长期记忆和偏好 | 强调 evolving memory 和任务协同 |
| ZCode | 开发者、开源工具使用者 | 编码 Agent 成本和门槛高 | 免费 GLM 编码 Agent，适合试验开放模型链路 |
| Context.dev | Agent 开发者、数据抓取和研究场景 | 网页对模型不友好，结构化读取成本高 | 把网站转为 Agent 可消费的格式 |
| Safari MCP server | Web 开发者、前端调试场景 | Agent 不能稳定观察真实浏览器状态 | 让 Agent 直接检查、截图、调试真实浏览器 |
| WorkBuddy | 内容、运营、产品和知识工作者 | 需要多角色专家协同而不是单一回答 | 用“AI 专家团队”包装任务流 |

这些产品的共同方向是：把 AI 从“能回答”变成“能嵌入”。谁能嵌入真实工具、真实上下文、真实权限、真实验收，谁才更接近可持续使用。

## 四、评估体系也在变：长任务、真实环境、多人协作成为重点

The Neuron 邮件还提到多个评估信号：WorldModelGym 测试世界模型是否能选择在真实环境中获胜的动作；SWE-Together 基于真实多轮 coding session，包含 109 个任务和公开 leaderboard；EdgeBench 加入 12 到 72 小时的长周期可执行 Agent 任务，衡量 Agent 是否能在长时间内改进。

这些 benchmark 的方向比单轮问答更重要。因为 Agent 真正难的不是回答一道题，而是在不完整信息、工具限制、环境变化和长时间执行中保持方向。一个模型如果能写出漂亮计划，但 12 小时后丢失上下文、重复尝试、烧掉预算、误操作权限，它就不适合进入生产系统。

| 新评估重点 | 为什么重要 | 对产品团队的启发 |
| --- | --- | --- |
| 多轮真实任务 | 更接近日常开发、运营和分析工作 | 不只测一次回答，要测持续协作能力 |
| 长周期执行 | 暴露循环、漂移、预算失控和状态管理问题 | 给 Agent 设置阶段检查点和停止条件 |
| 真实环境反馈 | 模型计划必须在实际系统中有效 | 用浏览器、日志、测试和用户反馈校正模型 |
| 公开 leaderboard | 有助于比较工具链和模型组合 | 不只追逐单模型榜单，也看系统能力 |
| 世界模型行动质量 | 评估“知道世界”是否能转化为“做对动作” | 对机器人、游戏、自动化和科学仿真尤其关键 |

这意味着，AI 产品的护城河不会只来自模型参数，而会来自评估数据、执行环境、失败案例库、权限模型和复核机制。真正成熟的 Agent 系统，要像工程系统一样被测试，而不是像营销演示一样被观看。

## 五、支付、身份和治理正在变成 Agent 落地的硬边界

Linas 的邮件把 AI 与金融科技的交叉讲得很具体。法国 Crédit Agricole、Mastercard 和 Worldline 完成了法国首个生产环境中的 agentic payment transaction：用户让数字 Agent 按预算、活动类型和地点搜索节日活动，再请求 AI 发起购买；Crédit Agricole 负责认证和授权，并强调可追踪处理。

同一封邮件还提到 OKX AI：一个让 AI Agent 工作并用稳定币收款的 marketplace，包含 Agent Marketplace、Task Marketplace、托管支付、即时结算和跨任务累积声誉的持久身份。赞助内容里也提到一个重要治理问题：企业部署能访问系统、触发工作流、代表员工行动的 Agent 后，传统身份与问责假设开始失效。

这类信息对 AI 产品很关键。Agent 一旦能发起购买、调动资金、访问敏感数据、代表员工执行流程，就不再只是 productivity tool，而是非人类身份主体。它需要身份、权限、审计、限额、撤销和责任归属。

| 场景 | 新机会 | 主要风险 |
| --- | --- | --- |
| Agent 支付 | 自动采购、差旅、订阅、B2B 结算 | 错误购买、欺诈、授权边界不清 |
| Agent marketplace | 任务外包给可评价的 AI 服务 | 质量不可控、身份冒用、责任难界定 |
| 稳定币结算 | 跨境、即时、低摩擦支付 | 合规、托管、反洗钱和波动外溢风险 |
| 企业身份治理 | 把 Agent 纳入 IAM 和审计系统 | 过度权限、提示注入、不可追踪操作 |
| 金融 Super App | AI + 支付 +社交形成闭环 | 平台封禁、数据合规、用户资金依赖 |

这也说明，未来的 Agent 产品不能只做“更聪明”。它必须回答“谁授权它做这件事”“它能花多少钱”“它能访问哪些数据”“失败时谁负责”“日志能不能被审计”。这些听起来像后端基础设施问题，但会决定 AI 是否能进入核心业务。

## 六、这对我们意味着什么

1. **把 Agent 交付物标准化。** 每个复杂任务都应要求产出需求摘要、执行记录、测试结果、演示视频或截图、风险说明和回滚方案，而不是只看最终回答。

2. **建立模型路由规则。** 用低成本模型整理上下文、生成初稿、列风险；用强模型做架构判断、复杂审查和关键决策；用工具链执行确定性步骤；用人类处理授权和责任。

3. **每次重要 AI 会话都做盲点审查。** The Neuron 提到的做法很值得采用：让模型说明最不确定的部分、用户遗漏的因素、可能改变结论的假设，以及需要由人、来源、日志或测试验证的内容。

4. **为长任务设置停止条件。** 对 12 到 72 小时级别的 Agent 任务，要定义检查点、预算上限、失败阈值、人工接管条件，避免“看似努力”的无限循环。

5. **把 Agent 当作身份主体管理。** 只要 Agent 能访问系统、调 API、花钱、发消息或改数据，就要纳入权限、审计、限额和撤销机制。

6. **优先试用能留下证据的工具。** 例如浏览器录屏、代码审查、网页结构化、真实浏览器调试、任务日志和多模型互审，这些比“更会聊天”的小功能更接近企业刚需。

7. **对 newsletter 中未经官方证实的信息保持核验习惯。** 涉及模型能力、产品发布、政府监管、支付合规、融资金额和 benchmark 排名的说法，应回到原始公告、论文、代码仓库或监管文件再用于正式决策。

今天的核心判断是：Agent 的竞争焦点正在从“生成能力”转向“交付制度”。模型越强，越需要成本控制、证据留存、权限治理和人工复核。真正能进入业务核心的 AI 系统，不是最会回答的系统，而是最能被信任、被审查、被接管的系统。
