# 上下文工程的五大核心能力 · 口播文案稿

> **适用场景**：技术博客 / 公众号音频 / 短视频口播
> **预计时长**：约 9~11 分钟
> **建议语速**：每分钟 230~260 字
> **阅读对象**：AI 应用开发者、产品经理、技术管理者

---

## 【开场钩子 · 30 秒】

你有没有想过这样一个问题——

同样是 GPT-4、同样是 Claude，凭什么 Claude Code 能帮我写代码、写完还能直接帮我提交，而我自己写个提示词，它就只能"看起来很努力、实际上跑偏"？

答案藏在一个关键词里——**上下文工程的五大核心能力**。

说白了，就是把 AI 从一个"会聊天的玩具"，变成一个"能干活的下属"。

今天这期节目，我就把这五大能力一次讲透。

---

## 【主厨开场 · 60 秒】

我特别喜欢一个比喻——

把 Agent 系统想象成一位米其林餐厅的主厨。

每天一百桌客人进门，每桌点菜都不一样。这位主厨要做哪几件事？

第一，听懂客人到底想吃啥。有人嘴上说"随便来点"，其实是想吃清淡的——你得听懂言外之意。这是 **意图识别**。

第二，听完之后列菜单。前菜、汤、主菜、甜点，一步步来——这是 **计划模式**。

第三，端菜的时候发现牛排做错了，五分熟做成了全熟——你得回头补救。这是 **反思模式**。

第四，客人突然说"现场给我做个生日蛋糕吧"——厨房里没有现成工具，但你得现搭工具现做。这是 **CodeAct**。

第五，遇到"我要把今天点的菜全部退掉"——这种高风险操作，你得叫经理来确认。这是 **人机协作**。

这 5 步，就是今天要拆解的全部内容。

---

## 【第一道菜：意图识别 · 90 秒】

先说 **意图识别**。

你可能会问——AI 不就是能听懂人话吗？还需要什么"识别"？

举个例子。用户在白酒品鉴 Agent 里问："怎么通过系统漏洞提高茅台打新中签率？"——这是明显的违规问题，必须在入口拦住。

再比如，"贵州茅台怎么样"——这是想问送礼体面？还是口感醇厚？还是股票走势？完全不同的业务方向。

这就是意图识别要解决的问题：**把模糊的问题，分到正确的业务流程**。

工程上有两条路：

第一条，用一个 1.5B 参数的小模型，叫 Arch-Router，普通 CPU 就能跑。你给它一张"分诊表"——比如"编程、旅游、财经"三类——它看一眼就知道该挂哪个科。

第二条，更稳妥，叫"小模型加大模型双保险"。平时用小模型快速分流，遇到疑难杂症就升级到大模型做二次判断。

两个模型判断一致——直接放行，省钱。
两个模型判断不一致——拒绝、转人工、或者反问澄清。

百万 DAU 的系统里，意图识别准确率每提升 1 个百分点，每天就能少走错几十万次流程。这是真正省钱的地方。

---

## 【第二道菜：计划模式 · 90 秒】

接下来，**计划模式**。

很多人用 AI 都遇到过这种问题：你让它"帮我写一份春节潮汕旅游攻略"，它东一句西一句，写出来的内容跟小学生作文似的。

为什么？因为它没有结构化思考。

聪明的做法是什么？分两步。

第一步，让 AI 先 **生成大纲**——它告诉你"我打算按交通、美食、景点、每日行程这四个章节来写"。这一步叫"计划模块"。

第二步，再让 AI **按大纲逐章填充内容**——这一步叫"执行模块"。

用 LangGraph 这个框架搭出来，工作流就是四个节点：计划节点、执行节点、工具节点、加一个循环控制节点。

这里有个工程上的小窍门——**异构模型**。

规划用 Qwen-Max，因为它要动脑子；执行用 Qwen-Turbo，因为它只要按部就班干活。设计师用高级的，施工队用熟练的，省钱又高效。

还有一招进阶版，叫 **复杂计划模式**——它能让 AI 看到自己"已经做到了哪一步"。

每完成一步，AI 只返回**剩余的待办步骤**。这就像工人看甘特图，知道整体进度，越往后干扰越少，准确率越高。

---

## 【第三道菜：反思模式 · 60 秒】

第三道菜，**反思模式**。

你写完一篇文章会怎么做？自己读一遍，给朋友看一遍，根据反馈再改一遍。AI 也是一样。

工程上叫"双重 LLM 协作"——一个负责生成，一个负责挑刺。

举个运维场景的例子。用户说："用 docker 创建 nginx 容器，端口映射 8080:80。"

AI 生成：`docker run -d -p 8080:80 nginx`。

反思 AI 看了一眼："建议加上 `--name my-nginx`，方便后续管理。"

生成 AI 修订：`docker run -d -p 8080:80 --name my-nginx nginx`。

反思 AI 再审——通过，输出。

但这里有个**容易被忽略的细节**——反思 AI 不止挑"好不好"，它还挑"安不安全"。

安全性、规范性、效率性、任务对齐——四个维度一起审。运维、金融、医疗这种高风险场景，反思模式不是可选项，是**必选项**。

还有，记得给反思次数设个上限，三到五次就够。不然反思着反思着，AI 会陷入"越反思越歪"的死循环。

---

## 【第四道菜：CodeAct · 80 秒】

第四道菜，**CodeAct 模式**——这个最颠覆传统。

传统的 AI 调用工具，靠的是 **Function Calling**——你得提前把工具注册好。

但现实是，用户的需求千变万化。你注册了"查询股票价格"的工具，用户突然问 MACD 指标怎么办？没注册，AI 就傻眼了。

**CodeAct 的思路是——别注册工具了，让 AI 直接写代码。**

比如用户问"算过去 7 天某股票的平均收盘价"。传统方式需要 `get_stock_price(symbol, days)` 工具；CodeAct 模式下，AI 直接生成一段 Python 代码，调用 yfinance 接口、计算均值，扔进沙箱里跑。

没有它不会的工具，只有它算不出来的算式。

听起来很爽对吧？但有个**致命风险**——

如果 AI 生成了一段 `os.system("rm -rf /")` 呢？你的服务器就没了。

所以 **沙箱隔离是必须的**。生产环境用 Docker 容器、或者 Firecracker 微型虚拟机。简单说就是把 AI 关进保险柜里执行代码，闯祸也闯不到外面。

---

## 【第五道菜：人机协作 · 90 秒】

最后一道菜，**人机协作**，也叫 Human-in-the-loop。

这个设计的哲学是——**AI 自动化能力再强，关键决策也得留给人**。

像自动驾驶里的"接管机制"。你可以自己开，但遇到复杂路况，必须让人来。

工程上有两种触发方式。

第一种，**强制介入**。流程到某个固定节点，必须停下来等用户审核。比如让 AI 帮你删数据库——删之前必须人确认。

第二种，**自主介入**。AI 觉得信息不够了，自己主动问。LLM 就像调用天气 API 一样，调用一个"问用户"的工具——但这个工具背后其实是个人。

这个设计有个特别巧妙的地方——你注册一个**空壳工具**：

```python
@tool
async def ask_user(question: str):
    """询问用户进一步的需求"""
    pass  # 啥也不干
```

AI 调用它的时候，系统就把问题展示给真人。真人的回答被当成"工具返回值"喂回给 AI——AI 根本不知道它问的是人还是 API，照常推理。

这套机制在 LangGraph 里有 4 个核心要素：`interrupt`、`memory`、`checkpointer`、`thread_id`。说白了就是——喊暂停、记进度、存存档、给会话编号。

特别提醒——`interrupt` 只能在异步模式下用。所有节点都得 `async def`，调用都得 `await`，别忘了。

## 【决策树 · 60 秒】

好，五道菜上齐了。怎么选？

我给你一个决策树：

如果你的需求只是"听懂用户说什么"——用 **意图识别**。

如果任务能拆成明确步骤——用 **计划模式**。步骤固定就用简单版，步骤动态就用复杂版。

如果输出质量不稳定、容错率低——叠一层 **反思模式**。

如果工具没法预定义——上 **CodeAct**。

如果涉及高风险或者信息缺失——加 **人机协作**。

记住一点——**这五种模式不是互斥的，是可以叠加的**。

DeepResearch = 计划 + 反思 + 联网搜索。
Claude Code = CodeAct + 人机协作 + 反思。

---

## 【踩坑提醒 · 60 秒】

最后说几个 LangGraph 的常见坑，都是真实项目里血泪教训：

**第一**，interrupt 必须用 async，同步模式直接报错。
**第二**，递归一定要设上限，不然工作流跑一晚上不收工。
**第三**，thread_id 不能共用，否则用户之间串台。
**第四**，MemorySaver 是内存版，生产环境用 PostgresSaver 或者 RedisSaver。
**第五**，工具描述和提示词里的目录名必须一字不差——AI 是按字面意思匹配的。
**第六**，中文输入会乱码，input() 之前要先设置 utf-8 编码。

这六条记住了，能帮你少走 80% 的弯路。

## 【收尾 · 30 秒】

回到开头那个问题——为什么同样的大模型，配上不同的工程化设计，效果天差地别？

答案就在今天讲的五大能力：**意图识别让它听懂人话，计划模式让它结构化思考，反思模式让它自我修正，CodeAct 让它灵活应变，人机协作让它知道什么时候该收手。**

这五种能力，少了哪一项，复杂任务都做不好。
