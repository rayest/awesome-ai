> 文档状态：Draft v1.0

> 适用范围：`awesome-ai/apps/auron-web` 及其配套后端、数据与 AI 服务  
> 关联文档：[AI Native PRD](./ai-native-prd.md)、[企业知识库建设方案](./enterprise-knowledge-base-plan.md)、[CRM 数据模型](./crm-schema.md)

---

## 1. 文档目的

本计划用于指导 Auron 从传统 CRM / 报价 ERP 页面升级为 AI Native 业务平台，明确：

- 为什么做、先做什么、暂时不做什么；
- AI 能力如何嵌入现有客户、打样、工艺和报价流程；
- 前端、后端、数据、模型、评测和安全分别需要建设什么；
- 每个阶段的交付物、依赖、验收标准和风险；
- 如何从“AI 给建议”逐步演进到“AI 在授权范围内执行任务”。

AI Native 在本项目中的定义不是“给每个页面加聊天框”，而是让 AI 具备以下闭环：

1. 自动获得当前业务上下文；
2. 理解用户目标和业务约束；
3. 查询跨表数据并形成可解释判断；
4. 调用确定性工具生成业务方案；
5. 在权限和审批约束下执行动作；
6. 记录证据、人工修改和最终结果；
7. 用真实反馈持续评测和优化。

## 2. 当前业务基线

### 2.1 已覆盖的业务链路

依据 `crm-schema.md`，现有 28 张活跃 `crm_*` 表覆盖：

```text
客户档案
  → 联系人
  → 跟进记录
  → 打样通知
  → 打样工艺单
  → 织造用料 / 纱线排列 / 尺寸要求
  → 报价单
  → 染色 / 缝制 / 辅料 / 其他费用
  → 报价总计与毛利
```

现有前端已经提供：

- 经营总览 Dashboard；
- 客户档案、联系人、跟进看板；
- 打样通知、打样工艺；
- 报价单；
- 产品主数据；
- 物料、机型、工序、染色、部位、部件、辅料、尺码和样品种类字典；
- 员工 / 角色、审计日志。

### 2.2 当前数据边界

现有数据不覆盖：

- 正式订单 / 合同；
- 库存流水与安全库存；
- 回款、应收和财务结算；
- 大货生产排程；
- 出运物流；
- 客诉和质量问题。

因此第一阶段 AI 不得：

- 声称能够判断真实库存是否充足；
- 自动承诺交期；
- 自动创建正式订单或合同；
- 自动判断回款风险；
- 编造不存在的生产、物流或质量数据。

涉及上述能力时，必须明确显示“当前数据不支持”，并提示需要补充的数据源。

## 3. 产品战略与优先级

### 3.1 用例筛选标准

优先建设同时符合以下条件的业务环节：

- 高频：每天或每周重复发生；
- 高价值：直接影响收入、毛利、交期或客户转化；
- 高认知负担：需要跨表查询、比较历史记录或处理模糊信息；
- 可验证：存在明确业务结果，可判断 AI 建议是否正确；
- 可控风险：可以先采用草稿、建议或人工审批模式；
- 数据已存在：不依赖当前系统缺失的事实。

### 3.2 功能优先级

| 优先级 | 能力 | 价值 | 数据成熟度 | 风险 | 计划阶段 |
|---|---|---:|---:|---:|---|
| P0 | AI 报价草稿 | 很高 | 高 | 中 | Phase 1 |
| P0 | AI 异常诊断与处置建议 | 很高 | 中高 | 低至中 | Phase 1 |
| P0 | 主数据归一与语义匹配 | 高 | 中 | 低 | Phase 0–1 |
| P1 | 打样 / 工艺 Copilot | 高 | 中高 | 中 | Phase 2 |
| P1 | CRM 跟进 Copilot | 中高 | 高 | 中 | Phase 2 |
| P2 | 自然语言业务操作 | 高 | 依赖工具层 | 中高 | Phase 3 |
| P3 | 预测、排程和自主执行 | 潜在很高 | 当前不足 | 高 | 后续评估 |

### 3.3 第一版产品组合

第一版不建设“万能助手”，只建设两条窄而深的闭环：

1. **AI 报价草稿**
   - 从当前客户、打样工艺和字典数据生成报价建议；
   - 使用确定性公式计算成本、税额和毛利；
   - 标出缺失条件、异常价格和风险；
   - 由报价员修改并提交审批。

2. **AI 异常处置**
   - 对 Dashboard 已识别的异常给出原因、影响范围和证据；
   - 给出可执行的处理方案；
   - 允许用户确认后创建跟进或重算任务；
   - 不自动修改正式报价。

## 4. AI Native 目标体验

### 4.1 从页面导航转向目标驱动

传统路径：

```text
用户打开多个页面
  → 自己查客户、工艺、报价和字典
  → 自己比较历史记录
  → 自己计算并创建下一步任务
```

AI Native 路径：

```text
用户提出目标或进入异常
  → 系统自动装配上下文
  → AI 查询证据并制定计划
  → 确定性工具完成计算
  → AI 给出方案、依据和不确定项
  → 用户确认
  → 系统执行并写入审计记录
```

### 4.2 统一交互原则

- AI 能力出现在任务发生的位置，不强制进入独立聊天页；
- 所有结论尽量附带来源和可跳转记录；
- 建议、草稿、正式数据使用不同视觉状态；
- 高风险动作必须在动作发生前再次确认；
- AI 无法获得必要数据时，停止并说明缺少什么；
- 不用自然语言替代结构化表单，AI 应填写表单草稿；
- 用户修改 AI 结果后，系统保存修改差异用于评测；
- 所有运行都可以重放、追踪和审计。

## 5. 目标技术架构

```text
┌──────────────────────────────────────────────────────┐
│ Auron Web                                             │
│ Dashboard / 报价 / 打样 / CRM / AI 运行详情            │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│ AI Orchestrator                                       │
│ 上下文装配 / 计划 / 模型调用 / 工具选择 / 状态机 / 重试 │
└──────────────┬──────────────────────┬────────────────┘
               │                      │
┌──────────────▼────────────┐  ┌──────▼─────────────────┐
│ Read Tools                │  │ Action Tools            │
│ 查客户、工艺、报价、字典   │  │ 建草稿、建任务、提交审批 │
└──────────────┬────────────┘  └──────┬─────────────────┘
               │                      │
┌──────────────▼──────────────────────▼────────────────┐
│ Domain Service / Rules Engine                         │
│ 权限、字段校验、成本公式、毛利、状态流转、幂等          │
└──────────────┬───────────────────────────────────────┘
               │
┌──────────────▼───────────────────────────────────────┐
│ CRM / Base / Future Backend DB                        │
└──────────────────────────────────────────────────────┘

横向能力：
- AI Run / Tool Call / Approval / Audit Log
- Prompt 与策略版本
- 离线评测与线上反馈
- 权限、脱敏、速率限制、成本控制
```

### 5.1 设计约束

- 模型不得直接读写数据库；
- 所有数据访问必须通过具名工具和业务 Service；
- 所有计算结果必须由确定性代码产生，模型只负责选择参数和解释；
- 写操作必须支持幂等键；
- 每次运行必须记录模型、提示词版本、工具输入输出摘要和最终状态；
- 敏感字段在进入模型前进行最小化和脱敏；
- 前端不能把完整业务对象无差别发送给模型。

## 6. Agent 与工具设计

### 6.1 初期 Agent 形态

Phase 1 优先采用单 Agent + 多个明确工具，不立即拆分多 Agent。

建议的单 Agent 职责：

- 理解用户目标；
- 选择读取工具；
- 判断信息是否完整；
- 调用确定性报价计算；
- 生成结构化建议；
- 请求用户确认写操作；
- 解释执行结果。

当出现以下情况时再拆分：

- 报价、工艺和 CRM 指令出现大量互相冲突的规则；
- Agent 频繁选错相似工具；
- 单次上下文过大，影响准确率或成本；
- 不同领域需要完全不同的权限和评测集。

### 6.2 Phase 1 工具清单

#### 读取类工具

| 工具名 | 作用 | 主要数据源 |
|---|---|---|
| `get_customer_context` | 获取客户、联系人、负责人和最近跟进 | 客户、联系人、跟进记录 |
| `get_sample_context` | 获取打样通知、尺寸、辅料和工艺 | 打样通知、打样工艺及子表 |
| `get_quote_context` | 获取报价基础、明细、成本和毛利 | 报价 6 张表 |
| `search_similar_quotes` | 按客户、产品、材料、工艺搜索历史相似报价 | 报价、产品、工艺 |
| `search_material_dictionary` | 匹配物料及单价 | 物料字典 |
| `search_process_dictionary` | 匹配机型、工序、染色和辅料 | 8 类相关字典 |
| `get_dashboard_exceptions` | 获取当前可验证异常 | 报价、跟进、打样 |
| `get_record_evidence` | 获取用于展示的字段级证据 | 统一数据访问层 |

#### 计算类工具

| 工具名 | 作用 | 要求 |
|---|---|---|
| `calculate_material_cost` | 计算原料成本 | 复用现有公式，不允许模型自行计算 |
| `calculate_machine_cost` | 计算机台费 | 参数和单位严格校验 |
| `calculate_quote_total` | 汇总织造、染色、缝制、辅料和其他费用 | 返回税前、含税、毛利 |
| `compare_quote_versions` | 比较历史报价与当前草稿 | 输出字段级差异 |
| `evaluate_margin_risk` | 按规则评估毛利风险 | 阈值配置化 |

#### 写入类工具

| 工具名 | 作用 | 审批策略 |
|---|---|---|
| `create_quote_draft` | 创建报价草稿 | 用户确认后执行 |
| `update_quote_draft` | 更新 AI 草稿 | 用户确认后执行 |
| `submit_quote_for_approval` | 提交审批 | 强制二次确认 |
| `create_followup_task` | 创建客户跟进任务 | 用户确认后执行 |
| `create_recalculation_task` | 创建报价重算任务 | 用户确认后执行 |
| `save_ai_feedback` | 保存采纳、拒绝和修改原因 | 可自动执行 |

Phase 1 禁止工具：

- 正式发送报价；
- 批准报价；
- 修改已生效报价；
- 删除业务记录；
- 承诺库存或交期；
- 创建正式订单；
- 向外部联系人发送消息。

### 6.3 工具契约要求

每个工具必须包含：

- 清晰、无歧义的名称和描述；
- JSON Schema 输入输出；
- 当前用户和租户；
- 权限检查；
- 输入字段单位；
- 业务校验错误；
- 幂等键；
- 可审计的记录 ID；
- 对模型友好的错误提示；
- 超时、重试和降级策略；
- 单元测试和集成测试。

## 8. 分阶段实施计划

## Phase 0：基础设施与基线（第 1–3 周）

### 目标

建立可安全调用业务数据的 AI 基础层，形成离线评测基线。

### 交付物

- Agent Orchestrator 基础服务；
- `ai_run`、`ai_tool_call`、`ai_feedback` 数据结构；
- 用户、租户和角色透传；
- 读取类工具第一版；
- 成本 / 毛利确定性计算 Service；
- Prompt 版本管理；
- 20–50 个历史报价黄金样本；
- 20 个异常诊断样本；
- AI 运行详情和基础审计页面；
- 超时、重试、限流和成本统计。

### 验收

- Agent 无法绕过 Service 直接访问数据；
- 同一用户只能查询其有权限的数据；
- 任意结论都可追踪到工具调用和业务记录；
- 报价计算结果与现有公式一致；
- 离线样本可以重复运行并输出对比报告。

## Phase 1：AI 报价与异常 MVP（第 4–8 周）

### 目标

让报价员和老板在真实工作中使用 AI 草稿与异常建议。

### 交付物

- 报价详情页“AI 生成报价草稿”；
- 缺失字段检查；
- 相似历史报价推荐；
- 成本、税额和毛利风险解释；
- Dashboard 异常详情抽屉；
- 原因、影响、证据和处置建议；
- 创建重算 / 跟进任务；
- 采纳、拒绝、修改反馈；
- 人工确认与审计日志。

### 发布策略

1. 内部开发账号；
2. 1–2 名报价员 Shadow Mode；
3. AI 草稿可见但不能写入；
4. 开放创建草稿；
5. 开放提交人工审批；
6. 扩大到全部报价员。

### 验收目标

- 报价草稿结构完整率 ≥ 90%；
- 确定性计算一致率 = 100%；
- AI 不编造记录 ID 或价格来源；
- 关键字段缺失时不提交正式建议；
- 报价员平均草稿准备时间降低 ≥ 30%；
- 异常建议证据覆盖率 ≥ 95%；
- 所有写操作均存在确认记录。

## Phase 2：工艺与 CRM Copilot（第 9–14 周）

### 目标

扩展到多模态打样资料和客户跟进工作流。

### 交付物

- 图片 / PDF / 文本需求抽取；
- 打样通知字段草稿；
- 工艺参数缺失检查；
- 相似产品与历史工艺推荐；
- 工艺版本差异比较；
- 客户 30 秒摘要；
- 下一步跟进建议；
- 跟进消息草稿；
- 创建跟进任务。

### 额外要求

- 附件解析必须保留来源页码或区域；
- 工艺建议必须标记“历史推荐”而非“标准答案”；
- 外发消息保持草稿状态；
- 不自动承诺价格、库存和交期。

## Phase 3：统一目标入口与低风险自动化（第 15–20 周）

### 目标

让用户通过自然语言发起跨页面任务，并逐步开放低风险自动执行。

### 示例任务

- “找出本周毛利低于 15% 且还没重新报价的客户。”
- “给这些报价分别生成重算草稿，但不要提交。”
- “整理 30 天未跟进且有历史报价的客户，按金额排序。”
- “比较 Q-0317-A 和上一次报价，解释价格变化。”

### 可自动执行的低风险动作

- 保存个人草稿；
- 创建内部待办；
- 生成内部摘要；
- 生成对比报告；
- 定时运行只读检查。

高风险动作继续要求人工确认。

## 9. 前端开发计划

### 9.1 通用组件

- `AiTriggerButton`：触发 AI 能力；
- `AiRunStatus`：排队、运行、等待确认、成功、失败；
- `AiSuggestionCard`：建议、风险、置信度；
- `EvidenceList`：证据引用与跳转；
- `ApprovalPanel`：动作预览与确认；
- `AiDiffView`：AI 草稿与原始数据差异；
- `AiFeedback`：采纳、拒绝和修改原因；
- `AiRunDrawer`：运行过程与工具调用摘要；
- `MissingContextPanel`：缺少字段和补充入口。

### 9.2 页面改造

#### Dashboard

- 异常卡增加“AI 诊断”；
- 展开后展示原因、影响、证据、建议动作；
- 增加每日经营简报；
- 所有数字与异常均可追溯到记录；
- 不展示虚构预测。

#### 报价详情

- AI 草稿入口；
- 信息完整度检查；
- 相似报价；
- 成本构成与毛利风险；
- 字段级差异；
- 保存草稿和提交审批。

#### 打样工艺

- 文件 / 图片解析入口；
- 缺失参数提示；
- 相似工艺推荐；
- AI 填充字段采用明显草稿状态。

#### CRM

- 客户摘要；
- 待跟进原因；
- 下一步动作；
- 消息草稿；
- 创建跟进任务。

### 9.3 状态要求

必须覆盖：

- 未启用；
- 无权限；
- 无足够数据；
- 排队；
- 运行中；
- 等待补充信息；
- 等待确认；
- 部分成功；
- 成功；
- 失败；
- 已取消；
- 结果过期。

## 11. 模型与成本策略

### 11.1 模型分层

| 任务 | 模型要求 |
|---|---|
| 意图分类、字段归一 | 低延迟、低成本模型 |
| 报价上下文判断、异常诊断 | 高推理能力模型 |
| 图片 / PDF 理解 | 多模态模型 |
| 内容摘要、消息草稿 | 中等能力模型 |
| 确定性计算 | 不使用模型 |

### 11.2 成本控制

- 只发送任务所需字段；
- 优先传摘要和稳定 ID，不传整张表；
- 字典和历史检索先在服务端筛选；
- 缓存不随业务变化的上下文；
- 设置单次运行最大工具调用次数；
- 设置单租户日预算和告警；
- 相同业务对象的重复请求支持结果复用；
- 保存 token、延迟和估算成本。

## 12. 安全、权限与审批

### 12.1 风险等级

| 等级 | 示例 | 策略 |
|---|---|---|
| R0 只读 | 查询客户、读取报价 | 自动执行 |
| R1 草稿 | 生成摘要、报价草稿 | 自动生成，用户可编辑 |
| R2 内部写入 | 创建内部任务、保存草稿 | 动作前确认 |
| R3 业务提交 | 提交报价审批 | 强制二次确认 |
| R4 外部 / 不可逆 | 发消息、批准报价、删除记录 | Phase 1 禁止 |

### 12.2 安全要求

- 使用当前用户身份调用工具；
- 租户数据严格隔离；
- 高风险字段不进入模型日志；
- 附件解析遵守访问权限；
- 防止提示词注入影响工具权限；
- 工具输入再次执行服务端校验；
- 前端确认不能替代服务端授权；
- 审批时校验动作内容是否发生变化；
- 超过置信度或失败阈值时转人工。

## 13. 评测体系

### 13.1 离线评测

#### 报价

- 字段抽取准确率；
- 字典匹配准确率；
- 相似报价召回质量；
- 缺失字段识别率；
- 计算一致率；
- 毛利风险识别率；
- 无依据数字率；
- 草稿结构完整率。

#### 异常

- 异常分类准确率；
- 原因证据一致性；
- 影响记录召回率；
- 建议动作可执行率；
- 不支持场景拒答率。

### 13.2 线上指标

- AI 入口使用率；
- 草稿采纳率；
- 字段修改率；
- 完全拒绝率；
- 平均完成时间；
- 人工节省时间；
- 运行成功率；
- 工具调用失败率；
- P50 / P95 延迟；
- 单次运行成本；
- 用户主动反馈；
- 高风险误操作数。

### 13.3 质量门槛

任何写能力上线前必须满足：

- 确定性计算一致率 100%；
- 权限越界测试 0 个未修复问题；
- 不可逆动作 0 个自动执行入口；
- 关键字段无依据生成率低于发布阈值；
- 失败时能够安全终止；
- 运行和审批日志完整。

## 16. 风险登记

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 主数据名称不统一 | 匹配错误 | 标准名称 + 别名 + 人工确认 |
| 公式来源不完整 | 报价错误 | 公式迁移到确定性 Service |
| 模型编造价格 | 毛利损失 | 所有价格必须来自工具证据 |
| 权限透传错误 | 数据泄露 | 服务端强校验、越权测试 |
| 用户过度信任 | 错误提交 | 草稿标识、证据、确认节点 |
| 延迟过高 | 放弃使用 | 流式状态、后台运行、模型分层 |
| 成本不可控 | 运营压力 | 预算、缓存、上下文裁剪 |
| 数据模型缺少库存 / 排程 | 错误承诺 | 明确拒答与能力边界 |
| 评测只看演示效果 | 上线不稳定 | 历史样本 + 线上反馈闭环 |
| 多 Agent 过早复杂化 | 维护困难 | Phase 1 单 Agent 优先 |

## 17. Definition of Done

单个 AI 功能完成必须同时满足：

- 有明确用户目标和使用入口；
- 有结构化输入输出；
- 有数据来源和证据；
- 有业务 Service 校验；
- 有权限校验；
- 有 loading、失败、取消、过期状态；
- 有人工确认策略；
- 有运行和工具调用日志；
- 有离线评测样本；
- 有线上指标；
- 有降级和关闭开关；
- 有中文业务日志；
- 有前端构建和相关测试；
- 有文档和变更记录。
