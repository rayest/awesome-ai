## 0 · 业务范围与边界

你这套 crm_* 数据模型覆盖的是 **接单前端** —— 从客户的初步接触到最终给出报价：

```
        客户档案
           │
           ▼
        跟进记录 ──────┐
                       │ (决定要不要打样)
                       ▼
                   打样通知（业务员把客户需求转给师傅）
                       │
                       ▼
                   打样工艺单（师傅填写工艺细节）
                       │            ▲
                       │            └── 子表：织造用料 / 纱线排列 / 尺寸要求
                       ▼
                    报价单        ▲
                       │            └── 子表：染色 / 缝制 / 辅料 / 其他 / 总计
                       ▼
                  （交付 / 大货）
```

**不覆盖**（Base 里没有对应表）：
- ❌ 订单 / 合同 / 出运物流
- ❌ 回款 / 应收
- ❌ 库存流水（物料表里也没有在用 / 最低库存字段）
- ❌ 客户投诉 / 质量问题
- ❌ 大货生产排程

→ 这意味着前端不能再发明这些业务页面。如果将来需要，往后端加新表，且**新表必须有 Name 列、字段定义、关系图一起进文档**。

## 8 · 注意事项 / 决策记录

| # | 决策 | 原因 |
|---|---|---|
| D1 | 字段 schema 是 SSOT，**前端不能新增字段、不改字段类型** | 与飞书 Base 一致，不引入新的事实源 |
| D2 | **3 张 `-不用` 表**永远不引用 | base 明确标记废弃 |
| D3 | **报价单 link crm_报价单_织造工艺表**字段 14 是 dead link | 表本身已废弃，公式 F13、F11 引用此字段，前端跳过 |
| D4 | **crm_人员信息表.测试的**字段不展示 | 开发残留 |
| D5 | **crm_字典_辅料配置表.文本** 字段名不展示 | 业务上应用"辅料名"或"辅料分类"显示 |
| D6 | **物料表没有在用 / 最低库存字段**，库存预警要做只能前端硬算（基于 follow up 或新加） | 现状如此，等业务加字段 |
| D7 | **审计日志** base 里没有 crm_日志表 | `/settings/audit` 是产品差异化功能，目前靠用户行为埋点，不依赖 crm |
