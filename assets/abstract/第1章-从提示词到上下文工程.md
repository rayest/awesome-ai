# 第1章《从提示词到上下文工程》内容总结与案例说明

## 一、章节主线

```
提示词工程（单点优化、单向线性）
        ↓  暴露三大痛点
上下文工程（系统化构建 + 动态管理）
        ↓  载体是 Agent
Agent 演进路线：
  Function Calling → ReAct → 深度思考型 Agent → Code Agent → 多 Agent（MAS）
```

---

## 二、提示词工程的三大局限

### 1. 局限说明
- **意图偏移**：单条指令难以捕捉复杂业务意图
- **长程记忆缺失**：多轮交互中容易丢失初始目标
- **逻辑断层**：面对多步骤、跨领域任务时缺乏结构化引导

### 2. 生活化案例
**去医院看病**
- 提示词工程模式：进医院直接对医生说"我不舒服"，医生无从下手
- 上下文工程模式：先描述"我最近 3 天咳嗽、低烧 37.8°C、之前有过敏史"，医生就能快速判断方向

## 三、1.1 上下文工程的核心思想

### 1.1.1 从购车决策看上下文的构建

#### 核心理念
把"一句话模糊提问"转换成"几百到上千字的结构化小作文"，让 LLM 输出更贴合真实需求。

#### 五大关键步骤
1. **意图识别**：识别用户真实目的
2. **需求分解**：把模糊问题拆成多个具体维度
3. **多源知识检索**：从专业平台、测评、用户反馈获取信息
4. **外部工具调用**：实地验证（试驾、对比）
5. **结构化记忆构建**：形成可复用的对比单元

#### 完整对比案例

**场景**：买车，想问"理想 L6 怎么样？"

| 维度 | 提示词工程（差） | 上下文工程（好） |
|------|----------------|----------------|
| 用户输入 | "理想 L6 怎么样？" | "10 年燃油车驾龄 / 30km 通勤 / 5 口之家 / 预算 20-25 万 / 自动泊车即可 / 看重底盘和安全性 / 计划开 8 年 / 已试驾，宁德时代和欣旺达混发电池" |
| 模型输出 | 通用参数堆砌，看完依然不会选 | 针对性建议：是否值得买、是否符合需求、需关注什么 |
| 决策价值 | 几乎为零 | 很高 |

### 1.1.2 上下文工程与 Agent 的关系

#### 工程化定义
- **"上下文"** = 输入模型的核心语义载体
- **"工程"** = 程序化、系统化地动态构建与持续优化这些载体

#### 精炼定义
**Agent 驱动的智能上下文构建**——不仅关注"说什么"，更强调"如何系统地准备和演化说的内容"。

#### ReAct 本质案例
**任务**：查询"今天北京天气"

| 步骤 | ReAct 推理过程 | 上下文的变化 |
|------|--------------|------------|
| 1 | 推理：用户想知道北京天气，需要调用天气工具 | 上下文：用户问题 + 推理文本 |
| 2 | 行动：调用 get_weather(location="北京") | 上下文增加：工具调用指令 |
| 3 | 观察：返回 `{"temperature": 22, "weather": "晴"}` | 上下文增加：观察结果 |
| 4 | 推理：基于 22 度晴天，给出穿衣建议 | 上下文增加：新的推理 |
| 5 | 输出：最终回答 | 最终答案 |

**关键洞察**：LLM 的每一次决策都极度依赖当前步骤产生的观察，并将其整合进上下文，驱动下一轮推理。这正是上下文工程的早期实践形态。

## 四、1.2 上下文工程的系统结构

把提示词工程比作"单兵作战"，上下文工程则是"多兵种联合作战"。

### 1.2.1 上下文构建的四大模块

#### 四大模块对照表

| 模块 | 一句话职责 | 工程化方法 | 生活案例 |
|------|----------|----------|---------|
| **感知** | 把自然语言问题转为结构化任务，并路由到合适模型 | 意图识别、模型选择 | 餐厅前台：听清你要什么菜，分配给合适的厨师 |
| **规划** | 生成思维链，串联执行步骤 | ReAct / Plan-Act / Sequential Thinking | 旅行规划师：先订机票、再订酒店、最后订餐厅 |
| **执行** | 调用外部 API 或运行代码获取实时数据 | 工具调用、Code Agent | 采购员：实际下单、付款、催物流 |
| **反思** | 对规划和执行结果检查、修正 | 自反思提示、Reflection Agent | 项目复盘会：检查哪些环节出了问题 |

#### 实际场景案例：让 AI 帮你做"周末带家人去杭州旅游 3 天"的攻略

- **感知**：识别意图 = "旅游规划"；路由到擅长中文的 Qwen-Max 模型
- **规划**：
  ```
  思考链：
  1. 查杭州 3 天天气
  2. 查热门景点（适合老人孩子）
  3. 查酒店（西湖附近、家庭房）
  4. 查高铁票
  5. 整合成攻略文档
  ```
- **执行**：
  - 调用 `get_weather(city="杭州", days=3)` → `{"weather": "晴转多云"}`
  - 调用 `search_attractions(city="杭州", suitable="family")` → 景点列表
  - 调用 `search_hotels(city="杭州", location="西湖")` → 酒店列表
- **反思**：
  - 检查：天气是否影响行程？酒店是否靠近景点？是否漏掉西湖游船？
  - 发现问题：第二天行程太满 → 调整节奏，增加休息时间

#### 不同规划方法对比案例

**任务**：规划一次海外出差 7 天的行程

| 规划方法 | 过程描述 | 优点 | 缺点 |
|---------|---------|------|------|
| **ReAct** | 边推理边行动，每步决定下一步 | 灵活，适应变化 | 缺乏全局观，可能绕远路 |
| **Plan-Act** | 先生成完整 7 天计划，再执行 | 全局观强，效率高 | 计划可能与现实脱节 |
| **Sequential Thinking** | MCP Server 支持多轮自主规划与反思 | 兼顾规划与灵活性 | 依赖工具支持 |

### 1.2.2 上下文管理的核心策略

#### 策略一：记忆管理

| 类型 | 存储方式 | 典型实现 | 案例 |
|------|---------|---------|------|
| **短期记忆** | 内存 | 对话历史变量 | 客服对话：用户上一句说了什么 |
| **长期记忆** | 文件 / 向量数据库 + RAG | Mem0（基于 Agentic RAG） | 跨会话记住用户偏好（咖啡少糖） |

**生活案例**：
- **短期记忆**：和邻居聊天，记得刚才聊过"小区停车位"
- **长期记忆**：认识多年的老朋友，记得他"对花生过敏"

#### 策略二：上下文卸载（Context Offloading）

**核心理念**：上下文并非越多越好，过剩信息造成噪声干扰。

#### 案例一：Cursor 分析大型项目

| 做法 | 效果 |
|------|------|
| **反例**：把全部代码 + 分析结果塞进对话窗口 | Token 爆炸、模型分心 |
| **正例**：把详细分析写入独立 `analysis.md`，对话窗口只保留路径和摘要 | Token 节省、模型聚焦 |

#### 案例二：Skills 三级渐进式加载

**场景**：Agent 有一个"运维巡检"技能

| 加载阶段 | 内容 | Token 消耗 |
|---------|------|-----------|
| **初始** | 仅加载技能元数据（名称："运维巡检"，描述："检查服务器健康状态"） | 几十个 Token |
| **确认调用** | 加载 `SKILL.md` 详细说明书 | 几百个 Token |
| **深度操作** | 读取关联资源文档（Linux 命令手册） | 按需加载 |

**类比**：手机 App 图标（不占内存）→ 点开后才加载完整功能模块。

### 1.3.1 简单工具调用 Agent

#### 案例一：Function Calling

**任务**：用户问"明天北京天气怎么样？"

```json
[
  {"role": "user", "content": "查询北京明天天气"},
  {"role": "assistant", "tool_calls": [{"name": "get_weather", "arguments": {"location": "北京"}}]},
  {"role": "tool", "name": "get_weather", "content": "{\"temperature\": 22}"},
  {"role": "assistant", "content": "北京明天气温 22℃"}
]
```

**特点**：
- 依赖预训练/微调，模型学会在合适情境下触发工具
- LLM 输出仅限于"工具调用指令"和"最终回答"
- **黑盒问题**：为什么选这个工具而不选那个？开发者看不到推理过程

#### 案例二：ReAct 五步循环

**任务**：让 AI 帮查"特斯拉 Model Y 在杭州的售价"

| 步骤 | 思考（Reasoning） | 行动（Acting） | 观察（Observation） |
|------|------------------|---------------|-------------------|
| 1 | 用户想查特斯拉 Model Y 杭州售价 | 无 | 用户问题 |
| 2 | 需要调用汽车报价工具 | 调用 `get_car_price(model="Model Y", city="杭州")` | 工具返回 `{"price": 26.39万}` |
| 3 | 拿到了价格，再查优惠政策 | 调用 `get_discount_policy(model="Model Y")` | 返回 `{"discount": "8000元补贴"}` |
| 4 | 查 4S 店位置 | 调用 `find_dealership(model="Model Y", city="杭州")` | 返回 3 家 4S 店地址 |
| 5 | 整合所有信息输出 | 无 | 最终答案 |

**对比 Function Calling**：
- ReAct 推理过程可见（黑盒变白盒）
- 零训练依赖，开源模型也能用
- 至今是 Claude Code 等系统的底层范式

### 1.3.2 深度思考型 Agent

#### ReAct 的四大局限案例

**任务**：让 AI 帮你做一个"季度市场分析报告"

| 局限 | 表现 |
|------|------|
| **强依赖 LLM 能力** | Qwen-7B 可能规划出错，导致整个报告跑偏 |
| **执行效率** | 几十次串行调用，耗时 30+ 分钟 |
| **提示词脆弱** | 模板改一个字，可能整个流程崩 |
| **局部最优** | 走到一半发现方向错了，Agent 还在按原路径走 |

#### 四种设计模式案例

**任务**：让 AI 完成"Q4 营收下滑原因分析"

| 模式 | 过程描述 | 适合场景 |
|------|---------|---------|
| **CodeAct** | LLM 生成 Python 代码：`pandas.read_csv("sales.csv")` + 数据分析脚本 | 运维自动化、数据处理 |
| **Plan-Act** | 先生成完整分析计划（5 步），再按步执行 | 复杂长程任务（如市场分析） |
| **Reflection** | 初步分析后，反思 Agent 检查："样本量够吗？结论是否片面？" | 金融分析、医疗辅助（对结果可靠性要求高） |
| **Human-in-the-Loop** | 在"得出最终结论"前暂停，让人类审核 | 资金转账、发布生产变更 |

#### 深度思考型 Agent 案例：Google Gemini DeepResearch

**任务**："综述近 3 年来某疾病治疗方案的临床进展"

```
1. 多样性规划：生成多个研究路径
   - 路径 A：从免疫疗法切入
   - 路径 B：从靶向药切入
   - 路径 C：从临床试验数据切入
        ↓
2. 并行联网搜索：3 个路径同时检索文献
        ↓
3. 反思评估：哪些路径信息质量高？哪些是低效分支？
   - 淘汰路径 C（数据陈旧）
   - 合并路径 A 和 B
        ↓
4. 多轮"思考-执行-反思"循环
        ↓
5. 合成最终报告：逻辑严谨、信息密集
```

**核心能力**：从聊天机器人进化为可运行数十分钟的深度研究助手。

### 1.3.3 Code Agent（代码即行动）

#### 经典对比案例：计算 1 加到 100

**任务**：给 Agent 配置一个加法工具 `calc(a, b)`，计算 1+2+…+100

| 范式 | 执行方式 | 调用次数 | 错误风险 |
|------|---------|---------|---------|
| **ReAct Agent** | 每轮只算一次 `calc()`，99 轮才能算完 | 99 次 | 极易因长程推理疲劳导致计数错误 |
| **Code Agent** | LLM 直接生成代码：<br>`result = 0`<br>`for i in range(1, 101):`<br>`    result = calc(result, i)` | 1 次 | 极低，一次性输出 |

**代码示例**：
```python
# Code Agent 生成的代码
result = 0
for i in range(1, 101):
    result = calc(result, i)
print(f"1加到100的和是: {result}")
```

**关键区分**：
- **Code Agent** ≠ Cursor / Claude Code（这些是 Coding Agent）
- Code Agent 是一种**范式**：以生成并执行代码为核心决策手段
- 效率比 ReAct 高很多倍，被认为是"当前阶段 Agent 的最优范式"

#### Code Agent 适用场景案例

| 场景 | ReAct 表现 | Code Agent 表现 |
|------|----------|----------------|
| 服务器 CPU 飙高排查 | 多次调用 top/ps/netstat，可能漏信息 | 生成 `shell` 脚本，一次性收集所有指标 |
| 数据清洗（10 万行 Excel） | 几十轮调用，Token 爆炸 | 生成 pandas 脚本，几秒搞定 |
| 自动部署服务 | 反复调用 kubectl，可能状态不一致 | 生成 K8s manifest，一次性 apply |

### 1.3.4 多 Agent（MAS / SubAgent）

#### 核心架构案例

**场景**：构建一个"智能财务助手"

```
Manager Agent（小赵，财务主管）
├── SubAgent 1：差旅费专员
├── SubAgent 2：业务招待费专员
├── SubAgent 3：办公采购专员
└── SubAgent 4：薪酬专员
```

**职责分工**：
- **SubAgent**：各自只处理一类报销，工具集、规则、记忆完全聚焦
- **Manager Agent**：接收用户请求 → 拆分 → 分发给对应 SubAgent → 汇总结果

#### 上下文污染案例（反例）

**场景**：单 Agent 处理所有财务任务

```
单 Agent 上下文 = [
  用户问题（"我上个月出差北京，报销差旅费 5000 元"），
  推理轨迹 1（识别为差旅费），
  工具描述列表（差旅/招待/采购/薪酬 共 50 个工具），
  工具调用历史（之前处理过 10 单），
  历史对话（上周谈过的招待费），
  中间结果（验票逻辑），
  ... (Token 爆炸)
]
```

**问题**：
1. 50 个工具描述全在上下文 → 模型选择困难
2. 各种报销规则混杂 → 推理易出错
3. Token 快速耗尽 → 任务中途被截断

#### 上下文隔离案例（正例）

```
SubAgent 1（差旅费）上下文 = [
  用户问题（"差旅费 5000 元"），
  仅 3 个差旅费相关工具，
  仅差旅费相关规则，
  专注处理差旅费
]

Manager Agent 上下文 = [
  任务分配指令（"差旅费交给 SubAgent 1"），
  SubAgent 1 的执行结果（"已通过"）
]
```

**对比效果**：

| 维度 | 单 Agent | 多 Agent（MAS） |
|------|---------|----------------|
| 上下文噪声 | 极高 | 极低 |
| Token 消耗 | 单 Agent 巨大 | 总和 3 倍但单 Agent 更小 |
| 推理准确率 | 容易出错 | 显著提升 |
| 任务中断风险 | 高 | 低 |

#### 现实生活类比

**过去的财务小赵**（单 Agent）：
- 桌上堆满所有类型的报销票据
- 一个人处理差旅、招待、采购等所有事务
- 容易出错、效率低

**升职后的小赵**（Manager Agent）：
- 差旅费 → 派给小张
- 招待费 → 派给小李
- 自己只做最终确认

**效果**：
- 桌面整洁 = 上下文纯净
- 各司其职 = 专业化 SubAgent
- 总工作量变大但每个个体更高效

## 六、章节核心金句

> **工程方法的出现，往往源于实际应用落地过程中暴露出的真实痛点。**

> **上下文工程的核心：通过工程化手段，把一句话提示词变成结构化的"小作文"。**

> **多 Agent 系统的本质：不是分工协作，而是通过上下文隔离，让 LLM 在更可控、更专注的环境中发挥最大效能。**
