# 从提示词到上下文工程：一句话帮你把"问得模糊"变成"答得精准"

## 速查表

| 模式/概念 | 一句话定义 | 核心比喻 | 典型场景 |
|----------|----------|----------|----------|
| 提示词工程 | 优化单次输入文本，让模型更好回答 | 给厨师递一张菜谱 | 简单问答、单轮交互 |
| 上下文工程 | 系统化构建、管理、演化 LLM 输入语境 | 给厨师备齐食材 + 锅具 + 灶台 | 多轮对话、复杂任务、Agent |
| 四大模块 | 感知 / 规划 / 执行 / 反思 的协同闭环 | 一家餐厅的前厅 - 厨房 - 上菜 - 点评 | 任何复杂 Agent 系统 |
| 记忆管理 | 用短期 + 长期记忆突破上下文窗口限制 | 大脑的"工作记忆"+"日记本" | 长对话、跨会话任务 |
| 上下文卸载 | 只把最相关信息塞进窗口，其余引用化 | 旅行只带"地图索引"而非整本攻略 | 复杂项目分析、多文件项目 |
| Function Calling | 预训练模型内建的工具调用能力 | 智能音箱的预设指令 | 标准工具集、结构化调用 |
| ReAct | 推理 + 行动 + 观察的零训练循环 | 边想边干的"试错法" | 通用 Agent、灵活任务 |
| Code Agent | 直接生成并执行代码作为行动手段 | 让程序员写脚本一次跑完 | 运维自动化、数据处理 |
| 深度思考型 Agent | 多轮思考 - 执行 - 反思 的认知闭环 | 研究员调研数小时写报告 | 深度研究、复杂决策 |
| SubAgent / MAS | 多专家 Agent + 协调者分工协作 | 财务总监把票据分给不同专员 | 跨领域、上下文污染严重场景 |

## 0. 全章比方：从"递纸条"到"开餐厅"

想象你要请朋友吃饭。

- **提示词工程** = 你写一张纸条："帮我推荐个菜。" → 朋友给个泛泛的回答。
- **上下文工程** = 你提前告诉朋友：你吃辣、预算 200、5 个人、附近 3 公里、忌口香菜、上次踩雷过哪家 → 朋友给出精准推荐。

而 **Agent** = 朋友变成了一个真正的"餐厅老板"：他会自己买菜（感知）、列菜单（规划）、叫厨师做（执行）、吃完问你好不好吃（反思）。

进一步，如果餐厅变大、菜系变多，单一老板忙不过来 → 出现 **多 Agent（SubAgent）**：川菜师傅、粤菜师傅、甜点师各管一摊，老板只负责协调。这就是 **MAS**。

## 1. 上下文工程的核心思想

### 1.1 购車决策案例：上下文如何"质变"LLM 输出

**类比**：朋友问"理想 L6 怎么样？" vs 你问"我是 10 年燃油车主，30km 通勤、5 口之家、预算 20-25 万、要求自动泊车、看重底盘和电池，已试驾 L6 配宁德时代/欣旺达混发电池" → 后者 LLM 给出的是私人定制建议。

**痛点**：单纯修改提问措辞收效甚微；真正解决问题要**补全信息 + 整合外部知识 + 实地反馈**。

**关键洞察**：上下文工程不是"润色 prompt"，而是"信息采集 + 过滤 + 组合"的系统化范式。

### 1.2 上下文工程与 Agent 的关系

**精炼定义**：

> "上下文"是输入模型的核心语义载体；"工程"是通过程序化手段对这些载体进行动态构建与持续优化。

**更进一步的凝练**：**Agent 驱动的智能上下文构建**——不仅关注"说什么"，更强调"如何系统地准备和演化说的内容"。

**关键案例**：ReAct 范式（2023 年）就是"动态构建 + 丰富对话上下文"的过程——LLM 始终基于不断演进的上下文进行推理。

## 2. 上下文工程的系统结构

### 2.1 四大模块

| 模块 | 核心职责 | 关键能力 | 典型方法 |
|------|----------|----------|----------|
| 感知 | 非结构化输入 → 结构化任务描述 | 意图识别、模型路由 | 意图分类、用户画像 |
| 规划 | 生成解决问题的思维链 | 任务拆解、步骤编排 | ReAct、Plan-Act、Sequential Thinking |
| 执行 | 调用外部 API / 运行代码 | 实时数据获取 | Function Calling、CodeAct、Code Agent |
| 反思 | 校验结果、发现问题 | 自我修正、闭环优化 | Reflection Pattern、Sequential Thinking |

**横向对比：ReAct vs Plan-Act vs Sequential Thinking**

| 模式 | 风格 | 适用场景 |
|------|------|----------|
| ReAct | 边想边干、动态决策 | 灵活任务、工具未知的探索 |
| Plan-Act | 先谋后动、严格按序 | 长程、多跳、需稳定的任务 |
| Sequential Thinking | MCP Server、多轮自主规划 + 反思 | 复杂任务、扣子等成熟产品 |

### 2.2 上下文管理的核心策略

#### 记忆管理

| 类型 | 实现方式 | 适用场景 |
|------|----------|----------|
| 短期记忆 | 对话历史存内存 | 当前会话上下文 |
| 长期记忆 | 文件 / 向量数据库 + RAG | 跨会话、用户偏好、专业知识 |

**典型项目**：Mem0（基于 Agentic RAG 设计）。

#### 上下文卸载

**核心理念**：上下文并非越多越好；过剩信息造成噪声干扰。

**三大策略**：

1. **写文件 + 引用**：Cursor 在复杂项目中，把详细分析写入独立 md，窗口只保留路径 + 摘要
2. **Skills 按需加载**：Agent 初始只加载元数据（名称 + 描述），确认需要时再加载 SKILL.md，进一步操作再加载资源文档
3. **引用而非堆砌**：用"指针"代替"全文复制"

## 3. Agent 技术发展路线图

### 3.1 演进全景

```
2023 Function Calling → ReAct → 2024 Plan-Act + Reflection
   ↓
2025 H1 深度思考型 Agent（DeepResearch、扣子）
   ↓
2025 H2 Code Agent（代码即行动）
   ↓
2026 Multi-Agent / SubAgent（上下文污染解决方案）
```

### 3.2 Function Calling vs ReAct

| 维度 | Function Calling | ReAct |
|------|------------------|-------|
| 训练依赖 | 需结构化训练数据 | 零训练、纯提示词 |
| 思考可见性 | 黑盒（仅输出指令） | 思考与指令一起返回 |
| 适用模型 | 头部厂商原生支持 | 任何 LLM 通用 |
| 代表产品 | 早期 OpenAI GPT | Claude Code 等 |

**ReAct 5 步循环**：
1. 理解用户问题并推理
2. 决定是否需要调用工具
3. 选择工具 + 生成参数
4. 接收执行结果
5. 基于新信息更新上下文

### 3.3 ReAct 的四大局限

1. **高依赖 LLM 自身能力**：推理 / 指令遵循 / 格式化能力不足就崩
2. **执行效率低**：每步都需 LLM 调用，复杂任务串行耗时长
3. **提示词脆弱**：模板微小变动影响行为
4. **易陷局部最优**：步进式决策缺乏全局视野，可能"原地打转"

### 3.4 四大 Agent 设计模式

| 模式 | 核心思想 | 典型场景 |
|------|----------|----------|
| CodeAct | 工具抽象为 Python 代码执行器 | 运维自动化、数据处理 |
| Plan-Act | 先全局拆解再按步骤执行 | 长程多跳任务 |
| Reflection | 独立反思 Agent 校验 + 修正 | 金融、医疗等高可靠性领域 |
| Human-In-the-Loop | 关键节点强制人工审核 | 资金转账、生产变更 |

### 3.5 深度思考型 Agent

**代表产品**：OpenAI DeepResearch、智谱沉思、扣子。

**Google Gemini DeepResearch 流程**：
1. 多样性规划：生成多个潜在研究路径
2. 并行联网搜索：获取外部信息
3. 反思评估：淘汰低效分支、保留高价值线索
4. 多轮循环：思考-执行-反思
5. 信息合成：输出结构化最终报告

**关键能力**：处理"分析某新兴技术的市场前景"等开放性、信息密集型任务。

### 3.6 Code Agent

**核心对比**（1 加到 100 的累加）：
- **ReAct Agent**：99 轮对话，每轮调一次 calc()，低效且易错
- **Code Agent**：LLM 直接生成 for 循环代码，一次性输出结果

**优势**：效率比 ReAct 高 N 倍。

**区分要点**：Code Agent ≠ Cursor / Claude Code 等"AI 辅助编程工具"（Coding Agent）；Code Agent 是"以代码生成为核心决策手段"的范式。

### 3.7 多 Agent（SubAgent / MAS）

**类比**：财务总监小赵 vs 把票据分给小张（个人支付）和小李（业务招待）。

**架构**：
- **SubAgent**：专注子任务，拥有干净聚焦的上下文
- **Manager Agent**：负责任务分解、调度、仲裁、整合

**本质洞察**：MAS 解决的不是"工具选择困难"，而是**上下文污染**——单 Agent 上下文混杂推理轨迹、工具描述、中间结果、历史记录等异构信息，造成认知噪声 + 窗口耗尽。

**关键收益**：通过上下文隔离，整体上下文长度扩展 N 倍，每个 Agent 在纯净环境中发挥最大效能。

## 4. 横向对比：决策树

### 4.1 何时用哪种 Agent 范式？

```
任务复杂度？
├── 简单问答 → Function Calling / ReAct
├── 需多步推理 → Plan-Act
├── 需高可靠性 → + Reflection
├── 需深度调研 → 深度思考型 Agent
├── 工具未知 / 临时操作 → Code Agent
├── 跨领域复杂任务 → MAS
└── 高风险操作 → + Human-In-the-Loop
```

### 4.2 上下文工程 vs 提示词工程

| 维度 | 提示词工程 | 上下文工程 |
|------|------------|------------|
| 对象 | 单次输入文本 | LLM 完整输入语境 |
| 手段 | 人工设计模板 | 系统化采集 + 过滤 + 组合 |
| 适用 | 单轮、简单问答 | 多轮、复杂任务、Agent |
| 价值 | 表达技巧 | 整套作战系统 |

## 5. 工程踩坑清单

| 坑 | 表现 | 解决 |
|----|------|------|
| 把提示词工程当万能解 | 简单改措辞收效甚微 | 补全信息 + 整合外部知识 |
| 上下文越多越好 | 信息噪声干扰、窗口耗尽 | 上下文卸载：写文件 + 引用 |
| 单一 ReAct 应付所有任务 | 复杂任务效率低、易局部最优 | 按场景选 Plan-Act / Reflection / Code |
| 单 Agent 处理跨领域任务 | 上下文污染、推理失败 | 拆 SubAgent + Manager 协调 |
| 提示词模板微调导致行为大变 | ReAct 提示词脆弱 | 加结构化字段 + 减少自由发挥 |
| Function Calling 黑盒 | 决策不可解释 | ReAct 模式让思考可见 |

## 7. 贯穿主线

> 上下文工程不是"写更好的提示词"，而是"用 Agent 系统化构建、管理、演化 LLM 的完整输入语境"。**提示词工程是单兵作战，上下文工程是多兵种联合作战。**
