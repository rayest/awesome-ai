## 【开场钩子 · 30 秒】

你有没有过这种经历？

跟 AI 聊了一下午，关掉浏览器，第二天打开，发现它**完全不记得你**了。

你问它："昨天咱们聊的那个减肥方案呢？"
AI 答："抱歉，我不清楚您在说什么。"

这就是 LLM 的**致命短板**——**无状态**。

每次对话都是独立计算，模型不会自动保留历史。

好，本章我们就来解决这个**核心痛点**。

## 【核心比喻开场 · 60 秒】

我打一个比方。

人类记忆有 3 类：
- **短期记忆** = 你现在的思绪，几秒到几分钟就忘
- **长期记忆** = 你的日记本 / 知识库，能存一辈子
- **永久记忆** = 你的直觉 / 技能，已经变成本能

AI Agent 的记忆也分 3 类：
- **短期记忆** = 上下文窗口（Token）
- **长期记忆** = 外部存储（向量库 / 文件 / 图库）
- **永久记忆** = 模型微调后的"本能"（本章不涉及）

本章只讲前两类——**让 Agent 跨会话、跨天、跨月记得住用户**。

好，我们正式开始。

## 【第一节 · LLM 为什么"失忆" · 90 秒】

先说 LLM 的**无状态特性**。

每次 API 调用，模型都会**重新加载所有参数**，历史对话**不会自动保存**。

四大核心问题：
1. **上下文丢失**——长对话中早期信息被截断
2. **个性化缺失**——无法持续捕捉用户偏好
3. **学习能力受限**——难以从过往交互中提取反馈
4. **逻辑一致性难维持**——多轮对话中前后矛盾

验证方法很简单：分两次问 LLM。

第一次问："大明的儿子叫小明。"
第二次问："小明的爸爸是谁？"

如果 LLM 答"不知道"——**恭喜你，验证了 LLM 的无状态**。

## 【第二节 · 短期记忆的 3 种优化 · 90 秒】

那短期记忆怎么搞？

最简单：把历史对话**重新注入**到 messages 数组里。

刚才那个实验，把第一次对话的 user 和 assistant 消息都加到第二次请求中，LLM 就**神奇地记住了**。

但这只是**最笨的方法**。Token 消耗大，成本高。

3 种优化策略：

**滑动窗口**——保留最近 N 轮对话，超过的丢掉。简单但粗放。

**上下文摘要**——用 LLM 把历史压缩为关键事实。Claude Code 提供 `/compact` 命令，OpenClaw 在上下文耗尽前自动压缩。

**滑动窗口 + 摘要**——近期保留原文，远期压缩为摘要。**最推荐**。

**医疗问诊的滑动窗口示例**：
- 用户说"我感冒了头疼流鼻涕"——保留
- 用户问"今天天气怎么样"——系统识别为无关，过滤
- 用户说"嗓子疼浑身冷"——保留

后续引用历史时，**自动过滤闲聊**，只留病情描述相关的核心语义。

## 【第三节 · 长期记忆的两大模式 · 80 秒】

短期记忆只解决当前会话。跨会话怎么办？需要**长期记忆**。

两大工程实现模式：

**外部文件持久化**——把任务成果保存为物理文件，窗口只保留指针。
- 扣子在执行复杂任务时，把中间状态写入存储空间
- Cursor 在项目分析时，把长篇内容存成文件，对话窗口只输出摘要
- DeepResearch 把研究结果存为 md 文件

**基于 RAG 的语义检索**——历史对话向量化，相似度搜索召回。
- Mem0 是当前最成熟的方案
- 打破线性对话限制
- 根据当前语义需求，**精准唤醒很久以前的相关记忆**
- 持续自我进化，记忆层不断更新和修正用户画像

两种模式**不互斥**，生产中通常**组合使用**。

## 【第四节 · Mem0：4 层记忆架构 · 90 秒】

Mem0 不是简单存储插件，而是为 LLM 应用提供**自我进化的记忆层**。

它已经是 AWS Agent SDK 官方记忆提供方，GitHub 近 5 万 Star。

核心架构有 4 层记忆：

| 层级 | 生命周期 | 典型场景 |
|------|----------|----------|
| **对话层** | 瞬时 | Agent 临时状态 |
| **会话层** | 短期 | 多步引导流程 |
| **用户层** | 长期 | 个性化推荐 |
| **组织层** | 全局 | 团队知识库 |

类比人类：
- 对话层 = 便签
- 会话层 = 日记
- 用户层 = 档案

Mem0 处理记忆分 3 阶段：
1. **语义提取**——LLM 当过滤器，从对话中提取 7 类结构化事实（个人偏好 / 重要信息 / 计划意图等）
2. **冲突仲裁**——新旧事实比对，4 种原子操作：ADD（新增）/ UPDATE（更新）/ DELETE（删除）/ NONE（跳过）
3. **混合存储**——Qdrant 向量库（语义相似度）+ Neo4j 图数据库（实体关系）

**为什么需要图数据库**：向量库解决"相似性"，图库解决"逻辑链"。比如查询"用户喜欢的饮品类别"，图存储能通过"咖啡→饮品"的关系链精准推理。

## 【第五节 · Mem0 实战 · 90 秒】

来，实战一下。

**步骤 1：环境部署**

Qdrant 向量库（语义相似度检索）：
```bash
docker run -d --name qdrant -p 6333:6333 -v /root/qdrant_data:/qdrant/storage qdrant/qdrant:latest
```

Neo4j 图库（实体三元组 + 复杂关系）：
- `-p 7474:7474` 图形化管理界面
- `-p 7687:7687` Bolt 协议端口
- 必挂载命名卷，否则容器被删数据就丢

**步骤 2：Mem0 配置**

4 个关键块：
- `llm`：用 qwen3-max 负责语义分析
- `embedder`：用 text-embedding-v3 生成 1024 维向量
- `vector_store`：Qdrant
- `graph_store`：Neo4j

调用 `Memory.from_config(config)` 实例化。

**步骤 3：添加 + 召回**

```python
memory.add([
    {"role": "user", "content": "大明的儿子叫小明"},
    {"role": "user", "content": "大明在阿里云做产品经理"},
], user_id="demo-user")

results = memory.search("小明的爸爸是做什么工作的？", user_id="demo-user", limit=3, rerank=True)
```

Mem0 自动完成**向量化存储 + 知识图谱构建**：
- 写入 Qdrant："大明在阿里云做产品经理工作"
- 写入 Neo4j：大明-父亲-小明、大明-就职于-阿里云

## 【第六节 · 高可用分布式部署 · 80 秒】

生产环境单点部署有风险——数据库实例宕机 → 整个记忆系统不可用。

**3 节点 Qdrant 集群**（P2P 协议 + 自动分片）：
- 必须 3 节点及以上，2 节点做不到多数派共识
- 通过 `--bootstrap` 参数从节点加入集群
- 节点端口不同（6333/6433/6533）便于本地调试

**Redis 缓存层**（加速热点记忆召回）：
- 写入时：写 Qdrant 后**立即清空** Redis 中对应缓存
- 召回时：**优先查 Redis**，命中直接返回，未命中回填

**3 个核心函数**：
- `get_redis_client()`——初始化连接
- `search_with_cache()`——缓存优先 + 回源
- `invalidate_user_cache()`——写时清空

**生产建议**：迁移到 Kubernetes，利用自动扩缩容 / 跨节点调度 / 健康探针 / 滚动更新。

## 【决策树 · 60 秒】

来，给个选型决策树：

你的需求是什么？
- **当前会话上下文** → 上下文注入
- **多轮对话但不跨会话** → 滑动窗口
- **长对话 + Token 敏感** → 滑动窗口 + 摘要
- **跨会话个性化** → Mem0
- **复杂关系推理** → Mem0 + Neo4j 图
- **企业高可用** → 3 节点 Qdrant + Redis 缓存

## 【踩坑提醒 · 60 秒】

最后几个工程坑：

1. **LLM 无状态被忽视**——"刚才不是说了吗"AI 答不知道，记得每次都附历史 messages。
2. **Qdrant 单点宕机**——3 节点集群 + 健康探针。
3. **Neo4j 容器被删**——必须挂载命名卷 `-v`。
4. **缓存与存储不一致**——写时清空 Redis，TTL 控制。
5. **2 节点 Qdrant**——单节点失效时无法恢复，必须 3 节点及以上。
6. **Mem0 配置项缺失**——llm + embedder + vector_store + graph_store 四件套齐全。

## 【收尾 · 30 秒】

总结一句话：

**Agent 记忆 = 短期记忆（上下文窗口）+ 长期记忆（向量库 + 图库 + 文件）**。

LLM 本身无状态，但 Agent 可以通过**外部存储 + 精准召回**让 LLM 看起来"记得住"用户。
