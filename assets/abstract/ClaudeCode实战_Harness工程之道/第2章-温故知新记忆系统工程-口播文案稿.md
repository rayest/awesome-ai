# 温故知新：记忆系统工程实践 · 口播文案稿

> 适用场景：AI 工程师内部分享 / 技术播客
> 预计时长：约 10 分钟
> 建议语速：220~240 字 / 分钟
> 阅读对象：使用 Claude Code 的开发者 / Tech Lead

## 【开场钩子 · 30 秒】

你有没有过这种崩溃时刻——让 AI 写一个用户登录接口，它给你用 Express 写了一段代码；你纠正后第二天开新会话，它又用回了 Express。一模一样的错误，连 npm install 都给你复制粘贴过来。AI 失忆了？

真相是：它不是失忆，而是每次开会都是它的"入职第一天"。今天这一期，我们就把这个痛点的根治方法一次说透。

## 【失忆症根因 · 60 秒】

先说根因。所有大语言模型都有个共性特征——会话间缺乏持久状态。每当你关闭会话再开新会话，模型面对的就是一张完全空白的画布。

它默认的"约定"来自训练数据——全网公开代码的统计分布。在这个分布里，Express 比 Fastify 出现得多，npm 比 pnpm 普遍，JavaScript 比 TypeScript 样本量大。所以缺乏项目上下文时，模型自然倾向"最主流"方案——而这恰好撞上你团队的红线。

Claude 缺失的不是能力，而是上下文。**既然每次都是入职第一天，我们就为它准备一份专属的入职手册**——这就是 CLAUDE.md。

## 【五层记忆体系 · 90 秒】

这份手册该放在哪？答案是：不只一个地方。Claude Code 的记忆系统是分层架构，从全局到局部共 5 个层级。我们用"公司制度文件"来类比。

**企业级**是集团总部发布的《集团行为准则》——全员必须遵守的铁律，IT 管理员配置，个人无法覆盖。
**用户级**是你的个人偏好档案——放在用户主目录，对所有项目生效，比如回复语言、提交格式。
**项目级**是项目组的开发手册——放在项目根目录，提交到 Git，团队共享。
**规则级**是专项操作指引——`.claude/rules/` 目录下的多个 Markdown 文件，每个专注一个主题。
**本地级**是私人便签——`CLAUDE.local.md`，自动加入 .gitignore，只有自己看到。

冲突时怎么办？遵循"具体性优先"——项目级"2 空格"压过用户级"4 空格"。但企业级是绝对红线，任何下层配置都无法绕过。

## 【写作范式：少即是多 · 90 秒】

CLAUDE.md 怎么写？核心原则四个字：**少即是多**。

为什么？因为上下文窗口是张面积固定的办公桌。你写的每一行都在每次对话初始化时被完整注入，消耗 Token 配额。根据 Anthropic 的工程实践数据，System Prompt 已经内置约 50 条指令；用户自定义 CLAUDE.md 加入后，若总指令数突破 150 条，遵循质量会显著衰减。

怎么筛选内容？用"三问框架"——

第一问 WHY：揭示决策背后的逻辑。比如"我们选 Fastify 因为它的 schema-based validation 与类型安全优先战略一致"。
第二问 WHAT：划定行为边界。比如"必须用 pnpm；禁止 npm/yarn"。
第三问 HOW：固化操作流程。比如"数据库迁移请运行 pnpm db:migrate"。

大多数规则只需要 WHAT 层面一句话即可。只有容易被误解的关键决策，才值得补充 WHY。

**黄金检验标准**：如果删去某条规则后 Claude 依然能做出正确行为，那这条规则就不该出现。

## 【正面 vs 反面对比 · 60 秒】

来看一个反例：写"代码要写得好一些，注意性能"、"测试要全面"、"请遵循最佳实践"——这种正确的废话，对 AI 几乎等于无效信息。什么叫"好一些"？什么叫"全面"？什么叫"最佳实践"？

再看正面案例：项目级 CLAUDE.md 应该精确到具体工具名、命名约束、目录结构。比如 Fastify 4（不使用 Express）、pnpm 8（不使用 npm/yarn）、API 统一返回 `{ success, data, error }` 格式、错误码用 UPPER_SNAKE_CASE。

## 【条件化规则系统 · 90 秒】

随着项目演进，单一 CLAUDE.md 必然臃肿。这时候就要用 `.claude/rules/` 做模块化拆分。

关键机制是 YAML Frontmatter 的 `paths` 字段。比如 `testing.md` 文件声明 `paths: ["**/*.test.ts"]`，那么只有当 Claude 编辑测试文件时这条规则才激活；编辑业务代码时它就静默存储，不消耗任何 token。

这就像企业不同岗位配专属手册——前台遵循前台流程，仓库恪守仓库规范，不需要人人背诵全公司制度。

实战中，前端工程师维护 `frontend.md`、后端维护 `api-design.md`、DBA 把控 `database.md`——各司其职，大幅降低合并冲突概率。

⚠️ 一个常踩的坑：规则文件不配 paths，会被系统视为全局无条件规则强制加载，违背懒加载设计。

## 【3 种项目实战 · 80 秒】

最后看三种典型项目的 CLAUDE.md 配置差异。

**React 前端**：最痛的是状态管理混乱——Claude 可能混用 Redux、Context、本地 State。解法是在 CLAUDE.md 里画一张"状态管理决策树"：服务器数据用 TanStack Query、全局客户端状态用 Zustand、组件局部状态用 useState。这把工具清单升级为"场景到工具的映射逻辑"。

**Node.js 后端**：最痛的是分层架构职责边界。解法是明确"严禁跨层调用 Prisma"——这条铁律直接决定可维护性。

**Python 数据**：最痛的是数据处理不可逆性——滥用 inplace=True 会破坏可追溯性。解法是禁止 inplace=True、缺失值统一用 pd.NA。

三份文件的共同特征：篇幅二三十行，每行都是干货，**摒弃 Claude 早已熟知的通用知识，精准聚焦于本项目独有的约定和抉择**。

## 【决策树 · 60 秒】

最后给一个实操决策树。

问：你的项目规模多大？小型个人项目：一份 CLAUDE.md 就够。中型团队项目：项目级 + 用户级 + 本地级。大型企业项目：5 层全开 + 规则级模块化。

问：用什么工具辅助？项目初始化用 `/init` 命令；对话中发现规范漏洞用 `/memory` 命令实时更新。

## 【踩坑提醒 · 60 秒】

高频踩坑有 6 个：CLAUDE.md 过长突破 150 条临界值；写"代码要写得好"这种模糊要求；用 CLAUDE.md 装 React 基础浪费 token；规则文件不配 paths 强制加载；用户级和项目级冲突未深思；团队规范变更后忘记同步更新文件。

## 【收尾 · 30 秒】

到这里，记忆系统的工程实践就讲完了。CLAUDE.md 看似只是份配置文件，实则是把团队智慧沉淀为机器可读规则的工程化思维。下一章，我们将正式踏入 Skills 的广阔世界——如果说 CLAUDE.md 是入职手册，那么 Skills 就是为它量身定制的专业培训教材。
