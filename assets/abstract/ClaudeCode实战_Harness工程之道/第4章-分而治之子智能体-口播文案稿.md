# 分而治之 · 口播文案稿

> 适用场景：技术学习者、AI Agent 开发者、关注 Harness 工程的工程师
> 预计时长：10 分钟
> 建议语速：220~240 字/分钟
> 阅读对象：想让 Claude 从"全能实习生"升级为"成熟 CEO"的开发者

---

## 【开场钩子 · 30 秒】

你让 Claude 帮忙看 500 行错误日志，它直接被噪声淹没了——越看越乱，最后给你一句"看起来一切正常"。这是不是你的真实经历？今天这章解决的就是这个灵魂问题：怎么让 Claude 不再"事必躬亲"，而是学会当老板？

## 【CEO 委派模型 · 60 秒】

想象一下，CEO 自己跑去车间拧螺丝，那叫体验生活，不叫管理公司。Claude 主对话就有点像这种 CEO，啥事都自己干，500 行日志直接塞进上下文，注意力被稀释得干干净净。

解决方案：让 Claude 当 CEO，下面带一帮"独立办公室员工"——每位员工有自己的办公空间（独立上下文窗口）、专属的权限清单（工具白名单）、明确的工作职责（YAML 配置文件）。CEO 把脏活累活委派出去，自己只看员工的精炼汇报。这就是子智能体（Subagent）的核心思想。

子智能体有三大核心价值：隔离——500 行日志噪声永远锁在子智能体内部；约束——给 code-reviewer 只配 Read/Grep/Glob 三个只读工具，它物理上就没法改文件；复用——写好的子智能体定义可以团队共享、新人入职即用。

## 【第一节 · 5 种子智能体模式 · 90 秒】

子智能体不是一种，而是五种层层递进的能力谱系。

第一种是**只读型**——审计员模式，只配三个只读工具，专门用来做代码审查、安全审计、架构分析。配合 `permissionMode: plan` 还能让整个会话被系统标记为只读，比工具白名单更严格。

第二种是**执行型**——信噪比优化器。需要调用 Bash 跑命令，但必须用前缀白名单，比如 `Bash (npm test:*)` 表示只能跑 npm test 开头的命令。这种子智能体是日常开发最常用的。

第三种是**并行型**——3 个侦探同时办案。安全审计、代码质量、性能分析三件套同时开干，前提是子任务之间必须完全独立。这种模式对应软件工程里的 MapReduce。

第四种是**流水线型**——bug 修复的标准装配线。四个子智能体各管一段：bug-locator 只读定位、bug-fixer 读写修复、bug-verify 只读验证、bug-report 只读分析。每个阶段工具权限截然不同，最小权限贯穿全链。核心技术是"交接契约"——上一个阶段输出的格式，必须严丝合缝匹配下一个阶段的输入。

第五种是**团队型**——自组织协作。Team Lead 当项目经理，多个 Teammate 长期存续，Task List 同步进度，Mailbox 异步通信。成本最高，仅在真正需要多轮协作的复杂重构场景下使用。

## 【第二节 · 子智能体与 Skills 的协作 · 90 秒】

有个关键问题：子智能体的专业知识从哪来？答案是 Skills，但有两种完全相反的组合方向。

**方向 A：子智能体预加载 Skill**。打个比方，bug-fixer 是位"专家员工"，启动时自动加载 secure-coding 这本"操作手册"作为自己的领域知识。子智能体的 .md 正文是 System Prompt，Skill 全量内容作为知识上下文注入。这种模式适合流水线中需要长期维持角色状态的专家。

**方向 B：Skill 派生子智能体**。完全反过来，Skill 才是主角。通过 `context: fork` 配置，Skill 触发时会自动派生一个隔离的子智能体来执行任务，子智能体只是临时执行容器。这种模式适合一次性重型任务，比如 codebase-health-check 扫描整个代码库。

判断口诀：**谁是主角**？角色驱动、需多轮交互的选方向 A；单次性、严格隔离、复用流程的选方向 B。

## 【第三节 · Token 经济学 · 60 秒】

有人会问：多开一个子智能体不是更贵吗？答案是反直觉的——在大多数场景下，合理使用子智能体反而**降低**总 Token 消耗。

举个例子：主对话直接跑测试产生 10000 Token 的噪声，然后进行 5 轮后续对话，每轮都要重复传输这 10000 Token，累计 125000 Token。但如果把测试委派给 test-runner 子智能体，10000 Token 锁在子智能体内部，主对话只接收 100 Token 的摘要。最终总计 116500 Token，**节省 6.8%**。

而且对话轮数越多，节省比例越高——10 轮可达 10% 以上，20 轮可达 20% 以上。

但 Anthropic 的 Prompt Caching 削弱了这个优势——缓存命中只收 10% 价格。纳入缓存后，节省率从 6.8% 回落至 1%~2%。

不过别忘了，子智能体的真正价值不只是省钱——还有**上下文窗口保护**和**响应质量提升**这两项是 Prompt 缓存替代不了的。

## 【第四节 · 4 大软件工程原则 · 80 秒】

子智能体架构并不是凭空创造的新概念，而是把软件工程几十年的经典智慧迁移到 AI Agent 领域。

**单一职责原则**：每个子智能体只做一件事——code-reviewer 只审查、test-runner 只跑测试、bug-locator 只定位。

**舱壁模式**：源自造船工程的水密隔板。如果 log-analyzer 子智能体遇到死循环、幻觉爆发、格式崩溃，异常被严格封锁在子智能体沙箱内，主系统稳定运行。这正是微服务架构防级联雪崩的同一思想。

**MapReduce 模式**：并行型子智能体就是分布式计算里的 MapReduce。主智能体负责 Map 阶段分发任务、Reduce 阶段汇总结果，子智能体就是工作节点。

**责任链模式**：流水线型子智能体对应 Unix 管道、Java Servlet Filter Chain、Node.js Express 中间件——每个处理器只专注自己的事，把结果传给下一个环节。

## 【决策树 · 60 秒】

什么时候该用哪种模式？给你一个简化判断：

- 只需观察代码/数据 → 只读型
- 高噪声输入需精炼输出 → 执行型
- 多个独立维度需同时分析 → 并行型
- 多阶段有明确依赖关系 → 流水线型
- 子智能体间需多轮通信与协调 → 团队型

大多数日常开发场景，只用只读型和执行型就够。**切记不要过度设计**。

## 【踩坑提醒 · 60 秒】

四个生产陷阱必须提前了解：

第一，**CLAUDE.md 继承优先级**：子智能体定义文件 > 项目级 CLAUDE.md > 用户级 CLAUDE.md，子智能体可以覆盖项目级规则。

第二，**报文传输模式**：子智能体之间无法直接通信，必须靠主智能体显式传递。务必让每个阶段的输出格式**高度结构化**。

第三，**中断恢复机制**：子智能体被中断后内存数据会丢失，必须把中间产物持久化到 `.claude-work/` 目录，新启动的子智能体才能接续工作。

第四，**嵌套深度控制**：子智能体嵌套超过 3 层，调试难度指数级上升。如果出现这种情况，说明任务分解粒度有问题，应重新扁平化设计。

## 【收尾 · 30 秒】

子智能体是 AI Agent 时代的"委派式思考"——把 Claude 从"事必躬亲的全能实习生"升级为"懂得分工的成熟管理者"。一旦你掌握了这五种模式及其背后的软件工程原则，你便拥有了在任何新场景下构建适配性子智能体架构的直觉与能力。
