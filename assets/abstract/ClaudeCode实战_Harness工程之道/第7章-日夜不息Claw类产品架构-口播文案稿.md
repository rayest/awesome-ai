# 日夜不息 Claw 类产品架构 · 口播文案稿

> 适用场景：技术视频 / 课程讲解 / 公众号口播  
> 预计时长：9~11 分钟  
> 建议语速：220~250 字/分钟  
> 阅读对象：Agent 开发者、AI 工程实践者

## 【开场钩子 · 30 秒】

你有没有想过一个问题：

为什么 Claude Code 很强，但它并不会自己在飞书里 24 小时值班？

你得打开它，给它任务，它才开始干活。可真正的数字员工不应该是这样。它应该能收消息，能定时提醒，能自己排队处理任务，出错后还能恢复。

这就是第七章要讲的 Claw 类产品。

说白了，它不是在发明一个更聪明的模型，而是在给 Code Agent 装一套工程外骨骼。

## 【核心比喻开场 · 60 秒】

我们先打个比方。

普通 Code Agent 像临时顾问。你把它叫过来，它帮你解决一个问题，然后就下班。

Claw 类产品像一间 24 小时有人值班的自动化办公室。

前台负责接电话，秘书负责排日程，后厨负责真正干活，档案室负责记录每一次来往。

在 NanoClaw 里，前台就是飞书、Telegram 这些 Channel；秘书是 task-scheduler 和 group-queue；后厨是 Docker 容器里的 Agent Loop；档案室是 SQLite。

所以这一章的主线很简单：

怎么把一个一次性会话里的 Code Agent，变成一个始终在线、能接消息、能跑定时任务、还能安全隔离的数字员工？

## 【第一节 · 为什么从 NanoClaw 入手 · 90 秒】

先看 OpenClaw 和 NanoClaw。

OpenClaw 是完整产品，能力很多，可以理解成一个开源的办公自动化 Agent 平台。但问题是，它太大了。

如果你上来就读完整 OpenClaw，就像第一次学汽车结构，直接拆一辆重卡。不是不能学，是成本太高。

NanoClaw 的价值就在这里。

它是一个精简版，大约 4000 行 NodeJS，规模小到可以一次塞进大模型上下文里。这样你可以让模型帮你读完整项目，快速建立系统理解。

这一章里有个非常关键的公式：

现代 Agent 等于 Code Agent 加 Skills，再加外层 Harness。

Code Agent 负责推理和执行，Skills 给它标准化能力，Harness 负责在线、调度、路由、隔离和恢复。

所以 OpenClaw 看起来像能写 PPT、回邮件、推送晨报，本质上不是模型突然长出了办公系统能力，而是外层工程系统把它接进了真实工作流。

## 【第二节 · 一内一外：容器内和容器外 · 90 秒】

NanoClaw 的架构可以用四个字概括：一内一外。

容器内，是执行层。

这里有 Agent Loop，有 Skills，有浏览器，有工作目录。它像一个封闭工作间，Agent 可以在里面读文件、跑代码、打开网页、完成任务。

容器外，是管理层。

这里有八个模块：编排器、group-queue、Channels、IPC Watcher、Router、container-runner、task-scheduler、SQLite。

你可以把它想成酒店系统。

客人不会直接冲进后厨。他先找前台。前台登记需求，排队，转给后厨，后厨做好以后，再由前台送回给客人。

同理，用户在飞书里发消息，不会直接进入 Agent。消息先进入 Channel，再写入数据库，再由主循环判断是否要唤醒 Agent，再通过队列分配给容器。

这样设计最大的好处是安全。

如果 Agent 在容器里执行了危险操作，影响范围被限制在容器内部。它不会随便碰宿主机，也不会轻易看到其他群组的数据。

## 【第三节 · 六条功能链路 · 120 秒】

接下来讲本章最重要的六条链路。

第一条，消息接收与持久化。

飞书或者 Telegram 来了一条消息，Channel 先把它转成统一消息对象，然后立即写入 SQLite。

重点是立即写入。

为什么？因为 Agent 可能崩溃，容器可能重启，服务也可能中断。只要消息先落库，系统就能靠 `lastAgentTimestamp` 找出哪些消息还没处理。

这就是 at-least-once，至少处理一次。

第二条，主循环与消息处理。

主循环会定期扫描数据库，看有没有新消息。群聊里通常不会让 Agent 看所有闲聊，所以有触发词机制，比如只有 @ 机器人时才激活。

但它又不是只处理 @ 那一句。

它会把触发前积累的一段上下文也带进去。这样既避免 Agent 监控无关闲聊，又能让它理解完整意图。

第三条，定时任务。

NanoClaw 支持 cron 表达式、固定间隔和一次性任务。比如每天晚上十点提醒你写日报，或者每五分钟检查一次状态。

这里有个细节：调度要锚定计划时间。

如果系统停机两小时，恢复后不应该把错过的任务一口气补跑。正确做法是沿着原来的计划节奏继续，避免任务扎堆和时间漂移。

第四条，并发控制。

不是每条消息都立刻拉一个容器。group-queue 会先判断有没有空闲容器，有就把消息 pipe 进去，没有就排队，必要时再启动新容器。

默认最多五个 Docker 容器。容器空闲超过一段时间，就发送 `_close`，让它优雅关闭。

第五条，IPC 通信。

容器内外不是直接互相乱调接口，而是通过三个目录传递信息。

`input/` 是主进程给容器的输入。

`messages/` 是容器写回来的回复。

`tasks/` 是容器请求创建、暂停、取消定时任务的地方。

目录路径本身就成了权限边界。每个群组只能访问自己的目录。

第六条，结果路由。

Agent 做完以后，把结果写入 `messages/`。IPC Watcher 监听到新文件，Router 找到消息来源，再交给对应 Channel 发回飞书或 Telegram。

这样，一个闭环就完成了。

## 【第四节 · 飞书 Channel 怎么开发 · 90 秒】

本章还讲了一个实战：给 NanoClaw 开发飞书 Channel。

这里要注意，难点不在飞书 SDK。

飞书官方 SDK 可以帮你处理平台通信，但真正麻烦的是：怎么把飞书事件接进 NanoClaw 内部系统？

也就是说，你要把飞书消息转成统一消息对象，写入数据库，接入路由，再确保回复能从 Router 回到飞书。

作者采用的是一种 Vibe Coding 路线。

先用 Plan 模式梳理 Telegram Channel 的通信流程。搞清楚它怎么收消息、怎么入库、怎么交给容器、怎么路由回复。

然后让 OpenCode 模仿 Telegram 的实现，在 `src/channels/` 里写飞书 Channel。

最后人工审查，修鉴权、事件格式和边界问题。

这个思路很实用：不是让 AI 空写代码，而是先给它一个可模仿的成熟样板。

## 【第五节 · 国内模型与部署 · 80 秒】

另一个重要细节，是接入国内大模型。

NanoClaw 的容器内 Agent 需要访问模型服务。如果把 Key 和模型地址写死在代码里，既不安全，也不灵活。

所以更好的方式是 `.env` 加环境注入。

比如配置 `ANTHROPIC_BASE_URL`、`ANTHROPIC_MODEL`、Token，然后通过 SDK 的环境参数注入容器。这样你只改配置文件，就能切换到兼容 Anthropic API 的模型服务。

部署也很直接。

先编译 Docker 镜像，再安装依赖，再编译 TypeScript，最后启动服务。

启动后，如果没有消息，可能看不到容器在跑。因为系统是按需拉起容器。

你在飞书里发一句“你好”，容器被拉起，Agent 回复。你再发“每天晚上提醒我站起来活动”，task-scheduler 创建任务，到点推送提醒。

这才说明整条链路跑通了。

## 【决策树 · 60 秒】

怎么选？

如果你只是想用 AI 写代码，用 Claude Code 或 Cursor 就够了。

如果你想学习 Claw 类产品架构，或者做一个自己的 IM Agent，选 NanoClaw。

如果你要企业级办公自动化，多平台、多群组、多任务，那才考虑完整 OpenClaw。

如果你的重点是“让 Agent 接入飞书”，就看 Channel。

如果你的重点是“让 Agent 主动提醒和定时执行”，就看 task-scheduler。

如果你的重点是“多个群同时处理”，就重点研究 group-queue 和容器池。

## 【踩坑提醒 · 60 秒】

最后总结几个坑。

第一，IM 协议不统一。不要每个平台写一套业务逻辑，要用 Channel 抽象。

第二，消息不要先进内存再说。先落 SQLite，否则崩溃就丢。

第三，触发词要处理好。太宽会侵犯群聊隐私，太窄又会切断上下文。

第四，容器不能无限拉起。要有并发上限和空闲关闭。

第五，定时任务不能恢复后疯狂补跑，要锚定计划时间。

第六，不要把 API Key 写死在代码里，要用 `.env` 和环境注入。

第七，IPC 目录要按群组隔离。否则一个群的 Agent 可能看到另一个群的数据。

## 【收尾 · 30 秒】

这一章的核心可以压缩成一句话：

Claw 类产品不是让模型变聪明，而是给模型配上一套值班系统。

它让 Code Agent 从一次性工具，变成一个可以长期在线、接收消息、主动调度、安全执行、可靠恢复的数字员工。

如果前六章讲的是 Agent 的能力模块，那么第七章讲的就是：如何把这些能力组装成一个真正能运行在现实世界里的产品。
