# 登高望远：Claude Code 技术架构全景 · 口播文案稿

> 适用场景：AI 工程师内部分享 / 技术播客开场
> 预计时长：约 10 分钟
> 建议语速：220~240 字 / 分钟
> 阅读对象：使用过 Claude Code 或类似 AI 编码工具的开发者

## 【开场钩子 · 30 秒】

你有没有过这种经历——用 AI 写代码，前两周惊艳得像开了挂，第三周就被折磨到崩溃？每次开新会话它就失忆，代码风格时有时无，审查六七个文件直接"断片"。如果你以为是模型不行，那今天这一期，可能会颠覆你的认知。

真相是：你手里的工具，本质上不是一个聊天助手，而是一座冰山。你只看到水面上的对话窗口，却忽略了水面下隐藏的整套 Agent 框架。

## 【冰山比喻 · 60 秒】

我们先想象一座冰山漂在海面上。你看得见的——终端对话、代码生成、命令执行——只是露出的一角。真正撑起这座冰山的，是藏在水面下的庞然大物：CLAUDE.md 记忆系统、Skills 技能加载、子智能体任务委派、Hooks 事件钩子、MCP 外部协议、Headless 自动化，以及顶层的 Agent SDK。

接下来这一章，我们要做的就是"潜到海底看基座，再爬到顶楼看全景"。我会用一座摩天大楼的比喻，帮你把这个框架的四层架构一次性打通。

## 【第一节：从命令行到 Agent 框架 · 90 秒】

先说一个核心公式：**Agent = Model + Harness**。

模型本身只会"说话"，真正的"做事"能力来自包裹在外层的 Harness——你可以把它理解成"马具"。同一匹马配上不同马具，能拉车、能耕地、能赛跑。同一模型套上不同 Harness，表现可以天差地别。

这就是为什么行业里有一句话：**Harness 比 Model 更重要**。在 TerminalBench 基准测试里，光是对 Harness 做优化，就让同一模型从基线水平以下跃升到 Top 5。

所以 Claude Code 真正的定位是什么？是一个**可编程、可扩展、可组合**的 Agent 框架。可编程，意思是能用 Python 或 TypeScript 直接驱动它；可扩展，通过配置文件注入新能力；可组合，让模块像乐高一样自由拼装。

## 【第二节：四层架构 · 90 秒】

我们用一栋摩天大楼来理解这四层。

**最底层是记忆层**。这栋楼的地基。它的核心载体是一份叫 `CLAUDE.md` 的文件，相当于 Claude 的"员工入职手册"。每次对话开始时自动加载，省去你反复重申项目背景的麻烦。记忆层还分五级：企业级、用户级、项目级、规则级、本地级，越往下越具体，优先级越高。

**第二层是扩展层**。大楼的主体楼层。它有四大组件：Commands、Skills、SubAgents、Hooks。它们各自独立，又能灵活组合。Commands 是用户手动触发的命令模式；Skills 是 Claude 语义自动匹配的策略模式；SubAgents 是隔离上下文的线程池；Hooks 是事件驱动的中间件。

**第三层是集成层**。大楼的水电管网。它由两大部分组成：Headless 模式让 Claude 走进 CI/CD 流水线，MCP 协议让外部能力走进来。Anthropic 把 MCP 比作"AI 时代的 USB-C"——一套标准接口，任何工具和数据源都能以统一方式接入。

**第四层是编程层**。大楼的顶层工作室。Agent SDK 是关键分水岭——前三层你是"住户"，使用现成配置；跨过 SDK 这条线，你就成了"建筑师"，可以用代码直接编排全新的 Agent。

## 【第三节：组件触发对比 · 60 秒】

扩展层四个组件最易混淆，我教大家一个口诀。

Commands："**你叫它做**"——用户主动输入。
Skills："**它自己知道该做**"——语义自动匹配。
SubAgents："**它安排别人做**"——任务委派隔离。
Hooks："**不管谁做，到了这一步就执行检查**"——事件拦截。

还有一个二分法视角：**确定性触发 vs AI 判断触发**。Commands 和 Hooks 是确定性的——行为可预期；Skills 和 SubAgents 是 AI 判断的——行为有弹性。安全性高的场景，优先确定性 Hooks；灵活性高的场景，优先 AI 判断的 Skills。

## 【第四节：数据流实例 · 80 秒】

光说组件太抽象，我们跟踪一个真实请求走一遍。

假设你在终端输入："帮我审查 src/payment/ 目录的代码变更，确保没有安全隐患。"

第一步，记忆层自动加载 CLAUDE.md，让 Claude 知道项目背景。
第二步，Skills 层做语义匹配，自动加载 security-review 这个技能。
第三步，Claude 判断这个任务量大，委派一个子智能体隔离执行审查。
第四步，子智能体完成任务后，把精简结论回传给主会话。
第五步，Hooks 在 PostToolUse 节点自动触发，把审查结果格式化输出。

全程不需要你额外干预。这就是"可组合性"的威力——每个组件是独立积木，组合起来就是一条自动化流水线。

## 【第五节：Harness 与 Agentic Loop · 90 秒】

最后我们潜到最底层，看 Harness 的"心脏"——Agentic Loop。

整个循环分五步：接收输入 → 模型推理 → 判断是否调用工具 → 是则执行工具 → 把结果回流到对话历史，再进入下一轮推理。循环什么时候停？两个条件：要么模型主动说"我搞定了"；要么达到 max-turns 上限，防止无限循环烧钱。

到这里，你会发现一个耐人寻味的事实：**Claude Code 本身，正是基于 Agent SDK 构建的一个 AI Agent**。你在终端看到的所有行为——读文件、写代码、执行命令——本质上都是 Agent SDK 层面的工具调用。洞察到这一层，对工具的认知就从"一款产品"升维为"一种架构模式"。

## 【决策树 · 60 秒】

实操层面怎么选组件？我给你一个三步决策法。

第一步，问"触发方式"。需要人触发？步骤固定用 Commands，需要 Claude 判断用 Skills。Claude 自动触发？事件节点用 Hooks，CI/CD 用 Headless。

第二步，问"运行环境"。任务量大、需要隔离上下文？叠加子智能体。需要访问外部系统？叠加 MCP。

第三步，问"是否需要代码级控制"。需要的话，直接上 Agent SDK。

记住一句话：**用最简单的方案解决当前问题**。不要刚学会子智能体就处处委派，不要刚学会 Hooks 就给每个操作加守卫。

## 【踩坑提醒 · 60 秒】

最后说几个高频踩坑点。

第一，不写 CLAUDE.md，每个会话都从零开始——这是地基缺失。第二，Skills 的 description 字段写得模糊，导致误激活或漏激活——description 就是 Claude 决策的唯一依据。第三，Hooks 配得太多太密，性能和维护成本反而飙升——只在安全质量关键节点配置。第四，子智能体嵌套过深，上下文膨胀反而加剧——单层委派加总结回传。第五，Headless 模式不设 max-turns——Token 直接失控。

Plugins 这个东西，是给团队负责人准备的。把 Skills、Commands、Hooks、子智能体、MCP 配置统一打包，新人入职一条命令就能复用整套最佳实践。
