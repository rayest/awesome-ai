## 【开场钩子 · 30 秒】

小冰接了个看似简单的需求："让 Claude 帮我们分析下 Jira 中上周所有 P1 级 bug 的分布规律。" 她盘算：Jira 在公司内网，Claude 在云端服务器——这俩世界之间横着一道天然的网络隔离墙。在 MCP 出现之前，每个 AI 客户端想接入每个数据源都要写专属适配器——这就是 M×N 的复杂度难题。小冰立刻反应过来："就像 USB-C 普及前的充电线？出差恨不得带上六七根？" **MCP 就是 AI 工具集成领域的 USB-C。**

## 【MCP 是什么 · 60 秒】

MCP 全称 Model Context Protocol，由 Anthropic 于 2024 年 11 月发布，2025 年 12 月捐赠给 Linux 基金会——短短 13 个月，从单一企业项目蜕变为行业标准。

它解决的核心问题就是 M×N 复杂度。在 MCP 出现之前，每对 AI 客户端和数据源都要写一个适配器。换句话说，3 个 AI 客户端要接入 5 个数据源，需要写 15 个适配器。GitHub 想接入 Claude 得写一个适配器，Jira 想接入又得写另一个；一旦更换 AI 客户端，所有适配器还得推倒重来。

MCP 出现后呢？M 个客户端和 N 个数据源都遵循同一套协议，每边只需要写一次适配器——M+N 线性复杂度。这就是 USB-C 普及的逻辑：协议统一后，充电线、数据线、显示输出全用一根。

## 【第一节 · 核心架构与 3 种通信协议 · 90 秒】

MCP 架构包含三大角色。Host 是宿主，比如 Claude Code 桌面版或 CLI；MCP Client 是内嵌在 Host 中负责协议通信的模块；MCP Server 是暴露特定数据源能力的后端服务，比如 filesystem、GitHub MCP 这种。

通信基于 JSON-RPC 2.0 协议，这一选择赋予了 MCP 跨语言、跨平台的天然能力。

MCP 支持 3 种传输方式。stdio 适合本地进程通信，简单、零网络配置、启动快；HTTP + SSE 原本用于远程服务调用，但**已经被废弃**；新的 streamable_http 流式 HTTP 协议取代了它，兼容 HTTP 生态、可跨网络、需要部署服务。

配置文件 `.mcp.json` 是标准结构，本地 stdio 协议用 `command` 加 `args` 字段启动，远程 HTTP 协议用 `type: "http"` 加 `url` 字段。所有密钥都通过 `${GITHUB_TOKEN}`、`${DATABASE_URL}` 这种环境变量占位符注入——**配置可安全纳入版本控制，密钥不进仓库**。

## 【第二节 · 实战配置与热门服务器 · 90 秒】

一份为全栈开发者设计的 MCP 套件通常包含 4 大服务器。

filesystem 服务器负责本地文件读写；fetch 服务器抓取 HTTP 网页内容；GitHub 服务器处理 PR、Issues、代码检索；database 服务器（推荐 Bytebase DBHub）支持多数据库统一接入，**务必用只读账号**。

3 个典型场景：让 Claude "统计数据库最近 7 天的订单总额"——自动调 database 跑 SQL；"把这段 bug 描述创建为 GitHub Issue"——自动调 GitHub 服务器通过 API 提交工单；"获取 React 19 官方文档关于 Server Components 的最新说明"——自动调 fetch 实时抓取网页。

热门第三方 MCP 服务器还有 Notion MCP（文档读写）、Sentry MCP（错误追踪）、Context7（最新技术文档检索）。**评估 MCP 服务器信任度与评估 npm 包无本质区别**——核心看来源（官方 `@modelcontextprotocol` 命名空间 vs 个人开发者）、维护活跃度、下载量、社区口碑。

## 【第三节 · 三层纵深安全机制 · 90 秒】

MCP 极大地扩展了 Claude 的能力边界，但也引入了显著的安全风险。架构设计了三层纵深防御。

**第一层：首次连接的交互式审批**。每当你配置了一台新的 MCP 服务器，Claude 在首次尝试连接前会自动暂停，展示服务器详细信息（来源、权限范围等），请求显式授权。**这一步是强制流程，无法绕过**。

**第二层：工具级权限控制**。即使 MCP 服务器已获全局授权，Claude 调用具体工具时仍会保持警惕——特别是有副作用的操作（文件写入、代码执行、数据库修改），会再次拦截并展示指令细节，等待二次确认。

**第三层：OAuth 2.0 动态认证**。远程 MCP 服务器推荐用 OAuth 2.0 而非长期静态 API 密钥，便于与企业的统一身份认证系统集成，支持令牌自动过期与刷新。

3 类主要风险与防护：提示注入攻击（恶意内容通过 MCP 注入 Claude 上下文）——仅用可信来源；工具权限滥用（多个工具组合产生意外后果）——最小权限原则；冒名顶替（恶意工具伪装成合法工具）——优先官方命名空间。

**3 条铁律**：数据库连接**务必用只读账号**、API 密钥**通过环境变量注入严禁硬编码**、**优先官方命名空间**服务器。

## 【第四节 · MCP + Skills：厨房与菜谱 · 90 秒】

MCP 和第 3 章的 Skills 解决的是不同层面的问题，Anthropic 官方有个生动类比。

**MCP 是专业厨房**——配备所有必要设备与原料：冰箱（数据库）、炉灶（API 调用）、食材（数据源）、刀具（工具）。拥有厨房意味着厨师具备制作任何菜肴的能力，但具体"做什么"和"怎么做"完全依赖厨师的临场发挥。

**Skills 是标准菜谱**——提供详尽的操作指南：所需食材、操作步骤、火候控制、烹饪时长。有了菜谱，即使经验尚浅的厨师也能稳定复现出高水准的菜肴。但若缺乏配套的厨房设备，再完美的菜谱也无法落地。

两者的协作模式是：在 Skills 的 SKILL.md 中引用 MCP 工具，从而指导 Claude 按照特定工作流高效调用这些工具。比如 sprint-manager 这个 Skill 触发后，Claude 会按 Skill 定义的步骤调用 Jira MCP 的 `jira.get_sprint`、`jira.list_issues`、`jira.get_issue_details`，最后按 Skill 规定的输出格式生成 Sprint 状态报告。

这套组合让 Claude 既有 MCP 的实时数据获取能力，又有 Skills 的一致处理规范——不再依赖用户每次的详尽指令，也不仰仗临场判断。

## 【第五节 · 企业级部署 · 80 秒】

对拥有数十甚至上百名工程师的团队，依赖个人配置 MCP 服务器不可扩展。Claude Code 提供了企业级集中管理机制。

管理员通过 `managed-mcp.json` 配置文件或 Claude for Enterprise 管理后台，为组织内所有用户预配置 MCP 服务器。这些服务器**自动对所有成员生效，不需要个人操作**。管理员还可以预先授权服务器，**免除首次使用时的审批弹窗**。

这种模式实现了双赢：对开发者而言，公司的 Jira、Linear、内部知识库一旦以 MCP 形式接入，整个团队即可"零配置开箱即用"；就安全性而言，所有 MCP 访问均经由严格审查的服务器，集中式的审计与管控显著提升安全水位。

## 【决策树 · 60 秒】

什么场景用什么协议？本地文件读写、数据库查询用 stdio；远程 GitHub / Jira / Notion 协作用 streamable_http。

配置文件放哪里？个人项目用 `.mcp.json`；团队企业级用 `managed-mcp.json`。

安全怎么管？数据库用只读账号；API 密钥用环境变量；优先官方命名空间服务器；只授权必要的工具；启用 OAuth 2.0 替代静态密钥。

调试怎么搞？`claude mcp list` 看所有服务器状态；`claude mcp test <server>` 测试连通性；`claude mcp --debug` 启用调试模式；输出超过 10000 Token 系统会告警，超过 25000 Token 自动截断。

## 【踩坑提醒 · 60 秒】

五个最常见的坑：

第一，**硬编码 API Token**——配置文件泄露密钥。永远用 `${ENV_VAR}` 占位符。

第二，**数据库用读写账号**——一次误操作就删表。MCP 数据库连接**必须**用只读账号。

第三，**MCP 服务器来源不明**——提示注入或工具冒名风险。优先选 `@modelcontextprotocol` 官方命名空间。

第四，**stdout 混入日志**——JSON 解析失败。所有日志走 stderr，stdout 严格保留纯净数据。

第五，**把 MCP 当 Skills 用**——期望"流程规范"。MCP 是厨房、Skills 是菜谱，分工不同，**需要搭配使用**才能发挥完整威力。
