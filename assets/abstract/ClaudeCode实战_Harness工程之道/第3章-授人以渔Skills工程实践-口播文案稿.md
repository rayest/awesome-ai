# 授人以渔：Skills 工程实践 · 口播文案稿

> 适用场景：AI 工程师内部分享 / 技术播客
> 预计时长：约 10 分钟
> 建议语速：220~240 字 / 分钟
> 阅读对象：使用 Claude Code 的开发者 / 知识工程师

## 【开场钩子 · 30 秒】

你有没有过这种崩溃时刻——每次新会话都把"我们项目用 TypeScript、不用 JavaScript"的规矩重申一遍？更夸张的是，同一会话里从「生成 API 文档」切换到「审查代码」，Claude 就像失忆一样，又得重教一遍？这一期，我们把这种"日复一日为临时工做入职培训"的痛苦，根治一次。

## 【CLAUDE.md vs Skills · 80 秒】

核心问题：CLAUDE.md 难道不够用吗？

想象一个公司有两类知识资产。第一类是墙上标语——考勤制度、安全红线、人人可见、时刻生效，这对应着 CLAUDE.md。第二类则是部门文件柜里的 SOP——财务月结、运维故障处理、客服响应规范，只有对应岗位执行特定任务时才翻阅。这，就是 Skills。

CLAUDE.md 的代价是「常驻」——不管你是在调试 bug、跑测试还是写配置，文档规范的 Token 都会持续消耗，白白占位置。Skills 解决的正是「按需投放知识」问题：把专业知识封装在特定目录里，只有当真正需要时，才被精准加载。

## 【Skill 文件夹结构 · 60 秒】

一个 Skill 不是一段 Prompt，而是一个完整的工程化目录。它包含四类资产：

- SKILL.md：核心骨架，主技能文件
- scripts/：可执行脚本，让 Skill 能主动「做」事
- reference/：按需加载参考文档，记忆库
- templates/：输出模板，统一格式

命名上有 5 条铁律：SKILL.md 必须全大写、目录名 kebab-case、不放 README.md、name 字段品牌中立、字符集限 `[a-z0-9-]`。

## 【YAML Frontmatter · 80 秒】

SKILL.md 顶部的 YAML 定义了 Skill 的「身份证」，可拆为 3 大逻辑组。

身份字段：name、description、argument-hint 定义「它是什么」。
权限字段：disable-model-invocation、user-invocable、allowed-tools、model 规定「谁能调用、能做什么」。
执行字段：context、agent、hooks 决定「在哪执行、执行中发生什么」。

特别要区分两个容易混淆的开关：disable-model-invocation 是「禁止 Claude 自动触发」，user-invocable 是「不在菜单中显示」。两者正交，控制两个独立维度。

## 【渐进式披露 · 90 秒】

这是 Skills 系统最巧妙的设计。

想象走进图书馆找资料——你不会一次性读完所有藏书，而是先浏览分类编目，再查图书目录，最后精读所需章节。Skills 正是这套流程的三层模型：

- 层级 1 索引层：仅扫描所有 Skill 的 description，约 100 Token
- 层级 2 概览层：加载目标 Skill 的 SKILL.md 主文件，约 5000 Token
- 层级 3 执行层：按需加载被引用的具体资源文件，动态消耗

举一个真实数字：一个完整的财务分析 Skill 包含 5300 Token。当用户问「毛利率怎么算」，Claude 只需加载 SKILL.md 加 profitability.md，约 1800 Token，其他 3500 Token 保留在磁盘上，零消耗。这比传统全量加载节省了 57%~98%。

## 【description 预算机制 · 60 秒】

但是有个致命陷阱：所有 Skill 的 description 共享一个 16000 字符的预算池，由数量平分。装 20 个 Skill，平均每个只剩 800 字符。

如果某个 description 超过 800 字符，它会被**静默排除**——Claude 完全不知道这个 Skill 的存在，仿佛消失了。三个避坑技巧：给纯内部 Skill 设 disable-model-invocation 释放空间、运行 /context 查看被排除的 Skill、用环境变量手动扩大预算。

## 【双通道触发 · 80 秒】

Skills 支持两种激活路径。

显式调用：用户输入 `/skill-name`，类似终端命令，明确无歧义，适合高风险操作。

语义匹配：Claude 根据对话意图自动研判——这才是 Skills 的核心价值。用户说"帮我写 API 文档"，Claude 在幕后智能决策是否调用 api-doc-generator Skill。

description 字段是语义匹配的唯一信号，不是人类阅读的说明书，而是 Claude 决策的「语义指纹」。Vercel 数据显示，若缺乏明确指引，Agent 有 56% 的概率完全不会去查看可用的 Skills。高效 description 公式：功能定义 + 触发场景 + 核心能力。

## 【防止过触发与欠触发 · 60 秒】

两个常见失效模式。欠触发——该调没调，根因是 description 写了技术语言，与用户口语化表达存在语义鸿沟。修复策略：「翻译」用户语言，加入同义词与口语说法。

过触发——不该调被激活，根因是 description 太宽泛。修复策略：引入负向约束，Not for 句式排除干扰项。

验收标准：相关任务触发率 ≥ 90%，无关任务误触发率 ≤ 5%，用 10+10 测试用例验证。

## 【参考型 vs 任务型 · 60 秒】

判断标准只有一句话——「如果 Claude 自动执行这个任务，最坏的情况是什么？」

如果答案让你紧张，比如自动提交未测试代码、自动部署有 bug 的版本、自动删除生产数据——必须选择任务型 Skill，加 disable-model-invocation: true，把控制权牢牢握在用户手中。/commit、/deploy、/notify 都属此类。

如果答案无关痛痒，比如多展示一段参考文档——可以选择参考型 Skill，让 Claude 智能判断。API 设计规范、代码审查清单都属此类。

核心原则：副作用越大，控制权越要收紧。

## 【决策树 · 60 秒】

4 种设计模式按需组合。

需要标准化输出？添加 templates 目录，模板驱动模式。需要确定性计算？添加 scripts 目录，脚本增强模式——脚本比模型推理更精准、更复现。知识量大于 500 行？拆分至 reference 目录，知识分层模式。需要安全边界？配置 allowed-tools，工具隔离模式。

## 【踩坑提醒 · 60 秒】

9 个高频踩坑：SKILL.md 堆砌知识导致 Token 失控；description 写得太长被静默排除；用 Bash 全通配等于放弃安全防线；用 !command 但不限范围导致 Shell 注入；不做触发测试就上线；把 /commit 配成参考型等于自杀；目录名用 camelCase 跨平台出错；Skill 里放 README.md 干扰指令理解。

## 【收尾 · 30 秒】

Skills 看似只是「又一个配置文件」，实则把人类数十年知识管理经验数字化——CLAUDE.md 是墙上标语、Skills 是部门文件柜、reference 是知识库、templates 是输出模具、scripts 是自动化工具集。下一章，我们讲子智能体——当一个 Skill 不够用时，如何让多个 Skill 协同作战。
