# Skills · 口播文案稿

> 适用场景：AI 技术博主 / 公众号 / 课程
> 预计时长：约 10 分钟
> 建议语速：220~250 字/分钟
> 阅读对象：Agent 开发者、AI 产品经理

## 【开场钩子 · 30 秒】

你有没有过这种经历？

项目里要接 100 个 AI 工具，结果光工具描述就吃掉了**三分之一的上下文窗口**。

用户问"1+1=?"，AI 答"2"，**上下文开销是问题的 13 400 倍**。

这就是 2025 年 AI Agent 工程的**最大痛点**——**工具爆炸**。

好，本章我们要用一个**标准 USB Hub**——**Skills**——彻底解决这个问题。

## 【核心比喻开场 · 60 秒】

我打一个比方。

**传统 MCP** = 给电脑接 100 个外设，每个都要预装驱动，**全部描述信息常驻内存**。系统还没启动，光驱动列表就吃掉了 33% 的内存。

**Skills 渐进式加载** = 接了一个**标准 USB Hub**——只告诉系统"我有个 Hub"（L1 标签）；用户插入什么设备（U 盘 / 键盘 / 鼠标），才加载对应驱动（L2 / L3 资源）。

核心区别：**Skills 让 LLM 知道"有什么"但不强制加载"是什么"**——需要时再加载。

好，我们正式进入第五章。

## 【第一节 · Skills 是什么 · 90 秒】

先说 Skills 是什么。

Skills 是**标准化结构化格式的专家经验包**。

什么意思？你把"解决问题的完整上下文"——思考路径、执行流程、依赖工具、数据来源、输出规范——封装成一个独立单元。

类比人类：
- **知识管理** = 项目复盘文档——任务完成后按模板整理
- **技术架构** = 本地化 RAG——动态检索 + 结构化内容

Skills vs 普通文档 / RAG 的区别：

- **普通文档**：自由格式，依赖人查找
- **RAG 检索**：半结构化，语义匹配
- **Skills**：强结构化 + 标签引导 + 资源协同执行

关键洞察：Skills 的内容组织**更结构化、调用更精准**，且**支持代码与资源的协同执行**。

## 【第二节 · 4 大组件结构 · 80 秒】

好，Skills 怎么组织？

**4 大组件**：

| 组件 | 必选 | 作用 | 类比 |
|------|------|------|------|
| **SKILL.md** | ✅ 必选 | 核心标签 + 说明书 | 商品标签 + 说明书 |
| **scripts/** | 可选 | 可执行代码 | 内置工具 |
| **references/** | 可选 | 领域参考资料 | 技术手册 |
| **assets/** | 可选 | 静态资源 + 模板 | 包装盒 |

典型目录长这样：
```
ops-inspection/                    # 运维巡检 Skill
├── SKILL.md                       # 核心说明（必选）
├── scripts/
│   └── collect_system_info.py    # CPU/内存/磁盘采集脚本
├── references/
│   └── inspection-standards.md   # 健康阈值标准
└── assets/
    └── report_template.md        # 巡检报告模板
```

注意：**SKILL.md 是唯一必选组件**——其他都可以没有，但必须有 SKILL.md。

## 【第三节 · 3 级渐进式加载 · 90 秒】

接下来是 Skills 的**核心创新**——**3 级渐进式加载**。

传统 MCP 的痛点（真实数据）：

> 7 个 MCP Server + 100 多个工具 → 工具描述占 **67 000 Token** ≈ LLM 上下文窗口的 **33%**
> 用户仅问"1+1=?"（5 Token）→ 上下文开销 **13 400 倍**

Skills 的 3 级加载：

| 级别 | 加载内容 | 加载时机 | 类比 |
|------|----------|----------|------|
| **L1 标签** | SKILL.md 元数据 | Agent 启动时一次性 | 商品标签 |
| **L2 文档** | SKILL.md 完整正文 | 判断某 Skill 可能适用时 | 商品说明书 |
| **L3 资源** | scripts/ + references/ + assets/ | 实际执行过程中按需 | 商品配件 |

**两个核心洞察**：

第一，scripts/ 中的代码**不会送入 LLM 上下文**——由 Agent 内置的 Bash 工具直接执行，仅返回运行结果。这正是 **CodeAct 模式的典型延伸**——代码视为动作而非文本。

第二，即便一个 Skill 内部打包了数百个工具定义、完整数据字典或上百页参考手册，只要当前任务无须使用，**这些内容就不会进入 LLM 的上下文**。

## 【第四节 · 如何设计支持 Skills 的 Agent · 80 秒】

那怎么设计支持 Skills 的 Agent？

**两个维度**：基础工具 + 控制流程。

**基础工具**（最少 2 个）：

- **Read 工具**：读取任意文件内容（SKILL.md / 参考文档）
- **Bash 工具**：执行系统命令（遍历目录 / 运行脚本）

**关键原则**：所有工具都应**保持通用性**，避免为特定 Skills 定制专用接口。

**Agent 控制流程**（ReAct 架构天然适配）：

```
[初始化阶段]
遍历 Skills 目录 → 读取每个 SKILL.md 的 L1 标签
   ↓
将标签注入系统提示词

[任务处理阶段]
LLM 判断某任务可能匹配某 Skill
   ↓
触发 Read 工具读取完整 SKILL.md（L2 文档）
   ↓
若文档指示需执行脚本/查阅资料 → Bash 执行 scripts/ + Read 获取 references/
   ↓
结合加载内容 + 执行结果 → LLM 生成最终响应
```

LLM 负责推理与决策，工具负责感知与行动——这与 ReAct 架构**完全契合**。

## 【第五节 · SkillLoader 实现 · 80 秒】

来，看实战代码。

`SkillLoader` 类负责一级加载——启动时扫描所有 Skills 目录，提取元数据。

3 个核心设计点：
1. `_load_all()` 递归扫描——一次性加载所有 Skills 索引，避免后续重复 IO
2. `rglob("SKILL.md")`——递归查找任意层级的 SKILL.md 文件
3. `_parse_frontmatter()` 解析 YAML——提取 L1 标签（name + description + tags）

二级加载用 `get_content(name)` 方法——返回 `<skill name="...">完整内容</skill>`，便于 LLM 识别"这是 Skill 内容"。

最后把 `load_skills` 工具集成到 Agent Loop 的工具路由中，**通过工具调用触发二级加载**。

系统提示词也关键——必须包含"先使用 load_skills 获取相关的专业知识"的行为指引，**训练 LLM 主动加载**。

## 【第六节 · 零代码开发运维巡检 Skill · 80 秒】

来，实战演示。

**用扣子编程开发 Skill**——SKILL.md 写作有 5 大要素：

| # | 要素 | 作用 |
|---|------|------|
| 1 | **定时机** | 明确触发条件 |
| 2 | **立目标** | 一句话说明核心问题 |
| 3 | **理规则** | 详细执行逻辑 |
| 4 | **给示例** | 典型输入输出对 |
| 5 | **划边界** | 能力边界与限制条件 |

扣子编程支持自然语言一键生成 Skill 文件夹——左侧展示开发过程，右侧沙盒环境测试，下载即可复用。

生成示例：写个"运维巡检 Skill"，检查 CPU / 内存 / 磁盘 / Docker 容器状态，输出巡检报告。

**关键设计**：用 `psutil` 库采集系统指标，采集逻辑封装到 `scripts/collect_system_info.py`，阈值标准存到 `references/inspection-standards.md`。

**基于 OpenClaw 测试**——把 Skill 上传到 `/usr/lib/node_modules/openclaw-cn/skill`，通过飞书机器人对话测试。

**关键成果**：**整个过程无须编写任何运维脚本、无须配置告警推送逻辑、无须手动调度任务**——仅通过几轮自然语言对话，便完成了传统运维中通常需要数小时编码、测试与部署的工作。

## 【决策树 · 60 秒】

来，给个选型决策树：

你的工具数量？
- **< 10 个** → MCP 工具集即可
- **10-50 个** → Skills + 渐进式加载
- **> 50 个** → Skills + 索引 + 分级权限

你的 Skill 来源？
- **内部编写** → SKILL.md 5 大要素齐全
- **AI 生成** → 扣子编程 / Cursor
- **第三方** → OpenClaw / Claude Code

你的加载策略？
- **粗粒度** → L1 标签 + L2 文档
- **细粒度** → L1 + L2 + L3 资源

## 【踩坑提醒 · 60 秒】

最后几个工程坑：

1. **元数据未提取**——LLM 不知道有哪些 Skills 可用，启动时一定要调 `_load_all()`。
2. **正文过早加载**——浪费上下文窗口，L1 仅加载标签。
3. **scripts 进 LLM**——scripts/ 代码塞进上下文是大忌，必须由 Bash 直接执行。
4. **Skill 名称冲突**——同名 Skill 互相覆盖，优先取 frontmatter 的 name。
5. **description 写得太泛**——LLM 不知何时调，5 大要素必须齐全。
6. **平台不兼容**——遵循 SKILL.md 标准结构才能跨平台。

## 【收尾 · 30 秒】

总结一句话：

**Skills 的本质 = 标准化结构化文档 + 渐进式按需加载**。

让 LLM **知道有什么**（L1 标签），但**不强制加载是什么**（L2 / L3 按需）。

关键数据：7 个 MCP Server + 100 工具 → 67 000 Token 浪费 → 改成 Skills 后仅 ~200 Token 标签。
