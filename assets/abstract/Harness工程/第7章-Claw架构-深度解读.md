# Claw 类产品：从 NanoClaw 看 OpenClaw 的工程实现

## 速查表

| 概念 | 一句话定义 | 核心组件 | 典型场景 |
|------|----------|----------|----------|
| OpenClaw | 开源 AI Agent 办公自动化平台（"养虾"） | IM 集成 + 定时任务 + Agent Loop | 24 小时在线数字员工 |
| NanoClaw | OpenClaw 精简版（~4000 行 NodeJS） | 同架构，代码量极简 | 二次开发 / 学习研究 |
| 容器内 | 核心 Agent 运行环境（Docker） | Agent Loop + Skills + 浏览器 | 安全沙箱执行 |
| 容器外 | 管理层（IM 通信 + 任务调度） | 8 大模块 | 调度 / 路由 / 持久化 |
| IPC | 进程间通信（文件系统） | input / messages / tasks 三目录 | 容器内外消息传递 |
| at-least-once | 至少处理一次语义 | lastAgentTimestamp 游标 | 断点续传 |

---

## 0. 全章比方：养一只永远在线的"数字虾"

想象你养了一只小龙虾：

- **它 24 小时在线**——不管你什么时候私聊或 @ 它，它都秒回
- **它能自己安排工作**——不需要你每天提醒，它能定时推送晨报、提醒你喝水
- **它在沙箱里干活**——即使它执行了危险操作，后果也只限于沙箱内，不会影响你的电脑

这就是 **OpenClaw**（中文社区昵称"龙虾"）—— 一个**始终在线 + 自主调度**的 AI Agent 数字员工。

本章拆解其工程实现。**建议从精简版 NanoClaw 入手**（~4000 行 NodeJS，~35 000 Token），而不是直接啃 40 万行的 OpenClaw。

## 7.1 Claw 类产品的核心架构

### 7.1.1 学习路径：从 NanoClaw 开启探索

**关键公式**：

```
现代 Agent = Coding Agent + Skills
```

**OpenClaw 的成功本质**：

| 表面现象 | 实际机制 |
|----------|----------|
| 能写 PPT、回复邮件、整理晨报 | **非原生能力**——由内部封装的 Code Agent（基于 Pi 框架）实现 |
| 始终在线 | IM 集成（飞书 / Telegram / 钉钉）+ 定时任务 |
| 自主调度 | 内置 Cron / Interval 调度器 |

**OpenClaw 的角色定位**：

```
┌──────────────────────────────────────────┐
│              OpenClaw 角色定位             │
├──────────────────────────────────────────┤
│  基础设施：IM 协议 + 任务编排 + 调度 + 容器  │
│  ────────────────────────────────────     │
│  核心 Agent：Code Agent（基于 Pi 框架）     │
│  ────────────────────────────────────     │
│  Skills：办公 / 邮件 / PPT 等              │
└──────────────────────────────────────────┘
```

**为什么从 NanoClaw 入手**：

| 指标 | OpenClaw | NanoClaw |
|------|----------|----------|
| 代码量 | ~40 万行 | ~4 000 行 |
| Token 数 | 不可估量 | ~35 000 Token |
| LLM 全量阅读 | ❌ 不可能 | ✅ 一次喂给 GLM-5（200K 上下文） |
| 学习门槛 | 高 | 极低 |

**核心洞察**：让 LLM **一次掌握全项目**——把代码塞进上下文，让模型作为"代码理解助手"。

**两大组成部分**：

| 层级 | 角色 | 关键模块 |
|------|------|----------|
| **容器内**（执行层） | 核心 Agent 运行环境 | Agent Loop + Skills + 浏览器 |
| **容器外**（管理层） | IM 通信 + 任务调度 | 8 大模块 |

**为什么把 Agent 放在容器内**：

> 安全隔离——即便 Agent 执行了危险操作（误删文件 / 向群聊发送垃圾信息），后果也仅限于容器内部，不会直接影响宿主机安全。

**核心逻辑**：解决**信息如何在受限的容器环境与外部 IM 平台之间安全、可靠地流转**的问题。

### 6 大功能链路详解

#### 功能 1：消息接收与持久化

> **ASCII 流程图：消息从 IM 进入 SQLite**

```
  Telegram / 飞书
       │
       ↓ HTTP/WebSocket
  Channels（自注册工厂）
       │
       ↓ 抽象为标准消息对象
  持久化优先策略
       │
       ↓ 元数据立即写入
  ┌─────────────────┐
  │  SQLite 数据库    │
  │  ┌─────────────┐ │
  │  │ messages 表 │ │
  │  │ router_state│ │
  │  │ lastAgent   │ │  ← 持久化游标
  │  │ Timestamp   │ │
  │  └─────────────┘ │
  └─────────────────┘
       │
       ↓ 恢复时：recoverPendingMessages()
       │ 对比游标 → 找出未处理消息 → 重新入队
  至少处理一次（at-least-once）保证
```

**3 大设计亮点**：

| # | 设计 | 作用 |
|---|------|------|
| 1 | **自注册工厂模式** | 把 Telegram / 飞书等异构平台协议抽象为标准消息对象 |
| 2 | **持久化优先策略** | 第一时间写入 SQLite，提供天然断点续传能力 |
| 3 | **lastAgentTimestamp 游标** | 重启后对比数据库找出未处理消息重新入队，确保 at-least-once |

#### 功能 2：主循环与消息处理

> **ASCII 流程图：编排器主循环**

```
  startMessageLoop（每 2 秒轮询）
       │
       ↓ SELECT * FROM messages WHERE timestamp > lastAgentTimestamp
  找到各群组新消息
       │
       ↓ 判断：群组是否需要触发词（@机器人）？
       │
   ┌───┴────┐
   │ 不需要 │ → 全部消息激活 Agent
   └────────┘
   ┌───┴────┐
   │ 需触发 │ → 检查消息是否含触发词
   └────────┘
       │
       │ 含触发词？
       ├─ 否 → 跳过（不激活）
       └─ 是 → 激活 Agent
              │
              ↓ 不是只处理触发消息
       把触发词之前积累的多条上下文消息一并代入
              │
              ↓ 既保证隐私（无关闲聊 Agent 看不到）
              又保留连贯性（Agent 能看到完整意图演进）
```

**精巧设计**：触发词之前积累的多条上下文消息一并代入——**既保护隐私，又保留对话连贯性**。

#### 功能 3：定时任务调度

**3 种调度类型**：

| 类型 | 表达式示例 | 触发时机 |
|------|-----------|----------|
| **cron 表达式** | `0 22 * * *` | 每天 22 点 |
| **固定间隔** | `*/5 * * * *` | 每 5 分钟 |
| **一次性任务** | `2025-12-31 23:59` | 指定时间触发一次 |

**2 种上下文模式**：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **隔离模式** (isolated) | 每个任务创建全新会话，看不到任何历史 | 独立任务 |
| **群组模式** (group) | 使用群组当前会话，能感知历史对话 | 连续任务 |

**巧妙策略：锚定于计划时间**：

```
  任务：每 5 分钟执行一次
  系统中断：2 小时

  ✗ 错误：从当前时间起每 5 分钟往后堆叠（连续触发多次追赶）
  ✓ 正确：从原本应该执行的时间点继续（避免时间漂移）
```

#### 功能 4：并发控制与容器分配

> **ASCII 流程图：group-queue 容器分配**

```
  主循环找到需处理的消息
       │
       ↓ 不立即启动容器
  group-queue 排队
       │
       ↓ 检查：当前是否有空闲容器？
       │
   ┌───┴─────────┐
   │ 有空闲容器   │ → pipe 传输到容器 stdin
   │              │ → Agent 立即处理 → 返回结果
   │              │ → 实现实时对话
   └──────────────┘
   ┌───┴─────────┐
   │ 无空闲容器   │ → 消息入队等待
   │ （默认最多5个）│ → 触发启动新容器
   └──────────────┘

  空闲等待状态：
  ┌────────────────────────────────────┐
  │  Agent 完成工作 → 等待新输入        │
  │  空闲 > 30 分钟 (IDLE_TIMEOUT)     │
  │  → group-queue 发送 _close 指令   │
  │  → 容器优雅关闭，释放资源          │
  └────────────────────────────────────┘
```

**4 大状态**：处理中 / 空闲等待 / 处理定时任务 / 已关闭

**关键设计**：

- **默认最多 5 个 Docker 容器**（可配置）
- **30 分钟 IDLE_TIMEOUT**——空闲超时自动关闭
- **优雅关闭**——发送 `_close` 指令而非强制 kill

#### 功能 5：IPC 通信与容器内处理

> **ASCII 流程图：文件系统 IPC 三目录**

```
  ┌─────────────────────────────────────────────┐
  │  容器外（主进程）          容器内（Agent）   │
  │                                              │
  │  ┌──────────────┐         ┌──────────────┐  │
  │  │ input/ 目录  │ ──写入──→│  Agent Loop  │  │
  │  │ (主→容器)   │         │  读取 + 处理  │  │
  │  │ 消息+关闭指令 │         │              │  │
  │  └──────────────┘         └──────┬───────┘  │
  │                                  │           │
  │  ┌──────────────┐         ┌──────┴───────┐  │
  │  │ messages/目录 │ ←──写入──│  写入处理结果  │  │
  │  │ (容器→主)   │         │              │  │
  │  │ 待发送消息    │         └──────────────┘  │
  │  └──────────────┘                           │
  │                                              │
  │  ┌──────────────┐         ┌──────────────┐  │
  │  │ tasks/ 目录  │ ←──写入──│  定时任务请求  │  │
  │  │ (容器→调度器)│         │ (创建/暂停/取消)│  │
  │  └──────────────┘         └──────────────┘  │
  └─────────────────────────────────────────────┘

  权限边界：
  普通群组的容器只能访问自己的 IPC 目录
  无法窥探或干扰其他群组
```

**3 个 IPC 子目录**：

| 目录 | 方向 | 内容 |
|------|------|------|
| `input/` | 主进程 → 容器 | 消息 + 关闭指令 |
| `messages/` | 容器 → 主进程 | 待发送消息 |
| `tasks/` | 容器 → 调度器 | 定时任务 CRUD |

**设计精髓**：**目录路径本身天然充当权限边界**——普通群组的容器只能访问自己的 IPC 目录。

#### 功能 6：结果路由与回复发送

> **ASCII 流程图：结果路由闭环**

```
  容器内 Agent 处理完成
       │
       ↓ 写入 messages/ 目录
  IPC Watcher 监听
       │
       ↓ 读取新输出文件
  路由器模块
       │
       ↓ 根据消息来源找到对应 Channel
  通道（飞书 / Telegram）
       │
       ↓ 发送到 IM 软件
  ┌──────────────────┐
  │  IM 软件显示回复   │
  │  ↓                │
  │  用户看到结果      │
  │  ↓                │
  │  继续发送新指令   │  ← 循环
  └──────────────────┘
```

### 关键代码：使用 sdkEnv 注入环境变量

**用途**：让容器内 Agent 能访问 .env 配置的所有环境变量。

```python
import os
from dotenv import load_dotenv

load_dotenv()

container_config = {
    "image": "nanoclaw-agent:latest",
    "env": {
        "ANTHROPIC_BASE_URL": os.getenv("ANTHROPIC_BASE_URL"),
        "ANTHROPIC_AUTH_TOKEN": os.getenv("ANTHROPIC_AUTH_TOKEN"),
        "ANTHROPIC_MODEL": os.getenv("ANTHROPIC_MODEL"),
    },
    "volumes": {
        os.getenv("WORKDIR", "/tmp/nanoclaw-work"): {
            "bind": "/workspace",
            "mode": "rw",
        },
    },
}
```

**关键设计点**：
- `load_dotenv()`：从 .env 文件加载所有配置
- `container_config["env"]`：通过 Docker SDK 注入环境变量
- **无需在代码中硬编码 API Key**——全部从 .env 读取
- **可灵活切换模型**——只改 .env 即可

### 关键代码：完整部署脚本

**用途**：一键完成 NanoClaw 完整部署（build + start）。

```bash

set -e

cd container
./build.sh
cd ..

echo "[2/4] Installing dependencies..."

echo "[3/4] Building TypeScript..."
npm run build

echo "[4/4] Starting NanoClaw..."
npm run dev
```

## 工程踩坑清单

| 坑 | 表现 | 解法 |
|----|------|------|
| **异构 IM 协议难统一** | 每接一个 IM 都要重写协议层 | Channels 自注册工厂模式 |
| **Agent 进程崩溃导致消息丢失** | 用户消息没被处理 | lastAgentTimestamp 游标 + at-least-once |
| **容器资源耗尽** | 100 个群同时发消息 | group-queue 并发控制（默认 5 容器） |
| **空闲容器浪费资源** | 容器一直运行不释放 | 30 分钟 IDLE_TIMEOUT 自动 _close |
| **定时任务时间漂移** | 系统恢复后任务连续触发 | 锚定于计划时间，不从当前时间追赶 |
| **群组之间互相干扰** | 容器能看到其他群消息 | 独立 IPC 命名空间（目录权限边界） |
| **触发词处理切断上下文** | Agent 看不到完整对话演进 | 把触发词之前积累的消息一并代入 |
| **国内大模型接入难** | 容器内硬编码 Anthropic API | sdkEnv 注入 + .env 灵活配置 |
| **触发词配置不一致** | 私聊 @机器人 vs 群聊 @机器人 | 群组级 `requireTrigger` 配置 |

## 贯穿主线

> **Claw 类产品的核心 = 始终在线 + 自主调度 + IM 集成**
>
> 把普通 Code Agent 转换为数字员工的关键是：
> 1. **容器内**执行（安全沙箱）
> 2. **容器外**管理（8 大模块协同）
> 3. **文件系统 IPC** 通信（简单可靠 + 天然权限边界）
> 4. **at-least-once 语义**（断点续传保证）
> 5. **锚定计划时间**（定时任务不漂移）
