## 【开场钩子 · 30 秒】

你有没有过这种经历？

AI Agent 该听懂用户说话的时候，它答非所问。
该写代码的时候，它去写诗了。
该让人类确认的时候，它自己拍板删了库。

这就是 Agent **没有五大能力**的代价——听不清、想不明、做不细、纠不了、控不住。

好，本章我们就把这 5 大能力——**意图识别 → 规划 → 反思 → 执行 → 人机协作**——从理论变成可复制的代码。

## 【核心比喻开场 · 60 秒】

我打一个比方。

想象你是一家米其林餐厅的主厨，每天要服务 100 桌客人。

- 客人说"随便来点" → 你得先**听懂**他到底想吃啥（意图识别）
- 听完之后你要**列菜单**：前菜→汤→主菜→甜点（计划模式）
- 端菜时发现"牛排五分熟但客人要全熟" → 要**回头补救**（反思模式）
- 客人突然问"能现场给我做个生日蛋糕吗" → 厨房里**现搭工具**现做（CodeAct）
- 遇到"我要把今天点的菜全部退掉" → 得**叫经理来确认**（人机协作）

本章就是教你把这 5 步都做成"自动化的流水线"。

好，我们正式进入第二章。

## 【第一节 · 意图识别：医院分诊台 · 90 秒】

第一个能力——**意图识别**。

类比：医院分诊台。心脏病人不会被分到儿科。

Agent 系统同理：
- 输入："帮我写个爬虫" → 路由到编程工具
- 输入："贵州茅台怎么样" → 路由到财经 OR 礼品 OR 收藏（**歧义**）

两大工程路径：

**Arch-Router（轻量级分诊护士）**——1.5B 小模型，普通 CPU 就能跑。给一张"分诊表"+ 客人描述，输出 `{"route":"编程"}`。

**小模型 + 大模型双保险**——小模型做第一道分诊，大模型二级会诊。
- 两者一致 → 信任小模型（节省成本）
- 不一致 → 触发兜底策略：礼貌拒绝 / 转人工 / 反问澄清 / 记录回流

**关键数据**：意图识别准确率 **≥ 98.5%** 才能满足企业级上线标准。每提升 1%，百万 DAU 系统每天少走错几十万次流程。

## 【第二节 · 计划模式：装修房子 · 90 秒】

第二个能力——**计划模式**。

类比：装修房子。老练的项目经理会：
1. **先出方案**：量房 → 出设计图 → 列施工清单
2. **再按方案施工**：水电 → 瓦工 → 木工 → 油漆 → 软装

而不是直接拎着锤子上场。

**简单计划模式**（旅游攻略）——4 件套节点：plan_node（生成大纲）+ execute_node（按大纲执行）+ tool_node（执行工具调用）+ should_continue_node（判断 `Final Answer` 终止）。

**复杂计划模式**（通用计划）——3 个升级：
- **历史回看**：LLM 能看到"已完成步骤+结果"
- **终止判定**：用 Pydantic 结构化输出 `Action(Union[Response, Plan])`
- **执行器**：用 `create_react_agent` 预构建组件

**关键洞察**：随着轮次推进，LLM 看到的上下文干扰**越来越少**——这是工程上保证准确率的核心设计。

## 【第三节 · 反思模式：作家改稿 · 90 秒】

第三个能力——**反思模式**。

类比：写作修改。你写完一篇文章会：
1. 自己读一遍（自我反思）
2. 给朋友看一遍（他人审视）
3. 根据反馈修改
4. 再读、再改……直到满意

反思模式把这件事自动化——用**两个 LLM 互相配合**。

**四维审查**：

| 维度 | 检查项 | 反例 |
|------|--------|------|
| 安全性 | 是否有破坏性操作风险 | `rm -rf /` |
| 规范性 | 是否符合最佳实践 | 应该加 `--name` 却没有 |
| 效率性 | 是否存在更优解 | 应该用 compose 却用裸 run |
| 任务对齐 | 是否真正解决用户问题 | 用户要查日志却生成了重启 |

**生产铁律**：反思模式不是"可选项"，而是**高风险场景的必选项**（运维、金融、医疗）。

**关键工程细节**：`iterations` 计数器必须设上限（比如 3 次），否则可能死循环。

## 【第四节 · CodeAct 模式：万能工匠 · 80 秒】

第四个能力——**CodeAct 模式**。

类比：瑞士军刀 vs 固定工具箱。
- Function Calling = 工厂的固定工具箱——工具必须预先注册
- **CodeAct** = 瑞士军刀 + 车床 + 3D 打印机——**现场造工具**

**真实痛点**：用户问"算过去 7 天某股票的平均收盘价"。
- Function Calling 路线：得提前注册 `get_stock_price(symbol, days)` 工具
- **CodeAct 路线**：LLM 直接生成一段 Python 代码（调用 yfinance + 计算），当场跑

**三级沙箱防护**（按强度递增）：
- **L1 本地 exec**：教学/Demo（把刀放在桌上）
- **L2 受限解释器**：受信任内部工具（给刀装个保护套）
- **L3 容器化沙箱**：Docker / gVisor / Firecracker（把刀锁进保险柜）

**工业级系统必须采用 L2 或 L3**——LLM 生成（或被诱导生成）`os.system("rm -rf /")` 的风险是真实的。

## 【第五节 · 人机协作：自动驾驶接管 · 90 秒】

第五个能力——**人机协作（Human-in-the-loop）**。

类比：自动驾驶的"接管机制"：
- L2 自动驾驶：可以自己开，但遇到复杂路况必须让人类接管
- L4 自动驾驶：基本不用接管，但仍有紧急按钮

**Agent 当前阶段 ≈ L2/L3**——自动化很强，但**关键决策必须留给人**。

**两种触发形式**：

| 形式 | 触发方式 | 案例 |
|------|----------|------|
| **强制介入** | 流程到固定节点必停 | 大纲审核、删库命令确认 |
| **自主介入** | LLM 觉得需要人时主动叫 | 信息缺失追问、模糊意图澄清 |

**巧妙设计**：注册一个**空壳工具** `ask_user`——LLM 觉得信息不够时，会像调用 `get_weather` 一样调用 `ask_user("您想买几个？")`。LLM 不知道问的是人还是 API，照常推理。

## 【决策树 · 60 秒】

来，给个选型决策树：

你的 Agent 卡在哪？
- **只是要"听懂用户说什么"** → 2.1 意图识别
- **任务可拆解成明确步骤** → 2.2 计划模式
- **输出质量不稳定** → + 2.3 反思模式
- **工具无法预定义** → 2.4 CodeAct 模式
- **涉及高风险或信息缺失** → 2.5 Human-in-the-loop

**实战经验**：这五种模式**不是互斥的**，而是可以**叠加**：
- DeepResearch = 计划模式 + 反思模式 + 联网搜索
- Claude Code = CodeAct + 人机协作 + 反思模式

## 【踩坑提醒 · 60 秒】

最后几个工程坑：

1. **interrupt 必须 async**——同步模式下 `interrupt()` 直接报错，所有节点 `async def`。
2. **递归死循环**——设置 `recursion_limit: 50`。
3. **同 thread_id 串扰**——每个用户/会话分配独立 thread_id。
4. **checkpointer 内存泄漏**——生产用 PostgresSaver / RedisSaver 替代 MemorySaver。
5. **结构化输出 + Few-shot 冲突**——提示词里**只放规则**，不放示例。
6. **工具描述 ≠ 计划目录**——docstring 与提示词中的目录名**一字不差**。

## 【收尾 · 30 秒】

总结一句话：

**Agent 的本质 = 状态机 + LLM**——每个工作流节点都是一次"读取状态 → 调用 LLM → 更新状态"的过程。LangGraph 把这件事工程化了。

理解了这个，所有模式（计划/反思/CodeAct/人机协作）就只是**状态机的不同边和节点组合方式**。
