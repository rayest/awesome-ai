# 第2章《上下文的"元驱动"：意图识别、规划等模块的工程化实战》内容总结

## 一、章节主线

围绕上下文工程四大模块（感知、规划、执行、反思），落地五大设计模式的工程化实现：

```
感知模块 → 意图识别（2.1）
规划模块 → 计划模式（2.2）
执行模块 → CodeAct 模式（2.4）
反思模块 → 反思模式（2.3）
协同模式 → Human-in-the-loop（2.5）
```

## 二、核心概念

### 2.1 意图识别（感知层）

**定义**：让 LLM 解析用户话语背后的真实意图，同时作为安全、合规的第一道防线。

**两大实现路径**：

| 路径 | 代表方案 | 适用场景 |
|------|---------|---------|
| 轻量级路由模型 | Arch-Router-1.5B | 通用路由、CPU 可运行 |
| 小模型微调 + 大模型兜底 | Qwen2.5-1.5B 微调 + Qwen3-Max 少样本 | 垂直领域、高准确率要求 |

**Arch-Router 关键设计**：
- 两个抽象概念：`Domain`（领域）+ `Operation`（操作）
- 结构化提示词模板：`TASK_INSTRUCTION`（任务定义）+ `FORMAT_PROMPT`（输出规则）
- 强制 JSON 输出：`{"route": "route_name"}`，无匹配返回 `other`

**混合架构（推荐生产方案）**：
- 小模型微调（主路由）：业务数据 30~200 条/类（正样本）+ 5000~20000 条负样本 + 20~100 条长尾样本
- 大模型少样本（兜底）：通过结构化提示词直接分类
- 冲突处理：一致→执行业务；不一致→终止 / 转人工 / 主动澄清 / 记录回流

**微调参数参考**：

| 参数 | 全量微调 | LoRA |
|------|---------|------|
| learning_rate | 5e-6 / 5e-5 | 3e-4 |
| global_batch_size | 256 | 256 |
| num_train_epochs | 3 | 5 |
| lora_dim | 0 | 8/16/32/64 |
| lora_alpha | 0 | 0.5~1 × lora_dim |

**意图识别案例**：用户输入"贵州茅台怎么样"——表层是疑问句，深层意图可能为：礼品体面 / 口感 / 工艺 / 收藏 / 股价走势，需结合用户画像与上下文推断。

### 2.2 计划模式（规划层）

**定义**：先让 LLM 生成结构化任务计划，再按步骤执行的分阶段策略。

#### 简单计划模式（旅游攻略）

```
START → plan_node → execute_node ⇄ tool_node
                            ↓
                          (Final Answer) → END
```

**4 个核心节点**：
- `plan_node`：调 LLM 生成大纲，引用预设资料目录
- `execute_node`：按大纲撰写，绑定 tools
- `tool_node`：解析 tool_calls 并执行
- `should_continue_node`：检测 `Final Answer` 决定终止

**提示词设计要点**：
- 计划模块：角色 + 资源边界 + 格式规范 + 不超出资料
- 执行模块：强约束 + `Final Answer` 终止信号

**异构模型策略**：
- 计划模块：Qwen-Max（强理解）
- 执行模块：Qwen-Turbo（快且便宜）

#### 复杂计划模式（高斯求和）

**核心改进**：
- 引入 `past_steps` 占位符，让 LLM 感知执行位置
- 用 JSON 替代 `Final Answer` 文本标记：
  - `{"actions": {"response": "..."}}` → 终止
  - `{"actions": {"steps": [...]}}` → 继续

**Pydantic 结构化输出**：
```python
class Plan(BaseModel): steps: List[str]
class Response(BaseModel): response: str
class Action(BaseModel):
    actions: Union[Response, Plan]
```

**PlanAgent 类封装**：通过 LangGraph 预构建 `create_react_agent` 简化子任务执行，递归上限 50 防死循环。

**计划模式案例**：撰写春节潮汕旅游攻略——人类只需 3 步（收集信息 → 结构规划 → 逐章填充），而 LLM 通过计划-执行架构能严格按引言/交通/美食/景点/结语结构生成完整文档。

### 2.3 反思模式（反思层）

**定义**：双重 LLM 协作（执行器 + 评判器）的"生成-评估-迭代"闭环，基于语言反馈的强化学习。

**架构**：
```
generate → reflect → check_reflection
              ↑           ↓
              └── (未最优) ┘
```

**3 层终止条件**：
1. 反思返回"无建议/无须优化" → 终止
2. 反思包含 `stop_sign` 停用词（安全风险） → 立即终止
3. `iterations >= 3`（硬上限） → 终止

**核心状态字段**（AgentState）：
- `user_query`、`best_command`、`reflection`：对应提示词占位符
- `iterations`：迭代计数防死循环

**反思模式案例**：运维场景生成 `docker run nginx` 命令——反思模块从安全性、POSIX 合规性、效率、需求契合度 4 维度审查，单轮即通过则结束，最多 3 轮。

---

### 2.4 CodeAct 模式（执行层）

**定义**：将代码本身作为 Agent 行动手段，让 LLM 直接生成可执行代码片段，由运行时环境执行。

**与传统工具调用对比**：

| 维度 | Function Calling / ReAct | CodeAct |
|------|--------------------------|---------|
| 工具形态 | 预注册的黑盒函数 | 动态生成的代码 |
| 灵活性 | 受限于预定义工具集 | 任意时刻构造"临时工具" |
| 适用场景 | 标准化工具体系 | 工具未知、任务多变 |

**轻量级执行器**（示例）：
```python
@tool
def execute_python(code: str) -> str:
    local_vars = {}
    exec(code, {}, local_vars)
    return str(local_vars.get("result", "执行成功"))
```

**工作流**：`llm_call ⇄ process_node`，通过 Markdown `python` 代码块提取 + `_create_agent_action()` 解析代码。

**CodeAct 案例**：绘制近 7 天股票收盘价折线图——LLM 自动生成 Matplotlib 代码 → 执行器运行 → 图像保存到指定目录 → 输出趋势分析，无需预注册绘图工具。

---

### 2.5 Human-in-the-loop（协同层）

**定义**：当前 Agent 作为人类 Copilot 的定位下，在关键决策节点保留人类最终控制权。

**LangGraph 中断 4 要素**：
- `interrupt`：主动挂起工作流
- `memory`：状态存储载体
- `checkpointer`：记录执行进度
- `thread_id`：会话唯一标识

#### 强制人类介入（报告大纲审核）

**架构**：`plan_node → human_node → execute_node`

**恢复机制**：
```python
ret = await agent.ainvoke(Command(resume=get_user_input), config=thread_config)
```

**判定逻辑**：用户输入"确认"→ 保留原计划；其他内容→ 视为修订后计划。

#### 自主触发人类介入（商品咨询）

**核心设计**：创建"空操作"工具 `ask_user`（body 为 `pass`），LLM 通过工具调用自主决策何时追问。

**条件分支判断**（`enter_tools`）：
- 调用 `ask_user` → 路由 `human_node`
- 调用其他业务工具 → 路由 `tool_node`
- 无工具调用 → `END`

**HITL 案例**：用户问"我想买一些苹果，多少钱？"——LLM 识别数量缺失 → 调用 `ask_user` 工具 → 中断等待 → 用户回复"3个" → 继续推理 → 调用 `get_price` 业务工具 → 输出"3个苹果共 6 元"。

## 三、案例与对比

### 五大模式工程实现对比

| 模式 | 触发层 | 关键技术 | LangGraph 节点 | 终止信号 |
|------|--------|----------|----------------|----------|
| 意图识别 | 感知 | 路由模型 / 微调 + 兜底 | 无（预处理） | JSON `route` 标签 |
| 简单计划 | 规划 | 双模块 + Final Answer | plan/execute/tool/should_continue | `Final Answer` 文本 |
| 复杂计划 | 规划 | past_steps + JSON Action | execute/planstep/should_end | `Response` 类型 |
| 反思 | 反思 | 双 LLM 协作 + 迭代上限 | generate/reflect/check_reflection | 3 层终止条件 |
| CodeAct | 执行 | exec() + 代码块解析 | llm_call/process_node | `output` 非空 |
| HITL | 协同 | interrupt + checkpointer | llm/human/tool | 用户输入 / `ask_user` |

### 异构模型成本优化策略

| 阶段 | 模型选择 | 理由 |
|------|----------|------|
| 计划生成 | Qwen-Max | 强理解、推理 |
| 计划执行 | Qwen-Turbo | 响应快、成本低 |
| 意图识别 | 1.5B 小模型 | 高频请求降本 |
| 兜底判断 | Qwen3-Max | 边缘案例质量保障 |

## 五、关键金句

> **意图识别**："是保障 Agent 在真实生产环境中安全、合规、可控运行的第一道防线，更是提升其交互智能性与系统效率的关键技术。"

> **计划模式**："在复杂任务场景中，若能在 LLM 正式处理用户问题前，为其提供一份清晰且可操作的行动计划，并引导其严格按步骤推进，将显著提升任务完成的成功率与执行效率。"

> **反思模式**："这种'生成-评估-迭代'的闭环机制，本质上是一种基于语言反馈的强化学习，使 Agent 能够在无外部奖励信号的情况下，通过自我反思不断优化输出质量。"

> **CodeAct 模式**："不再将'工具'视为静态的、预先注册的黑盒函数，而是将整个编程语言视为一个通用、开放的工具空间。"

> **Human-in-the-loop**："尽管自动化能力不断提升，但在关键决策节点上，人类仍需保有最终控制权。这种设计既提升了系统的安全性与可靠性，也契合现阶段用户对 AI 系统可控、可干预的核心诉求。"

> **整体定位**："本章将聚焦这四大模块在工程实践中的典型实现……通过本章，读者不仅能理解 Agent 的理论基础，更能掌握其在真实系统中的实现。"
