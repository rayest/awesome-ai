# 从提示词到上下文工程 · 口播文案稿

> 适用场景：AI 知识科普 / 技术博主 / 公众号 / 课程导入
> 预计时长：约 10 分钟
> 建议语速：220~250 字/分钟
> 阅读对象：AI 产品经理、开发者、技术管理者

---

## 【开场钩子 · 30 秒】

你有没有过这种经历？

问 AI："理想 L6 怎么样？"
AI 答你一堆：增程器、激光雷达、智驾芯片……
你听完更迷糊了。

换个问法："我是十年老司机，通勤 30 公里，5 口之家，预算 20-25 万，看重底盘和电池，已经试驾过理想 L6。"
AI 秒变贴心顾问，给你列详细对比。

问题来了——你只是改了措辞吗？
不是。
你**提供了一整套上下文**。

这，就是今天要聊的核心：**从提示词工程，到上下文工程**。

## 【核心比喻开场 · 60 秒】

我打一个比方。

你想让朋友推荐一家餐厅。
**提示词工程** = 你递一张纸条："帮我推荐个菜。"
**上下文工程** = 你提前告诉朋友：你吃辣、预算 200、5 个人、忌口香菜、上次踩雷过哪家。

而**Agent** 是什么？
Agent 是那个**真正开餐厅的朋友**——
他会自己去菜市场挑菜（感知），
会列今天的菜单（规划），
会叫厨师下锅（执行），
做完了还会问你味道咋样（反思）。

这整套动作，串起来就是上下文工程。

好，我们正式开始。

## 【第一节 · 为什么从"提示词"到"上下文" · 90 秒】

先说大背景。

2025 年之前，AI 圈子都在聊**提示词工程**——Prompt Engineering。
说白了就是：怎么写好一段话，让模型更好回答。
这种模式下，**人机交互是单向线性的**。

但随着应用深入，三大问题暴露出来：

**第一，意图偏移**。
你说"贵州茅台怎么样"，模型不知道你想问礼品、口感、还是股价。

**第二，长程记忆缺失**。
聊了 50 轮，模型早忘了你开头说了啥。

**第三，逻辑断层**。
面对多步骤任务，模型输出像没头苍蝇。

怎么办？
业界给出的答案是：**上下文工程（Context Engineering）**。

它不再纠结"一句话怎么写"，而是回答一个更本质的问题：
**如何系统化地构建、管理、动态演化 LLM 的输入语境？**

说白了——**把一句话提示词，升级成一整套作战系统**。

## 【第二节 · 上下文工程 = Agent 驱动的智能构建 · 90 秒】

来，我们给"上下文工程"下个工程化的定义。

"上下文"是输入模型的核心语义载体。
"工程"是用程序化手段，对这些载体进行**动态构建 + 持续优化**。

那谁来做这件事？
答案是 **Agent**。

举个例子，**ReAct 范式**。
它是 2023 年提出的：推理 + 行动 + 观察，循环往复。
说白了就是：模型先想一步，干一步，看结果，再想下一步。

看起来简单对吧？
但它的本质就是**动态构建对话上下文**——
模型每一步都基于最新观察，更新自己的"工作记忆"。

虽然 2023 年就有了，但当时大多停留在 Demo 阶段。
上下文窗口溢出、推理偏离、记忆碎片化，这些问题还没充分暴露。
所以**上下文工程**这个概念，**是真实痛点倒逼出来的**。

进一步凝练：
> **Agent 驱动的智能上下文构建**。
> 不仅关注说什么，更关注"如何系统地准备和演化说的内容"。

## 【第三节 · 四大模块 + 两大策略 · 90 秒】

好，理论讲完了。
一个成熟的上下文工程系统，长什么样？

把它比作一家餐厅——
- **感知模块** = 前厅：把"客户随口说"翻译成"菜单语言"，识别意图
- **规划模块** = 厨房的菜单设计：先想好怎么分工、用哪些工具
- **执行模块** = 厨师做菜：调外部 API、跑代码、拿实时数据
- **反思模块** = 吃完点评：菜做对了吗？需要重做吗？

四大模块闭环，构成 Agent 的核心。

但是，光有四大模块还不够。
你得会**管理**上下文。

因为 LLM 的窗口是有限的。
管理有两大策略：

**第一，记忆管理**。
短期记忆存内存，长期记忆存文件或向量数据库。
典型项目：**Mem0**，基于 Agentic RAG 做长期记忆。

**第二，上下文卸载**。
注意：上下文**不是越多越好**。
过剩信息会变成噪声，反而干扰模型。

怎么卸载？
比如 Cursor 写代码时，会把详细分析写到独立 md 文件，窗口里只留**路径 + 摘要**。
比如 **Skills** 技术——初始只加载元数据（名称 + 描述），用到了再加载完整说明。

一句话总结：**引用而非堆砌，按需而非全量**。

## 【第四节 · Agent 演进：从 ReAct 到 Code Agent · 90 秒】

来，聊聊 Agent 这几年的发展。
我给你画条时间线。

**2023 年**：
- **Function Calling**：OpenAI 6 月推出，需要结构化训练数据，黑盒
- **ReAct**：零训练，纯提示词引导，思考过程可见

**2024 年**：
- **Plan-Act 模式**：先谋后动，全局拆解再按步执行
- **Reflection 模式**：独立反思 Agent 校验 + 修正

**2025 上半年**：
- **深度思考型 Agent**：OpenAI DeepResearch、智谱沉思、扣子
- 核心：多轮"思考-执行-反思"循环，能跑几十分钟写报告

**2025 下半年**：
- **Code Agent** 横空出世
- 核心：**代码即行动**
- ReAct 算 1 加到 100，要 99 轮对话
- Code Agent 直接生成一个 for 循环，一次跑完
- 效率提升 N 倍

最关键是：**Code Agent 不是 Cursor、Claude Code 这些 AI 编程工具**。
它是一种**以代码生成为核心决策手段**的范式。
理论上，任何任务都能用代码表达。

## 【第五节 · 多 Agent：解决上下文污染 · 60 秒】

最后一个大招：**多 Agent / SubAgent**。

为什么需要多 Agent？
表面原因：单 Agent 工具多了选不过来。
深层原因：**上下文污染**。

什么叫上下文污染？
单 Agent 跑久了，窗口里混杂：
- 推理轨迹
- 工具描述
- 中间结果
- 历史对话
- 用户意图

这些信息互相干扰，模型聚焦能力下降，窗口迅速耗尽。

怎么解决？
**分而治之**。
- 多个 **SubAgent** 专注子任务，**上下文字段清晰**
- 一个 **Manager Agent** 负责任务路由 + 结果整合

打个比方。
以前找财务小赵报销，他桌上堆满各种票据——差旅、招待、采购。
现在他升职了，把票据分给小张（个人支付）和小李（业务招待）。
小张小李各管一摊，干净高效；小赵只管最后确认。

**整个系统的上下文长度扩展了 N 倍，每个 Agent 在纯净环境里发挥最大效能。**

这就是上下文工程在复杂任务中的**自然延伸**。

## 【决策树 · 60 秒】

来，给大家一个**实战选型决策树**：

任务复杂度怎么样？
- **简单问答** → Function Calling / ReAct
- **多步推理** → Plan-Act
- **高可靠性要求** → 加 Reflection
- **深度调研** → 深度思考型 Agent
- **工具未知 / 临时操作** → Code Agent
- **跨领域复杂** → 多 Agent MAS
- **高风险操作** → 加 Human-In-the-Loop

一句话：**没有银弹，按场景组合**。

## 【踩坑提醒 · 60 秒】

最后，几个常见坑：

1. **把改措辞当银弹**——信息缺失时改措辞没用，要补全上下文
2. **上下文越多越好**——噪声干扰，窗口耗尽，要会卸载
3. **单一 ReAct 应付所有**——复杂任务效率低，要按场景选
4. **单 Agent 跨领域硬扛**——上下文污染导致失败，要拆 SubAgent
5. **忽略 Human-in-the-Loop**——高风险操作必须人工确认

记住：
> **提示词工程是单兵作战，上下文工程是多兵种联合作战。**

## 【收尾 · 30 秒】

最后一句话送给你：

**LLM 之间的能力差距在缩小；
但 Harness（工程化能力）之间的差距在拉大。**

掌握上下文工程，
就掌握了把 LLM 变成真正"能干活"的 Agent 的钥匙。
