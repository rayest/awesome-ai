## 【核心比喻开场 · 60 秒】

我打一个比方。

想象你养了一只小龙虾——但它是**数字虾**：

- **24 小时在线**——不管你什么时候私聊或 @ 它，它都秒回
- **能自己安排工作**——不需要你每天提醒，它能定时推送晨报、提醒你喝水

这就是 **OpenClaw**——一个**始终在线 + 自主调度**的 AI Agent 数字员工。

本章拆解其工程实现。**建议从精简版 NanoClaw 入手**——不是直接啃 40 万行 OpenClaw。

## 【第一节 · 学习路径：从 NanoClaw 入手 · 90 秒】

首先，**关键公式**：

> **现代 Agent = Coding Agent + Skills**

OpenClaw 为什么能写 PPT、回邮件、整理晨报？**不是原生能力**——是内部封装的 Code Agent（基于 Pi 框架）+ Skills 实现的。

OpenClaw 的角色更像一个**功能外壳**——在核心 Agent 之上封装了：
- IM 协议
- 任务编排
- 并发调度
- 定时任务

但是！**OpenClaw 40 万行代码**，初学者根本啃不动。

怎么办？**用 NanoClaw**。

| 指标 | OpenClaw | NanoClaw |
|------|----------|----------|
| 代码量 | ~40 万行 | **~4 000 行** |
| Token 数 | 不可估量 | **~35 000 Token** |
| LLM 全量阅读 | ❌ 不可能 | ✅ 一次喂给 GLM-5（200K 上下文） |

**核心洞察**：让 LLM **一次掌握全项目**——把代码塞进上下文，让模型作为"代码理解助手"。

## 【第二节 · NanoClaw 架构：一内一外 · 90 秒】

NanoClaw 架构可概括为**一内一外**：

```
宿主机（容器外）：
  编排器 / group-queue / Channels / container-runner /
  IPC Watcher / 路由器 / task-scheduler / SQLite

       ↓ Docker Spawn

Docker 容器（容器内）：
  Agent Loop（基于 Claude Agent SDK / Pi 框架）
    + Skills（文件系统操作 / 代码执行 / 浏览器等）
    + 工作目录 + 挂载卷
```

**两大组成部分**：

| 层级 | 角色 | 关键模块 |
|------|------|----------|
| **容器内** | 核心 Agent 运行环境 | Agent Loop + Skills + 浏览器 |
| **容器外** | IM 通信 + 任务调度 | 8 大模块 |

**为什么把 Agent 放在容器内**：

> 安全隔离——即便 Agent 执行了危险操作（误删文件 / 向群聊发送垃圾信息），后果也仅限于容器内部，不会直接影响宿主机安全。

容器外有 8 大模块：
1. 编排器（index.ts）——主循环 + 群组注册
2. group-queue——并发队列
3. Channels——IM 协议注册工厂
4. IPC Watcher——跨群组通信
5. 路由器——消息 XML 格式化
6. container-runner——容器 Spawn
7. task-scheduler——Cron/Interval
8. SQLite——消息/任务/会话/群组存储

## 【第三节 · 6 大功能链路 · 90 秒】

接下来讲 6 大功能链路。

**链路 1：消息接收与持久化**

IM 软件 → Channels 自注册工厂 → 抽象为标准消息 → **立即写入 SQLite**（持久化优先策略）→ 重启后用 lastAgentTimestamp 游标找出未处理消息重新入队。

**核心设计**：**at-least-once 语义**——保证消息至少处理一次。

**链路 2：主循环与消息处理**

编排器每 2 秒轮询数据库。**精巧设计**：如果群组需要触发词（@机器人），激活 Agent 时**不是只处理触发消息**——而是把触发词之前积累的多条上下文消息一并代入。

这样**既保护隐私**（无关闲聊 Agent 看不到），**又保留连贯性**（Agent 能看到完整意图演进）。

**链路 3：定时任务调度**

3 种类型：cron 表达式 / 固定间隔 / 一次性任务。
2 种上下文模式：**隔离模式**（全新会话）/ **群组模式**（能看到历史）。

**巧妙策略：锚定于计划时间**。假设任务每 5 分钟一次，系统中断 2 小时后**不会**连续触发多次追赶——而是从原本应该执行的时间点继续，**避免时间漂移**。

**链路 4：并发控制与容器分配**

group-queue 是并发控制器。**默认最多 5 个 Docker 容器**（可配置）。

- 有空闲容器 → pipe 传输到 stdin → Agent 立即处理
- 无空闲容器 → 消息入队 + 启动新容器

**空闲等待 30 分钟**自动发送 `_close` 优雅关闭，释放资源。

**链路 5：IPC 通信**

**目录路径本身天然充当权限边界**。3 个 IPC 子目录：

| 目录 | 方向 |
|------|------|
| `input/` | 主 → 容器（消息+关闭指令） |
| `messages/` | 容器 → 主（待发送消息） |
| `tasks/` | 容器 → 调度器（定时任务 CRUD） |

普通群组的容器只能访问**自己的 IPC 目录**——无法窥探其他群组。

**链路 6：结果路由**

容器内 Agent 处理完成 → 写入 messages 目录 → IPC Watcher 监听 → 路由器 → 找到对应 Channel → 发送到 IM 软件 → 用户看到回复。

## 【第四节 · 实战：飞书 Channel 开发与部署 · 80 秒】

来，看实战。

**飞书 Channel 开发难点**：

| 维度 | 难度 |
|------|------|
| 与平台服务器通信 | 低（官方 SDK） |
| 接入 NanoClaw 内部模块 | **高**（数据库读写 / 消息路由对接） |

**Vibe Coding 流程**（用 OpenCode）：
1. 切换 Plan 模式，梳理 Telegram 通信流程
2. 理解现有流程
3. 指示 OpenCode 模仿 Telegram，在 `src/channels/` 写飞书
4. 人工审查 + 必要调整

**飞书应用 3 步创建**：
1. 在飞书开放平台创建**自建应用** → 拿到 App ID + App Secret
2. 配置消息接收（webhook）
3. 把凭证填入 `.env`

**接入国内大模型**（关键技巧）：

```bash
ANTHROPIC_BASE_URL=https://api.minimaxi.com/anthropic
ANTHROPIC_MODEL=MiniMax-M2.7
```

利用 Claude Agent SDK 的 `sdkEnv` 参数，**安全地将 .env 配置注入容器环境**——只改 .env 即可灵活切换模型。

**部署 NanoClaw**——4 条命令：

```bash
./build.sh        # 编译 Docker 镜像
npm install       # 安装依赖
npm run build     # 编译 TypeScript
npm run dev       # 启动服务
```

启动后终端显示运行界面，**docker ps** 无容器（系统空闲）。

**飞书测试**：在飞书客户端发送"你好" → **容器被拉起** → 秒回"你好！我是 NanoClaw"。

**定时任务测试**：发送"每天 22 时 20 分提醒我站起来活动一下" → 22:20 准时推送"⏰ 该站起来活动一下啦"。

## 【踩坑提醒 · 60 秒】

最后几个工程坑：

1. **异构 IM 协议难统一**——Channels 自注册工厂模式
2. **Agent 进程崩溃导致消息丢失**——lastAgentTimestamp 游标 + at-least-once
3. **容器资源耗尽**——group-queue 并发控制（默认 5 容器）
4. **空闲容器浪费资源**——30 分钟 IDLE_TIMEOUT 自动 _close
5. **定时任务时间漂移**——锚定于计划时间，不从当前时间追赶
6. **群组之间互相干扰**——独立 IPC 命名空间
7. **国内大模型接入难**——sdkEnv 注入 + .env 灵活配置

## 【收尾 · 30 秒】

总结一句话：

**Claw 类产品的核心 = 始终在线 + 自主调度 + IM 集成**。

把普通 Code Agent 转换为数字员工的关键是：
1. 容器内执行（安全沙箱）
2. 容器外管理（8 大模块协同）
3. 文件系统 IPC 通信（简单可靠 + 天然权限边界）
4. at-least-once 语义（断点续传保证）
5. 锚定计划时间（定时任务不漂移）
