## 一句话总结

Loop Engineering 的重点不是把 prompt 写得更漂亮，而是把 AI 工作设计成可拆解、可验证、可裁决的循环系统；更深一层看，人类真正要设计和审查的是承载判断的“决策变量”和数据结构。

## 内容类型

这是一篇“文章朗读 + 评论解读型”材料。原文介绍 Antonio Vijano 对 Loop Engineering 的三层划分，评论者 Alex 进一步补充了自己的实践观点：AI 自动化的核心不只是 workflow 编排，而是把关键决策压缩成结构化输出，让人、AI、工具都能围绕同一个数据结构协作。

## 核心观点

1. **Level 1：把日常开发变成任务流**
   日常软件任务可以接入 Linear / Kanban 这类任务系统：需求从 backlog 移到 to-do 后触发 agent，实现完成后进入 review，再由独立 agent 或人类做评审和合并决策。这里的关键变化是：不再围绕单次 prompt 工作，而是围绕任务描述、状态流转和审查节点工作。

2. **Level 2：复杂任务需要自定义工作流**
   当任务会因为上下文过长而退化，或者需要跨大量代码、报告、提交记录进行判断时，单纯并行多个 agent 不一定有效。更好的方式是拆成专业角色：例如一个 agent 做 `Git diff checker`，产生候选修复提交；另一个 agent 做 `Patch validator`，独立验证候选是否真的修复问题并映射到 release。材料里的案例从 35% 明确结论提升到 90%。

3. **Level 3：让 AI 生成工作流，但必须事后验证**
   当你不知道正确 workflow 应该怎么设计时，可以让多个 frontier model 先设计方案、互相批判，再交给实现 agent 长时间执行。例如构建 Go 版 blockchain client 的案例结果不错。但反例也很重要：研究论文分析任务中，复杂编排很漂亮，最终报告却不如直接让模型写。这说明 AI 生成 workflow 有用，但你如果不理解它在做什么，就很难修正它的失败。

4. **三层不是升级路线，而是任务匹配**
   Level 1 适合清楚、重复、日常的开发任务；Level 2 适合你理解领域、可以拆角色并设计验证机制的任务；Level 3 适合你不知道该怎么拆、但能接受先运行再审查的探索型任务。

5. **评论者的核心补充：真正的抓手是数据结构**
   不要只问“AI 应该做什么”，要问“我想得到什么结果，以及这个结果应该用什么结构表达”。当每一步都产出一个文件、JSON、表格、候选列表、判断变量或验证状态，下一步就可以根据这些结构化结果继续执行、分支或裁决。这样，人类审查的不再是所有执行细节，而是关键决策变量。

## 内容结构

### 1. 从 prompt engineering 到 task description engineering

Level 1 的核心不是写更强的 prompt，而是把任务描述本身工程化。好的任务描述应该包含目标、上下文、约束、验收标准、截图或录屏要求、PR / review 规则等。材料里提到 `grill me` 这类 skill，本质上就是让 agent 反向采访用户，把含糊需求打磨到可以交付实现的程度。

### 2. 从单 agent 到角色分解

Level 2 的教训是：多个 agent 投票不等于高质量。四个调查员并行处理同一问题，可能只是得到更多“不确定”。真正有效的是拆职责：

- 一个角色负责搜索候选证据。
- 一个角色负责独立验证证据。
- 一个裁决机制负责合并结论。

这比“让模型再认真一点”更可靠，因为它降低了单个上下文里的认知负担，也让每个输出更容易被验证。

### 3. 从人工设计 workflow 到 AI 生成 workflow

Level 3 允许模型参与 workflow 设计，甚至让多个模型互相批判方案。它适合大型、开放、结构不清的问题。但材料也提醒：编排复杂不代表结果好。研究任务的失败案例说明，当任务本身过大、知识密度太高、验证标准不清时，workflow 可能只是把错误包装得更有秩序。

### 4. 从 workflow 到 decision variables

评论者最有价值的补充是：workflow 的关键节点应该沉淀为可复用的数据结构。比如：

- `candidate_commits`: 候选提交列表
- `validation_status`: 是否独立验证通过
- `release_mapping`: 修复对应的 release
- `decision_reason`: 为什么做这个分支选择
- `next_action`: 下一步执行路径

这些变量既可以由人填写，也可以由 AI 生成，还可以由脚本或工具计算。真正重要的是：它们承载了系统的判断。

## 可照做操作步骤

1. **先判断任务属于哪一层**
   如果是重复、明确、验收标准清楚的任务，用 Level 1；如果你知道领域并能拆出验证角色，用 Level 2；如果你不知道流程怎么设计，但可以接受事后审查，用 Level 3。

2. **把输入从“请求”改成“任务描述”**
   至少写清楚目标、背景、非目标、约束、交付物、验收标准、失败时要汇报什么。

3. **为任务设计结构化输出**
   不要只让 agent “分析一下”。要求它输出候选列表、判断字段、状态枚举、证据引用、下一步动作等结构化结果。

4. **把复杂任务拆成角色**
   一个 agent 负责找证据，一个 agent 负责验证，一个 agent 或人类负责裁决。不要让同一个上下文同时承担搜索、判断、验证、总结所有职责。

5. **给每个角色设置验收点**
   例如候选提交必须包含 commit hash、涉及文件、修复理由；验证者必须说明为什么该提交真的解决原报告，并指出对应 release。

6. **保留人类品味闸门**
   AI 很适合执行设计决策，但不应默认替代关键架构、产品取舍、复杂重构和审美判断。重要 PR 不建议只由 AI review 后直接合并。

7. **用失败结果反推 workflow**
   如果结果差，不要只改 prompt。检查是不是任务太大、角色拆错、输出结构不清、验证机制缺失，或决策变量没有定义好。

## 流程视觉图

这篇材料可以沉淀成一个场景化工作方式：

1. **任务进入控制台**
   需求先变成明确 ticket，而不是一句临时 prompt。

2. **控制台选择 loop 层级**
   日常任务走任务流；复杂但你懂领域的任务走验证流；未知流程的任务让模型先设计 workflow。

3. **每个关键节点产出决策变量**
   例如候选、证据、状态、风险、下一步。

4. **验证角色独立审查**
   验证者不复用实现者的上下文，以减少自证偏差。

5. **人类只审最关键的判断**
   人类不必盯着所有执行细节，但要审输入质量、决策变量、验证证据和最终裁决。

## 常见误区

- **误区 1：把 loop 理解成“反复调用模型”**
  真正的 loop 是任务状态、角色职责、结构化输出、验证节点和裁决规则的组合。

- **误区 2：以为更多 agent 投票就更可靠**
  如果角色没有差异，四个 agent 只是四份相似的不确定。角色分解和独立验证比数量更重要。

- **误区 3：把 AI review 当作最终质量保障**
  AI 可以发现一部分问题，但关键设计决策、复杂重构和长期可维护性仍需要人类品味。

- **误区 4：只优化 prompt，不优化输出结构**
  没有数据结构，下一步就只能靠自然语言猜测；有了决策变量，系统才能稳定分支、复用和审查。

## 值得质疑的地方

- 材料里对 Level 3 的成功案例依赖已有测试和规格，这可能低估了测试资产本身的价值。如果没有高质量测试，AI 生成 workflow 的可靠性会明显下降。
- “无人值守执行 24 小时”听起来很强，但更关键的问题是：失败时是否能定位原因、是否能复现、是否能从中改进 workflow。
- 对于研究类任务，复杂编排不一定优于直接让强模型回答。任务拆分如果破坏了整体理解，反而会降低质量。

## 对我的启发

Loop Engineering 的核心不是“让 AI 多做几轮”，而是重新定义人类在系统中的位置：人类从执行者变成任务描述设计者、决策变量设计者、验证机制设计者和最终品味裁决者。未来越多执行被自动化，越要把判断沉淀成可读、可审、可传递的数据结构。

## 一句话复盘

好的 Loop Engineering 不是更长的 prompt，而是更清晰的任务输入、更专业的角色拆解、更独立的验证机制，以及更值得人类审查的决策变量。
