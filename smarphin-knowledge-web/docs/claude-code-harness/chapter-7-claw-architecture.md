## 0. 全章比方：一间永远有人值班的自动化办公室

普通 Code Agent 像临时请来的顾问：你打开会话，它开始工作；你关掉窗口，它就下班。Claw 类产品要解决的是另一件事：让这个顾问变成 24 小时在线的数字员工。

这个数字员工需要三样东西：前台能收消息，秘书能排日程，工作间能安全执行任务。前台就是 IM Channel，秘书就是 task-scheduler 和 group-queue，工作间就是 Docker 容器里的 Agent Loop。容器外负责管理，容器内负责执行，二者通过文件系统 IPC 传话。

## 7.1 核心架构：一内一外

### 类比：酒店前台和后厨

客人不会直接冲进后厨点菜，而是先找前台。前台登记、排队、转单，后厨只负责做菜。Claw 架构也一样：IM 平台来的消息先进入容器外管理层，再被派发给容器内 Agent。

### 痛点：只会写代码的 Agent 不等于数字员工

如果没有外层 Harness，Agent 只能在一次会话里完成一次任务。它不会主动监听飞书、不会定时提醒、不会在崩溃后恢复未处理消息，也很难隔离不同群组的上下文。

### 架构/原理

```text
用户/群聊
  ↓
Channels：飞书 / Telegram 等协议接入
  ↓
SQLite：消息、任务、会话、群组先持久化
  ↓
group-queue：按群组排队与限流
  ↓
container-runner：拉起 Docker 容器
  ↓
容器内 Agent Loop + Skills
  ↓
IPC messages/ 输出结果
  ↓
Router 找回原 Channel 并回复用户
```

容器外有 8 个核心模块：编排器、group-queue、Channels、IPC Watcher、Router、container-runner、task-scheduler、SQLite。它们共同解决“消息如何可靠进入 Agent、结果如何回到用户、任务如何准时触发、容器如何受控运行”。

## 7.2 六条关键功能链路

| 链路 | 核心机制 | 工程价值 |
|------|----------|----------|
| 消息接收与持久化 | Channel 统一消息对象，立即写 SQLite | 崩溃后可恢复 |
| 主循环处理 | 轮询新消息，检查触发词 | 兼顾隐私与上下文连续性 |
| 定时任务 | Cron / Interval / 一次性任务 | 让 Agent 主动工作 |
| 并发控制 | 默认最多 5 个容器，空闲超时关闭 | 控制资源成本 |
| IPC 通信 | input / messages / tasks 三目录 | 简单、可审计、天然隔离 |
| 结果路由 | Router 找回来源 Channel | 回复回原 IM 会话 |

### 关键工程细节

1. **持久化优先**：消息先落库，再进入处理队列。系统重启时用 `lastAgentTimestamp` 找出未处理消息，形成 at-least-once 语义。
2. **触发词前上下文**：群聊里只有 @ 机器人时才激活 Agent，但激活后会带上触发前积累的相关消息，避免只看到最后一句。
3. **调度锚定计划时间**：系统停机后恢复，不把错过的任务一口气补跑，而是沿原计划继续，避免时间漂移。
4. **容器优雅关闭**：空闲超过阈值后发送 `_close`，让 Agent 自己收尾，而不是直接 kill。
5. **目录即权限边界**：普通群组容器只能访问自己的 IPC 目录，不能窥探其他群组。

## 7.3 开发与部署：从飞书 Channel 到国内模型

### 类比：给办公室新增一个前台窗口

接入飞书不是重新造 Agent，而是给同一套管理层新增一个 Channel。难点不在飞书 SDK 本身，而在把飞书事件转成 NanoClaw 内部统一消息，并接入数据库、路由、任务调度。

### 开发路径

| 步骤 | 动作 | 产物 |
|------|------|------|
| 1 | 研究已有 Telegram Channel | 找到消息流入和路由方式 |
| 2 | 用 Plan 模式拆解流程 | 明确需要复用的接口 |
| 3 | 模仿实现飞书 Channel | 新增 `src/channels/` 适配 |
| 4 | 人工审查和联调 | 修正鉴权、事件格式、路由边界 |

国内模型接入的关键是环境注入：把 `ANTHROPIC_BASE_URL`、`ANTHROPIC_MODEL`、Token 等写入 `.env`，再通过 SDK 环境参数注入容器。这样不需要硬编码 Key，也能切换到兼容 Anthropic API 的模型服务。

### 部署闭环

```bash
./build.sh
npm install
npm run build
npm run dev
```

启动后系统空闲时不一定有容器运行；当飞书消息进来，group-queue 才拉起容器。测试链路包括两类：即时对话能否秒回，定时提醒能否按计划触发。

## 横向对比：Claude Code / NanoClaw / OpenClaw

| 维度 | Claude Code | NanoClaw | OpenClaw |
|------|-------------|----------|----------|
| 主要定位 | 编程会话工具 | 学习和定制 Claw 架构 | 完整办公自动化平台 |
| 是否在线监听 | 否 | 是 | 是 |
| IM 集成 | 无 | 可扩展 Channel | 多平台能力更完整 |
| 定时任务 | 无 | 支持 | 支持并更完整 |
| 学习成本 | 低但闭源 | 低，代码量小 | 高，系统庞大 |
| 二次开发 | 受限 | 最适合入门 | 适合深度生产化 |

## 工程踩坑清单

| 坑 | 表现 | 解法 |
|----|------|------|
| IM 协议不统一 | 每个平台事件格式都不同 | Channel 注册表统一消息对象 |
| Agent 崩溃丢消息 | 用户发了但系统没处理 | 消息先落库 + 游标恢复 |
| 群聊隐私泄露 | Agent 看到所有闲聊 | 触发词机制 + 群组配置 |
| 上下文断裂 | 只看到 @ 的最后一句 | 携带触发前相关消息 |
| 容器资源耗尽 | 群组并发拉太多容器 | group-queue 限流 |
| 空闲容器浪费 | 没任务也一直运行 | IDLE_TIMEOUT + `_close` |
| 定时任务漂移 | 恢复后任务扎堆触发 | 锚定计划时间 |
| Key 泄露 | Token 写死进代码 | `.env` + 环境注入 |

## 贯穿主线：一句话哲学总结

Claw 类产品不是让模型变聪明，而是给模型配上一套“值班、排队、路由、隔离、恢复”的工程外骨骼，让一次性 Code Agent 进化成可长期运行的数字员工。

## 学习路径建议

1. 先理解 NanoClaw 的一内一外架构。
2. 再读容器外 8 个模块，重点看消息如何从 IM 进入数据库。
3. 接着看 IPC 三目录和 group-queue，理解隔离与并发。
4. 最后开发一个新 Channel，完成从读懂到改造的闭环。
